# Rust-101, ส่วนที่ 06: การคัดลอก (Copy), อายุขัย (Lifetimes)

เรามาทำงานต่อกับ `BigInt` ของเรากัน โดยเริ่มจากการนำเข้าสิ่งที่เราได้ทำไปแล้ว

```rust
use part05::BigInt;
```

ในเมื่อ `BigInt` เกี่ยวกับตัวเลข เราควรจะสามารถเขียนเวอร์ชันของ `vec_min` ที่คำนวณค่าต่ำสุดของรายการ `BigInt` ได้ ก่อนอื่น เราต้องเขียน `min` สำหรับ `BigInt`

```rust
impl BigInt {
    fn min_try1(self, other: Self) -> Self {
        // เพื่อความแน่ใจ เราตรวจสอบก่อนว่าตัวถูกดำเนินการ (operands) ทั้งสองตัวเป็นไปตาม invariant ของเราจริงๆ
        // `debug_assert!` คือแมโครที่ตรวจสอบว่าอาร์กิวเมนต์ (ต้องเป็นชนิดข้อมูล `bool`) เป็นจริง
        // และจะแพนิคถ้าไม่ใช่ มันจะถูกลบออกใน release builds ซึ่งคุณทำได้ด้วย `cargo build --release`
        debug_assert!(self.test_invariant() && other.test_invariant());
        
        // ตอนนี้ข้อสมมติของเราที่ว่าไม่มีศูนย์ต่อท้ายจะมีประโยชน์แล้ว:
        // ถ้าความยาวของตัวเลขสองตัวแตกต่างกัน เรารู้อยู่แล้วว่าตัวไหนมากกว่า
        if self.data.len() < other.data.len() {
            self
        } else if self.data.len() > other.data.len() {
            other
        } else {
            // **แบบฝึกหัด 06.1**: เติมโค้ดตรงนี้
            unimplemented!()
        }
    }
}
```

ตอนนี้เราสามารถเขียน `vec_min` ได้แล้ว

```rust
fn vec_min(v: &Vec<BigInt>) -> Option<BigInt> {
    let mut min: Option<BigInt> = None;
    // ถ้า `v` เป็น shared reference ไปยังเวกเตอร์ ดังนั้นค่าตั้งต้นสำหรับการวนลูปไปบนมันคือการเรียก
    // `iter` ซึ่งเป็นตัววนซ้ำที่ยืมสมาชิกมา
    for e in v {
        let e = e.clone();
        min = Some(match min {
            None => e,
            Some(n) => e.min_try1(n)
        });
    }
    min
}
```

เอาล่ะ เกิดอะไรขึ้นที่นี่? ทำไมเราต้องทำสำเนาดิบ (deep copy) ของ `e` และทำไมเราถึงไม่ต้องทำแบบนั้นในเวอร์ชันก่อนหน้านี้?

คำตอบซ่อนอยู่ในชนิดข้อมูลของ `vec_min` แล้ว: `v` นั้นแค่ถูกยืมมา แต่ `Option` ที่มันส่งกลับนั้นมีเจ้าของ (owned) เราไม่สามารถส่งหนึ่งในสมาชิกของ `v` กลับไปเฉยๆ ได้ เพราะนั่นจะหมายความว่ามันจะไม่อยู่ในเวกเตอร์อีกต่อไป! ในโค้ดของเรา เรื่องนี้เกิดขึ้นเมื่อเราอัปเดตตัวแปรชั่วคราว `min` ซึ่งก็มีชนิดข้อมูล `Option<BigInt>` เช่นกัน ถ้าคุณเอา `e.clone()` ออก Rust จะบ่นว่า "Cannot move out of borrowed content" (ไม่สามารถย้ายออกจากเนื้อหาที่ถูกยืมมา) นั่นเป็นเพราะ `e` เป็น `&BigInt` การกำหนดค่า `min = Some(*e)` ทำงานเหมือนกับการเรียกฟังก์ชัน: ความเป็นเจ้าของข้อมูลเบื้องหลังถูกโอนจาก `e` ไปหา `min` แต่นั่นไม่อนุญาตให้ทำ เนื่องจากเราเพิ่งจะยืม `e` มา ดังนั้นเราไม่สามารถทำให้มันว่างเปล่าได้! อย่างไรก็ตาม เรา *สามารถ* เรียก `clone` บนมันได้ จากนั้นเราจะเป็นเจ้าของสำเนาที่ถูกสร้างขึ้น และดังนั้นเราจึงสามารถเก็บมันใน `min` ได้

แน่นอนว่า การทำสำเนาดิบแบบนี้มีราคาแพง ดังนั้นเราจึงอยากหลีกเลี่ยงมัน เราจะมาดูเรื่องนั้นกันในส่วนถัดไป

## ชนิดข้อมูล `Copy` (`Copy` types)

แต่ก่อนที่เราจะไปตรงนั้น ผมควรจะตอบคำถามที่สองที่ผมยกขึ้นมาข้างต้นก่อน: ทำไม `vec_min` อันเก่าของเราถึงทำงานได้? เราเก็บ `i32` ที่น้อยที่สุดแบบ local โดยไม่ต้องโคลน และ Rust ก็ไม่ได้บ่นอะไร นั่นเป็นเพราะมันไม่ได้มี "ความเป็นเจ้าของ" อะไรมากมายนักเมื่อพูดถึงชนิดข้อมูลอย่าง `i32` หรือ `bool`: ถ้าคุณย้ายค่าจากที่หนึ่งไปอีกที่หนึ่ง ทั้งสองอินสแตนซ์จะ "สมบูรณ์" ทั้งคู่ เรายังพูดได้ว่าค่าได้ถูกทำซ้ำ (duplicated) นี่แตกต่างอย่างสิ้นเชิงกับชนิดข้อมูลอย่าง `Vec<i32>` ที่ซึ่งการย้ายค่าส่งผลให้เวกเตอร์เก่าและใหม่ชี้ไปยังบัฟเฟอร์เบื้องหลัง *ตัวเดียวกัน* เราไม่ได้มีเวกเตอร์สองตัว มันไม่มีการทำซ้ำอย่างถูกต้องเกิดขึ้น

Rust เรียกชนิดข้อมูลที่สามารถทำซ้ำได้ง่ายๆ ว่า **Copy types** `Copy` เป็นอีกเทรตหนึ่ง และมันถูกอิมพลีเมนท์ให้กับชนิดข้อมูลอย่าง `i32` และ `bool` จำได้ไหมว่าเรานิยามเทรต `Minimum` อย่างไรโดยการเขียน `trait Minimum : Copy {...}`? นี่บอก Rust ว่าทุกๆ ชนิดข้อมูลที่อิมพลีเมนท์ `Minimum` ต้องอิมพลีเมนท์ `Copy` ด้วย และนั่นคือเหตุผลที่คอมไพเลอร์ยอมรับ `vec_min` แบบเจเนอริกของเราในส่วนที่ 02 `Copy` เป็น **Marker Trait** ตัวแรกที่เราเจอ: มันไม่ได้จัดเตรียมเมธอดใดๆ แต่ให้สัญญาเกี่ยวกับพฤติกรรมของชนิดข้อมูล - ในกรณีนี้ คือการทำซ้ำได้

ถ้าคุณพยายามอิมพลีเมนท์ `Copy` ให้กับ `BigInt` คุณจะสังเกตเห็นว่า Rust ไม่ยอมให้คุณทำ ชนิดข้อมูลจะเป็น `Copy` ได้ก็ต่อเมื่อสมาชิก *ทั้งหมด* ของมันเป็น `Copy` และนั่นไม่ใช่กรณีของ `BigInt` อย่างไรก็ตาม เราสามารถทำให้ `SomethingOrNothing<T>` เป็น copy ได้ถ้า `T` เป็น `Copy`

```rust
use part02::{SomethingOrNothing, Something, Nothing};

impl<T: Copy> Copy for SomethingOrNothing<T> {}
```

อีกครั้ง Rust สามารถสร้างการอิมพลีเมนท์ของ `Copy` ให้โดยอัตโนมัติ ถ้าคุณเพิ่ม `#[derive(Copy, Clone)]` ไว้หน้าการนิยาม `SomethingOrNothing` ทั้ง `Copy` และ `Clone` จะถูกสร้างให้โดยอัตโนมัติ

## มุมมองเชิงปฏิบัติการ (An operational perspective)

แทนที่จะมองดูสิ่งที่เกิดขึ้น "ที่เปลือกนอก" (นั่นคือ ที่มองเห็นได้ใน Rust) เรายังสามารถอธิบายการส่งต่อความเป็นเจ้าของและวิธีที่ `Copy` และ `Clone` เข้ามาเกี่ยวข้องได้โดยการดูสิ่งที่เกิดขึ้นบนเครื่องคอมพิวเตอร์จริงๆ เมื่อโค้ด Rust ถูกประมวลผล การส่งค่า (เช่น `i32` หรือ `Vec<i32>`) ไปยังฟังก์ชันจะส่งผลให้เกิด **การคัดลอกตื้น (shallow copy)** เสมอ: Rust แค่คัดลอกไบต์ที่แทนค่านั้น และถือว่าเสร็จงาน นั่นเหมือนกับ copy constructor ตั้งต้นใน C++ อย่างไรก็ตาม Rust จะถือว่านี่เป็นปฏิบัติการที่ **ทำลายล้าง (destructive)**: หลังจากคัดลอกไบต์ไปที่อื่นแล้ว ค่าต้นฉบับจะต้องไม่ถูกใช้อีกต่อไป ท้ายที่สุดแล้ว ทั้งสองอาจจะใช้ pointer ร่วมกันก็ได้!

แต่ถ้าคุณระบุว่าชนิดข้อมูลเป็น `Copy` แล้ว Rust จะไม่ถือว่าการย้าย (move) เป็นการทำลายล้าง และเช่นเดียวกับใน C++ ค่าเก่าและค่าใหม่จะอยู่ร่วมกันได้อย่างมีความสุข ทีนี้ Rust *ไม่อนุญาต* ให้คุณเขียน copy constructor เอง (overload) นี่หมายความว่าการส่งค่าไปมาจะเป็นปฏิบัติการที่รวดเร็วเสมอ ไม่มีการจองหน่วยความจำใหม่หรือการเข้าถึงฮีปรูปแบบอื่นใดเกิดขึ้น ในสถานการณ์ที่คุณจะเขียน copy constructor ใน C++ (และดังนั้นจึงมีค่าใช้จ่ายแอบแฝงในการคัดลอกชนิดข้อมูลนี้ทุกครั้ง) คุณจะให้ชนิดข้อมูลนั้น *ไม่อิมพลีเมนท์* `Copy` แต่มีแค่ `Clone` สิ่งนี้ทำให้ค่าใช้จ่ายนั้นชัดเจน

## อายุขัย (Lifetimes)

เพื่อแก้ปัญหาเรื่องประสิทธิภาพของ `vec_min` เราต้องหลีกเลี่ยงการใช้ `clone` เราอยากให้ค่าที่ส่งกลับไม่มีความเป็นเจ้าของ (จำไว้ว่านี่คือต้นเหตุของความจำเป็นในการโคลนของเรา) แต่ถูกยืมมา กล่าวอีกนัยหนึ่ง เราต้องการส่งคืน shared reference ไปยังสมาชิกที่มีค่าน้อยที่สุด

ฟังก์ชัน `head` สาธิตว่าสิ่งนั้นจะทำงานได้อย่างไร: มันคืนค่า reference ไปยังสมาชิกตัวแรกของเวกเตอร์ถ้ามันไม่ว่างเปล่า ชนิดข้อมูลของฟังก์ชันบอกว่ามันอาจจะไม่คืนอะไรเลย หรือคืนค่า `T` ที่ถูกยืมมา เราสามารถได้รับ reference ไปยังสมาชิกตัวแรกของ `v` และใช้มันเพื่อสร้างค่าที่จะส่งกลับได้

```rust
fn head<T>(v: &Vec<T>) -> Option<&T> {
    if v.len() > 0 {
        Some(&v[0])
    } else {
        None
    }
}
```

ในทางเทคนิค เรากำลังส่งคืน pointer ไปยังสมาชิกตัวแรก แต่นั่นไม่ได้หมายความว่าผู้เรียกต้องระมัดระวังหรอกหรือ? จินตนาการว่า `head` เป็นฟังก์ชัน C++ และเราเขียนโค้ดดังนี้

```cpp
int foo(std::vector<int> v) {
    int *first = head(v);
    v.push_back(42);
    return *first;
}
```

นี่เหมือนกับตัวอย่างแรกสุดที่จูงใจเรื่องความเป็นเจ้าของของเรา ตอนต้นส่วนที่ 04 มาก: `push_back` อาจจองพื้นที่บัฟเฟอร์ใหม่ ทำให้ `first` เป็น pointer ที่ใช้ไม่ได้ อีกครั้งที่เรามี aliasing (ของ `first` และ `v`) และ mutation แต่คราวนี้ บั๊กซ่อนอยู่หลังการเรียก `head` Rust แก้ปัญหานี้อย่างไร? ถ้าเราแปลโค้ดข้างบนเป็น Rust มันคอมไพล์ไม่ผ่าน ดังนั้นชัดเจนว่าเราปลอดภัย - แต่ทำไมและอย่างไร? (สังเกตว่าเราใช้ `unwrap` เพื่อยืนยันอย่างชัดแจ้งว่า `first` ไม่ใช่ `None` ในขณะที่โค้ด C++ ข้างบนจะ dereference NULL-pointer อย่างเงียบๆ แต่นั่นเป็นคนละเรื่องกัน)

```rust
fn rust_foo(mut v: Vec<i32>) -> i32 {
    let first: Option<&i32> = head(&v);
    /* v.push(42); */
    *first.unwrap()
}
```

เพื่อตอบคำถามนี้ เราต้องพูดถึง **อายุขัย (lifetime)** ของ reference ประเด็นคือ การบอกว่าคุณให้เพื่อนยืม `Vec<i32>` หรือหนังสือ ยังไม่ดีพอ เว้นแต่คุณจะตกลงกันด้วยว่าเพื่อนของคุณจะยืมไปนานแค่ไหน ท้ายที่สุด คุณต้องรู้ว่าเมื่อไหร่คุณถึงจะกลับมาเชื่อมั่นในการเป็นเจ้าของข้อมูล (หรือหนังสือ) ของคุณได้อีกครั้ง

ทุกๆ reference ใน Rust มีอายุขัยที่เกี่ยวข้อง เขียนว่า `&'a T` สำหรับ reference ที่มีอายุขัย `'a` ไปยังอะไรสักอย่างที่มีชนิดข้อมูล `T` ชนิดข้อมูลเต็มๆ ของ `head` อ่านได้ดังนี้: `fn<'a, T>(&'a Vec<T>) -> Option<&'a T>` ที่นี่ `'a` คือ **ตัวแปรอายุขัย (lifetime variable)** ซึ่งเป็นตัวแทนว่าเวกเตอร์ถูกยืมมานานแค่ไหน ชนิดข้อมูลของฟังก์ชันแสดงออกว่าอาร์กิวเมนต์และค่าที่ส่งแลกกลับมีอายุขัยเดียวกัน

เมื่อวิเคราะห์โค้ดของ `rust_foo` Rust ต้องมอบหมายอายุขัยให้กับ `first` มันจะเลือกขอบเขต (scope) ที่ `first` ใช้การได้ ซึ่งก็คือส่วนที่เหลือทั้งหมดของฟังก์ชัน เนื่องจาก `head` ผูกอายุขัยของอาร์กิวเมนต์และค่าที่ส่งกลับไว้ด้วยกัน นี่หมายความว่า `&v` ต้องยืม `v` ตลอดระยะเวลาของฟังก์ชัน `rust_foo` ด้วย ดังนั้นเมื่อเราพยายามสร้าง unique reference ไปยัง `v` สำหรับ `push` Rust จึงบ่นว่า reference ทั้งสอง (อันนึงสำหรับ `head` และอีกอันสำหรับ `push`) ทับซ้อนกัน ดังนั้นจึงไม่มีอันไหนเป็น unique reference ได้ โชคดีจริงๆ! Rust จับความผิดพลาดของเราได้และทำให้มั่นใจว่าเราจะไม่ทำโปรแกรมพัง

ดังนั้น เพื่อสรุปเรื่องนี้: Lifetimes ช่วยให้ Rust ให้เหตุผลได้ว่า reference ใช้การได้นานแค่ไหน ความเป็นเจ้าของถูกยืมไปนานแค่ไหน เราจึงสามารถเขียนฟังก์ชันอย่าง `head` ที่ส่งคืน reference เข้าไปในข้อมูลที่มันได้รับมาเป็นอาร์กิวเมนต์ได้อย่างปลอดภัย และมั่นใจได้ว่าพวกมันถูกใช้อย่างถูกต้อง โดยดูแค่ชนิดข้อมูลของฟังก์ชันเท่านั้น ณ จุดใดๆ ในการวิเคราะห์ `rust_foo` ของเรา เราไม่จำเป็นต้องเข้าไปดูไส้ในของ `head` เลย แน่นอนว่านั่นสำคัญมากถ้าเราต้องการแยกโค้ดไลบรารีออกจากโค้ดแอปพลิเคชัน

โดยส่วนใหญ่ เราไม่จำเป็นต้องเติม lifetime ให้กับชนิดข้อมูลของฟังก์ชันอย่างชัดแจ้ง นี่ต้องขอบคุณ **การละ lifetime (lifetime elision)** ที่ Rust จะแทรก lifetime ที่เราไม่ได้ระบุให้โดยอัตโนมัติ ตามกฎง่ายๆ ที่มีเอกสารระบุไว้ชัดเจน
