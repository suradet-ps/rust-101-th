## Part 06: Copy, Lifetimes

เรายังคงทำงานกับ `BigInt` ต่อ ดังนั้นเริ่มต้นด้วยการนำเข้าสิ่งที่เรามีอยู่แล้วจากบทก่อน

```rust
use part05::BigInt;
```

ด้วย `BigInt` เป็นตัวเลข เราควรสามารถเขียนเวอร์ชันของ `vec_min` ที่คำนวณค่าต่ำสุดของรายการ `BigInt` ได้ ก่อนอื่น เราต้องเขียน `min` สำหรับ `BigInt` ก่อน

```rust
impl BigInt {
    fn min_try1(self, other: Self) -> Self {
```

เพื่อความมั่นใจ เราตรวจสอบก่อนว่าตัวถังทั้งสองตัวตรงตามเงื่อนไข invariant ของเราจริงๆ `debug_assert!` เป็นมาโครที่ตรวจสอบว่าอาร์กิวเมนต์ของมัน (ต้องเป็นไทป์ `bool`) เป็นจริง และจะแพนิกถ้าไม่จริง มันจะถูกลบออกใน release builds ซึ่งคุณสร้างด้วย `cargo build --release`

```rust
        debug_assert!(self.test_invariant() && other.test_invariant());
```

ทีนี้สมมติฐานของเราที่ว่าไม่มีเลขศูนย์ต่อท้ายมามีประโยชน์แล้ว: ถ้าความยาวของสองตัวเลขต่างกัน เราก็รู้แล้วว่าตัวไหนน้อยกว่า

```rust
        if self.data.len() < other.data.len() {
            self
        } else if self.data.len() > other.data.len() {
            other
        } else {
```

**แบบฝึกหัด 06.1:** เติมส่วนนี้ให้สมบูรณ์

```rust
            unimplemented!()
        }
    }
}
```

ทีนี้เราสามารถเขียน `vec_min` ได้

```rust
fn vec_min(v: &Vec<BigInt>) -> Option<BigInt> {
    let mut min: Option<BigInt> = None;
```

ถ้า `v` เป็น shared reference ไปยังเวกเตอร์ การวนลูปค่าเริ่มต้นจะเรียก `iter` ซึ่งเป็น iterator ที่ยืม (borrow) ข้อมูล

```rust
    for e in v {
        let e = e.clone();
        min = Some(match min {
            None => e,
            Some(n) => e.min_try1(n)
        });
    }
    min
}
```

ทีนี้ เกิดอะไรขึ้นที่นี่? ทำไมเราต้องทำสำเนาเต็มรูปแบบ (deep copy) ของ `e` และทำไมเราไม่ต้องทำแบบนั้นในเวอร์ชันก่อนหน้า?

คำตอบซ่อนอยู่ในไทป์ของ `vec_min` แล้ว: `v` ถูกยืม (borrowed) มาเท่านั้น แต่ `Option` ที่คืนค่าเป็น owned เราไม่สามารถคืนหนึ่งใน elements ของ `v` ได้ เพราะนั่นจะหมายความว่ามันไม่อยู่ในเวกเตอร์อีกต่อไป! ในโค้ดของเรา ปัญหานี้เกิดขึ้นเมื่อเราอัปเดตตัวแปรกลาง `min` ซึ่งก็มีไทป์ `Option<BigInt>` เช่นกัน ถ้าคุณลบ `e.clone()` ออก Rust จะบ่นว่า "Cannot move out of borrowed content" นั่นเป็นเพราะ `e` เป็น `&BigInt` การกำหนด `min = Some(*e)` ทำงานเหมือนกับการเรียกฟังก์ชัน: ownership ของข้อมูลพื้นฐานถูกโอนจาก `e` ไปยัง `min` แต่นั่นไม่ได้รับอนุญาต เพราะเราแค่ยืม `e` มาเท่านั้น เราจึงไม่สามารถล้างมันได้! อย่างไรก็ตาม เราสามารถเรียก `clone` บนมันได้ แล้วเราก็เป็นเจ้าของสำเนาที่ถูกสร้างขึ้น ดังนั้นเราจึงสามารถเก็บมันใน `min` ได้

แน่นอนว่าการทำสำเนาเต็มรูปแบบแบบนี้มีราคาแพง ดังนั้นเราอยากหลีกเลี่ยงมัน เราจะมาถึงเรื่องนั้นในบทถัดไป

## Copy types

แต่ก่อนจะไปที่นั่น ผมควรตอบคำถามที่สองที่ผมยกขึ้นมาข้างต้น: ทำไม `vec_min` เวอร์ชันเก่าของเราถึงทำงานได้? เราเก็บค่า `i32` ต่ำสุดไว้ในเครื่องมือโดยไม่ต้อง clone และ Rust ก็ไม่บ่น

นั่นเป็นเพราะว่าไม่ได้มี "ownership" ที่แท้จริงเมื่อพูดถึงไทป์อย่าง `i32` หรือ `bool`: ถ้าคุณย้ายค่าจากที่หนึ่งไปยังอีกที่หนึ่ง แล้ว instances ทั้งสองก็ "สมบูรณ์" เรายังบอกว่าค่าถูก duplicate นี่แตกต่างอย่างสิ้นเชิงกับไทป์อย่าง `Vec<i32>` ที่การย้ายค่าทำให้ทั้งเวกเตอร์เก่าและใหม่ชี้ไปยัง buffer พื้นฐานเดียวกัน เราไม่มีสองเวกเตอร์ ไม่มีการ duplicate ที่ถูกต้อง

Rust เรียกไทป์ที่สามารถ duplicate ได้ง่ายว่า **Copy types** `Copy` เป็นอีกหนึ่ง trait และมันถูก implement สำหรับไทป์อย่าง `i32` และ `bool` จำได้ไหมว่าเรานิยาม trait `Minimum` โดยเขียน `trait Minimum: Copy {...}`? นี่บอก Rust ว่าทุกไทป์ที่ implement `Minimum` ต้อง implement `Copy` ด้วย และนั่นเป็นเหตุผลที่คอมไพเลอร์ยอมรับ `vec_min` แบบ generic ของเราในบท 02 `Copy` เป็น **marker trait** ตัวแรกที่เราพบ: มันไม่ได้ให้เมธอดใดๆ แต่ทำสัญญาเกี่ยวกับพฤติกรรมของไทป์ - ในกรณีนี้คือการสามารถ duplicate ได้

ถ้าคุณพยายาม implement `Copy` สำหรับ `BigInt` คุณจะสังเกตว่า Rust ไม่ยอมให้คุณทำ ไทป์หนึ่งจะเป็น `Copy` ได้ก็ต่อเมื่อ elements ทั้งหมดของมันเป็น `Copy` และนั่นไม่ใช่กรณีของ `BigInt` อย่างไรก็ตาม เราสามารถทำให้ `SomethingOrNothing<T>` เป็น copy ได้ถ้า `T` เป็น `Copy`

```rust
use part02::{SomethingOrNothing, Something, Nothing};

impl<T: Copy> Copy for SomethingOrNothing<T> {}
```

อีกครั้ง Rust สามารถสร้าง implementations ของ `Copy` โดยอัตโนมัติได้ ถ้าคุณเพิ่ม `#[derive(Copy, Clone)]` ไว้หน้าคำจำกัดความของ `SomethingOrNothing` ทั้ง `Copy` และ `Clone` จะถูกสร้างให้อัตโนมัติ

## มุมมองเชิงปฏิบัติ

แทนที่จะดูว่าเกิดอะไรขึ้น "ที่ผิวเผิน" (คือมองเห็นได้ใน Rust) เราอธิบายการส่งผ่าน ownership และว่า `Copy` และ `Clone` เข้ากันอย่างไร โดยดูว่าเกิดอะไรขึ้นบนเครื่องจริงๆ

เมื่อโค้ด Rust ถูก execute การส่งค่า (อย่าง `i32` หรือ `Vec<i32>`) ไปยังฟังก์ชันจะทำให้ **shallow copy** เกิดขึ้นเสมอ: Rust แค่ copy bytes ที่แทนค่านั้น และถือว่าเสร็จสิ้น นั่นเหมือนกับ default copy constructor ใน C++ อย่างไรก็ตาม Rust จะถือว่านี่เป็นการดำเนินการที่ทำลายล้าง (destructive): หลังจาก copy bytes ไปยังที่อื่นแล้ว ค่าเดิมต้องไม่ถูกใช้อีกต่อไป ท้ายที่สุด ทั้งสองอาจ share pointer กันได้! แต่ถ้าคุณ mark ไทป์ว่าเป็น `Copy` แล้ว Rust จะไม่ถือว่าการย้าย (move) เป็นทำลายล้าง และเหมือนกับใน C++ ค่าเก่าและใหม่ก็สามารถอยู่ร่วมกันได้อย่างมีความสุข

ทีนี้ Rust ไม่อนุญาตให้คุณ overload copy constructor นี่หมายความว่าการส่งค่าไปมาจะเป็นการดำเนินการที่เร็วเสมอ ไม่มีการจัดสรรหน่วยความจำหรือการเข้าถึง heap ใดๆ เกิดขึ้น ในสถานการณ์ที่คุณจะเขียน copy constructor ใน C++ (และดังนั้นจะเสียค่าใช้จ่ายที่ซ่อนอยู่ทุกครั้งที่ copy ไทป์นี้) คุณจะทำให้ไทป์นั้นไม่ implement `Copy` แต่ implement `Clone` เท่านั้น นี่ทำให้ค่าใช้จ่ายชัดเจน: ผู้เรียกใช้ต้องเรียก `.clone()` เพื่อสร้างสำเนา

## Lifetimes

เพื่อแก้ปัญหาประสิทธิภาพของ `vec_min` เราต้องหลีกเลี่ยงการใช้ `clone` เราอยากให้ค่าที่คืนไม่ใช่ owned (จำได้ไหมว่านั่นเป็นที่มาของความจำเป็นต้อง clone) แต่เป็น borrowed กล่าวอีกนัยหนึ่ง เราอยากคืน shared reference ไปยัง `BigInt` ที่มีค่าน้อยที่สุด

หัวฟังก์ชันนี้แสดงว่ามันทำงานได้อย่างไร: มันคืน reference ไปยัง element แรกของเวกเตอร์ถ้ามันไม่ว่างเปล่า ไทป์ของฟังก์ชันบอกว่ามันจะคืน nothing หรือคืน borrowed `T` เราสามารถได้ reference ไปยัง element แรกของ `v` และใช้มันเพื่อสร้างค่าที่คืน

```rust
fn head<T>(v: &Vec<T>) -> Option<&T> {
    if v.len() > 0 {
        Some(&v[0])
    } else {
        None
    }
}
```

ทางเทคนิค เรากำลังคืน pointer ไปยัง element แรก แต่นั่นไม่ได้หมายความว่าผู้เรียกใช้ต้องระวังหรือ? ลองนึกภาพว่า `head` เป็นฟังก์ชัน C++ และเราเขียนแบบนี้

```cpp
/*
int foo(std::vector<int> v) {
    int *first = head(v);
    v.push_back(42);
    return *first;
}
*/
```

นี่เหมือนกับตัวอย่างแรกสุดของเราสำหรับ ownership ตอนต้นบท 04 มาก: `push_back` อาจ reallocate buffer ทำให้ `first` กลายเป็น pointer ที่ไม่ถูกต้อง อีกครั้ง เรามี aliasing (ของ `first` และ `v`) และ mutation แต่ครั้งนี้บั๊กซ่อนอยู่หลังการเรียก `head` Rust แก้ปัญหานี้อย่างไร? ถ้าเราแปลโค้ดข้างต้นเป็น Rust มันไม่ compile ดังนั้นเราปลอดภัยแล้ว - แต่อย่างไรและทำไม? (สังเกตว่าเราใช้ `unwrap` เพื่อ assert อย่างชัดเจนว่า `first` ไม่ใช่ `None` ในขณะที่โค้ด C++ ข้างต้นจะ dereference NULL pointer อย่างเงียบๆ แต่นั่นเป็นอีกเรื่องหนึ่ง)

```rust
fn rust_foo(mut v: Vec<i32>) -> i32 {
    let first: Option<&i32> = head(&v);
    /* v.push(42); */
    *first.unwrap()
}
```

เพื่อตอบคำถามนี้ เราต้องพูดถึง **lifetime** ของ reference ประเด็นคือ การบอกว่าคุณยืม `Vec<i32>` หรือหนังสือให้เพื่อน มันไม่เพียงพอ นอกเสียจากคุณยังตกลงกันด้วยว่าเพื่อนจะยืมมันนานแค่ไหน ท้ายที่สุด คุณต้องรู้ว่าเมื่อไหร่คุณสามารถพึ่งพาการเป็นเจ้าของข้อมูล (หรือหนังสือ) ของคุณได้

ทุก reference ใน Rust มี lifetime ที่เกี่ยวข้อง เขียนเป็น `&'a T` สำหรับ reference ที่มี lifetime `'a` ไปยังบางอย่างของไทป์ `T` ไทป์เต็มของ `head` อ่านดังนี้: `fn<'a, T>(&'a Vec<T>) -> Option<&'a T>` ที่นี่ `'a` เป็นตัวแปร lifetime ซึ่งแทนระยะเวลาที่เวกเตอร์ถูกยืมมา ไทป์ฟังก์ชันแสดงว่าอาร์กิวเมนต์และค่าที่คืนมี lifetime เดียวกัน

เมื่อวิเคราะห์โค้ดของ `rust_foo` Rust ต้องกำหนด lifetime ให้ `first` มันจะเลือก scope ที่ `first` มีผล ซึ่งก็คือส่วนที่เหลือทั้งหมดของฟังก์ชัน เพราะ `head` ผูก lifetime ของอาร์กิวเมนต์และค่าที่คืนเข้าด้วยกัน นี่หมายความว่า `&v` ก็ต้องยืม `v` ตลอดระยะเวลาของฟังก์ชัน `rust_foo` ด้วย ดังนั้นเมื่อเราพยายามสร้าง unique reference ไปยัง `v` สำหรับ `push` Rust บ่นว่า references สองตัว (ตัวหนึ่งสำหรับ `head` และอีกตัวสำหรับ `push`) ทับซ้อนกัน ดังนั้นไม่มีตัวไหนเป็น unique ได้ โชคดีจริงๆ! Rust จับความผิดพลาดของเราและทำให้แน่ใจว่าเราไม่ crash

สรุป: Lifetimes ช่วยให้ Rust ให้เหตุผลได้ว่า reference มีผลนานแค่ไหน นานแค่ไหนที่ ownership ถูกยืมไป เราจึงสามารถเขียนฟังก์ชันอย่าง `head` ที่คืน references เข้าไปในข้อมูลที่มันได้รับเป็นอาร์กิวเมนต์ และทำให้แน่ใจว่าถูกใช้อย่างถูกต้น โดยดูแค่ไทป์ฟังก์ชัน เราไม่ต้องดูเข้าไปใน `head` เลยในการวิเคราะห์ `rust_foo` นั่นแน่นอนสำคัญมากถ้าเราอยากแยกโค้ด library ออกจากโค้ด application

ส่วนใหญ่ เราไม่ต้องเพิ่ม lifetimes ให้กับไทป์ฟังก์ชันอย่างชัดเจน นี่ขอบคุณ **lifetime elision** ที่ Rust จะแทรก lifetimes ที่เราไม่ได้ระบุโดยอัตโนมัติ ตามกฎง่ายๆ ที่ documented ไว้ดี
