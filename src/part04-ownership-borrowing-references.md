# Rust-101, ส่วนที่ 04: ความเป็นเจ้าของ (Ownership), การยืม (Borrowing), การอ้างอิง (References)

Rust มีเป้าหมายที่จะเป็น "ภาษาระบบที่ปลอดภัย (safe systems language)" ในฐานะที่เป็นภาษาระบบ แน่นอนว่ามันต้องมี reference (หรือ pointer) แต่ในฐานะที่เป็นภาษาที่ปลอดภัย มันต้องป้องกันบั๊กแบบนี้ใน C++ ไม่ให้เกิดขึ้น:

```cpp
void foo(std::vector<int> v) {
    int *first = &v[0];
    v.push_back(42);
    *first = 1337; // นี่มันแย่มาก!
}
```

เกิดอะไรขึ้นที่นี่? `first` คือตัวชี้ (pointer) ที่ชี้เข้าไปในเวกเตอร์ `v` ปฏิบัติการ `push_back` อาจจะจองพื้นที่หน่วยความจำใหม่ (reallocate) สำหรับเวกเตอร์ ในกรณีที่บัฟเฟอร์เก่าเต็ม ถ้าเหตุการณ์นั้นเกิดขึ้น `first` จะกลายเป็นตัวชี้ที่ห้อยโหน (dangling pointer) ทันที และการเข้าถึงมันสามารถทำให้โปรแกรมพัง (หรือแย่กว่านั้น)

ปรากฏว่า การรวมตัวกันของสถานการณ์สองอย่างเท่านั้นที่สามารถนำไปสู่บั๊กเช่นนี้ได้: **การมีชื่อเรียกซ้ำซ้อน (aliasing)** และ **การเปลี่ยนแปลงค่า (mutation)** ในโค้ดข้างต้น เรามี `first` และบัฟเฟอร์ของ `v` เป็นชื่อเรียกซ้ำซ้อนกัน (aliases) และเมื่อ `push_back` ถูกเรียก ตัวหลังถูกใช้เพื่อทำการเปลี่ยนแปลงค่า ดังนั้น หลักการสำคัญของระบบชนิดข้อมูล (typesystem) ของ Rust คือการกำจัดการเปลี่ยนแปลงค่าในขณะที่มีการใช้ชื่อเรียกซ้ำซ้อน เครื่องมือหลักเพื่อบรรลุสิ่งนั้นคือแนวคิดเรื่อง **ความเป็นเจ้าของ (ownership)**

## ความเป็นเจ้าของ (Ownership)

ในทางปฏิบัติมันหมายความว่าอย่างไร? พิจารณาโค้ดต่อไปนี้:

```rust
fn work_on_vector(v: Vec<i32>) { /* ทำอะไรสักอย่าง */ }

fn ownership_demo() {
    let v = vec![1,2,3,4];
    work_on_vector(v);
    /* println!("The first element is: {}", v[0]); */               /* แย่มาก! */
}
```

Rust แนบความหมายเพิ่มเติมให้กับอาร์กิวเมนต์ของ `work_on_vector`: ฟังก์ชันนี้สามารถถือเอาว่ามันเป็นเจ้าของ `v` อย่างสมบูรณ์ และดังนั้นจึงสามารถทำอะไรกับมันก็ได้ เมื่อ `work_on_vector` จบลง จะไม่มีใครต้องการ `v` อีกต่อไป ดังนั้นมันจะถูกลบ (รวมถึงบัฟเฟอร์ของมันบนฮีปด้วย) การส่ง `Vec<i32>` ให้กับ `work_on_vector` ถือเป็น **การโอนความเป็นเจ้าของ (transfer of ownership)**: เคยมีใครบางคนเป็นเจ้าของเวกเตอร์นั้น แต่ตอนนี้เขาให้มันไปแล้วและไม่มีธุระอะไรกับมันอีกต่อไป

ถ้าคุณให้หนังสือกับเพื่อน คุณไม่สามารถบุกไปบ้านเขาในวันถัดไปแล้วหยิบหนังสือออกมาได้! มันไม่ใช่ของคุณอีกต่อไปแล้ว Rust ทำให้มั่นใจว่าคุณจะไม่ละเมิดกฎข้อนี้ ลองเปิดใช้งานบรรทัดที่คอมเมนต์ไว้ใน `ownership_demo` ดู Rust จะบอกคุณว่า `v` ได้ถูก **ย้าย (moved)** ไปแล้ว ซึ่งหมายความว่าความเป็นเจ้าของได้ถูกโอนไปที่อื่นแล้ว ในกรณีเฉพาะนี้ บัฟเฟอร์ที่เก็บข้อมูลไม่มีอยู่แล้วด้วยซ้ำ ดังนั้นเราโชคดีมากที่ Rust จับปัญหานี้ได้! โดยเนื้อแท้แล้ว ความเป็นเจ้าของจะกำจัดการมีชื่อเรียกซ้ำซ้อน (aliasing) ดังนั้นจึงทำให้ปัญหาแบบที่อภิปรายข้างต้นเป็นไปไม่ได้

## การยืม Reference แบบแบ่งปัน (Borrowing a shared reference)

ถ้าคุณย้อนกลับไปดูตัวอย่างของเรากับ `vec_min` และพยายามเรียกฟังก์ชันนั้นสองครั้ง คุณจะเจอข้อผิดพลาดเดียวกัน นั่นเป็นเพราะ `vec_min` เรียกร้องให้ผู้เรียกโอนความเป็นเจ้าของเวกเตอร์ไปให้ ดังนั้น เมื่อ `vec_min` ทำงานเสร็จ เวกเตอร์ทั้งก้อนจะถูกลบ แน่นอนว่านั่นไม่ใช่สิ่งที่เราต้องการ! เราไม่สามารถให้ `vec_min` เข้าถึงเวกเตอร์ โดยที่เรายังคงความเป็นเจ้าของมันไว้ได้หรือ?

Rust เรียกสิ่งนี้ว่า **การอ้างอิง (reference)** ไปยังเวกเตอร์ และมันพิจารณาว่า reference เป็น **การยืม (borrowing)** ความเป็นเจ้าของ สิ่งนี้ทำงานเหมือนกับการยืมในโลกความเป็นจริง: ถ้าเพื่อนของคุณยืมหนังสือไปจากคุณ เพื่อนคนนั้นสามารถมีมันไว้และใช้งานมันได้ (แต่คุณใช้ไม่ได้!) ตราบเท่าที่หนังสือยังถูกยืมอยู่ เพื่อนของคุณอาจจะให้คนอื่นยืมหนังสือต่อได้ด้วย แต่ในที่สุด เพื่อนของคุณต้องคืนหนังสือกลับมาให้คุณ ซึ่ง ณ จุดนั้นคุณจะมีความเป็นเจ้าของอย่างสมบูรณ์อีกครั้ง

Rust แยกแยะ reference ออกเป็นสองประเภท อย่างแรกคือ **Reference แบบแบ่งปัน (shared reference)** ตรงนี้แหละที่การเปรียบเปรยเรื่องหนังสือเริ่มใช้ไม่ได้... คุณสามารถให้ shared reference ของข้อมูลเดียวกันกับผู้คนจำนวนมากได้ ซึ่งทุกคนสามารถเข้าถึงข้อมูลนั้นได้ สิ่งนี้แน่นอนว่าทำให้เกิด aliasing ดังนั้นเพื่อที่จะรักษาสัญญาเรื่องความปลอดภัย Rust โดยทั่วไปจึงไม่อนุญาตให้มีการเปลี่ยนแปลงค่าผ่านทาง shared reference

ดังนั้น เรามาเขียน `vec_min` ใหม่ให้ทำงานกับ shared reference ของเวกเตอร์กัน ซึ่งเขียนว่า `&Vec<i32>` ผมถือวิสาสะแปลงฟังก์ชันจาก `SomethingOrNothing` ให้เป็นชนิดข้อมูล `Option` จากไลบรารีมาตรฐานด้วย

```rust
fn vec_min(v: &Vec<i32>) -> Option<i32> {
    use std::cmp;

    let mut min = None;
    // ครั้งนี้ เราขอตัววนซ้ำ (iterator) สำหรับเวกเตอร์ `v` อย่างชัดแจ้ง เมธอด `iter` เพียงแค่
    // ยืมเวกเตอร์ที่มันทำงานด้วย และจัดเตรียม shared reference ไปยังสมาชิกแต่ละตัว
    for e in v.iter() {
        // ในลูป ตอนนี้ `e` มีชนิดข้อมูลเป็น `&i32` ดังนั้นเราต้อง dereference มันเพื่อเอา `i32` ออกมา
        min = Some(match min {
            None => *e,
            Some(n) => cmp::min(n, *e)
        });
    }
    min
}
```

เมื่อ `vec_min` ไม่ได้รับความเป็นเจ้าของเวกเตอร์มาอีกต่อไป เราสามารถเรียกมันหลายๆ ครั้งบนเวกเตอร์ตัวเดิมได้ และยังทำสิ่งเหล่านี้ได้ด้วย:

```rust
fn shared_ref_demo() {
    let v = vec![5,4,3,2,1];
    let first = &v[0];
    vec_min(&v);
    vec_min(&v);
    println!("The first element is: {}", *first);
}
```

เกิดอะไรขึ้นที่นี่? อย่างแรก `&` คือวิธีการที่คุณให้คนอื่นยืมความเป็นเจ้าของ - ตัวดำเนินการนี้สร้าง shared reference ขึ้นมา `shared_ref_demo` สร้าง shared reference สามตัวไปยัง `v`: Reference `first` เริ่มต้นในบรรทัดที่ 2 ของฟังก์ชันและอยู่ยาวไปจนจบฟังก์ชัน Reference อีกสองตัวที่สร้างขึ้นเพื่อเรียก `vec_min` มีอายุอยู่แค่ช่วงเวลาของการเรียกฟังก์ชันนั้นๆ เท่านั้น

ในทางเทคนิค แน่นอนว่า reference ก็คือ pointer สังเกตว่าเนื่องจาก `vec_min` ได้รับแค่ shared reference ทาง Rust จึงรู้ว่ามันไม่สามารถเปลี่ยนแปลงค่า `v` ได้ ดังนั้น pointer ที่ชี้ไปยังบัฟเฟอร์ของ `v` ที่ถูกสร้างขึ้นก่อนเรียก `vec_min` จึงยังคงใช้การได้

## Reference แบบแก้ไขได้ที่ไม่ซ้ำใคร (Unique, mutable references)

มีวิธีที่สองในการยืมบางอย่าง reference ประเภทที่สอง: **Reference แบบแก้ไขได้ (mutable reference)** นี่คือ reference ที่มาพร้อมกับคำสัญญาว่าไม่มีใครอื่นอีกแล้วที่สามารถเข้าถึงผู้ถูกอ้างถึง (referee) ได้ - ตรงกันข้ามกับ shared reference จะไม่มี aliasing เกิดขึ้นกับ mutable reference ดังนั้นจึงปลอดภัยเสมอที่จะทำการเปลี่ยนแปลงค่าผ่าน reference แบบนี้ เนื่องจากไม่สามารถมี reference อื่นไปยังข้อมูลเดียวกันได้ เราอาจจะเรียกมันว่า **Reference แบบไม่ซ้ำใคร (unique reference)** ก็ได้ แต่นั่นไม่ใช่ชื่ออย่างเป็นทางการของพวกมัน

ตัวอย่างเช่น พิจารณาฟังก์ชันที่เพิ่มค่าสมาชิกทุกตัวของเวกเตอร์ขึ้น 1 ชนิดข้อมูล `&mut Vec<i32>` คือชนิดข้อมูลของ mutable reference ไปยัง `Vec<i32>` เนื่องจาก reference นี้แก้ไขค่าได้ เราสามารถใช้ตัววนซ้ำแบบแก้ไขได้ (mutable iterator) ซึ่งจะให้ mutable reference ไปยังสมาชิกแต่ละตัว

```rust
fn vec_inc(v: &mut Vec<i32>) {
    for e in v.iter_mut() {
        *e += 1;
    }
}
```

นี่คือตัวอย่างการเรียก `vec_inc`:

```rust
fn mutable_ref_demo() {
    let mut v = vec![5,4,3,2,1];
    /* let first = &v[0]; */
    vec_inc(&mut v);
    vec_inc(&mut v);
    /* println!("The first element is: {}", *first); */             /* แย่มาก! */
}
```

`&mut` คือตัวดำเนินการเพื่อสร้าง mutable reference เราต้องระบุว่า `v` เป็น `mut` (แก้ไขได้) เพื่อที่จะสร้าง reference แบบนี้: ถึงแม้ว่าเราจะเป็นเจ้าของ `v` อย่างสมบูรณ์ แต่ Rust พยายามปกป้องเราจากการเผลอไปแก้ไขข้อมูล ดังนั้นตัวแปรที่มีเจ้าของ (owned variables) ที่คุณตั้งใจจะแก้ไขค่าต้องถูกกำกับด้วย `mut` เนื่องจาก reference ที่ส่งไปยัง `vec_inc` มีอายุแค่เท่ากับช่วงเวลาของฟังก์ชันคอล เราจึงยังสามารถเรียก `vec_inc` บนเวกเตอร์ตัวเดิมสองครั้งได้: ช่วงเวลาของ reference ทั้งสองไม่ทับซ้อนกัน ดังนั้นเราจึงไม่เคยมี mutable reference มากกว่าหนึ่งตัว - เรายืม `v` แค่ทีละครั้งเท่านั้น อย่างไรก็ตาม เราไม่สามารถสร้าง shared reference ที่มีอายุคาบเกี่ยวกับการเรียก `vec_inc` ได้ ลองเปิดใช้งานบรรทัดที่คอมเมนต์ไว้ดู และคอยดู Rust บ่น นี่เป็นเพราะ `vec_inc` สามารถเปลี่ยนแปลงโครงสร้างของเวกเตอร์ (เช่น มันอาจเพิ่มหรือลบสมาชิก) และทำให้ reference `first` ใช้การไม่ได้ กล่าวอีกนัยหนึ่ง Rust คอยรักษาความปลอดภัยให้เราจากบั๊กแบบในตัวอย่าง C++ ข้างต้น

ก่อนหน้านี้ ผมบอกว่าการมี mutable reference จะกันไม่ให้เกิด aliasing แต่ถ้าคุณมองดูโค้ดข้างบนดีๆ คุณอาจจะบอกว่า: "เ เดี๋ยวสิ! `v` ใน `mutable_ref_demo` กับ `v` ใน `vec_inc` มันไม่ได้เป็น alias กันเหรอ?" และคุณพูดถูก มันเป็น แต่ทว่า `v` ใน `mutable_ref_demo` นั้นใช้การไม่ได้จริงๆ มันไม่ได้ active อยู่: ตราบเท่าที่ `v` ถูกยืมไป Rust จะไม่อนุญาตให้คุณทำอะไรกับมันเลย

## สรุป (Summary)

ระบบความเป็นเจ้าของและการยืมของ Rust บังคับใช้กฎสามข้อต่อไปนี้:

* มี **เจ้าของ (owner)** ของข้อมูลได้เพียง **คนเดียว** เสมอ
* ถ้ามี **mutable reference** ที่ active อยู่ จะไม่มีใครอื่นสามารถเข้าถึงข้อมูลนั้นได้อีก
* ถ้ามี **shared reference** ที่ active อยู่ ทุกการเข้าถึงข้อมูลนั้นจากที่อื่นต้องเป็น shared reference เหมือนกัน

ผลปรากฏว่า เมื่อรวมกับความสามารถด้าน abstraction ของ Rust แล้ว นี่เป็นกลไกที่ทรงพลังมากในการจัดการกับปัญหามากมายนอกเหนือไปจากความปลอดภัยของหน่วยความจำพื้นฐาน คุณจะได้เห็นตัวอย่างเร็วๆ นี้
