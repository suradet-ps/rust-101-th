## Part 04: Ownership, Borrowing, References

Rust มีเป้าหมายเป็น **ภาษาระดับระบบที่ปลอดภัย (safe systems language)** ในฐานะภาษาระดับระบบ แน่นอนว่ามันมี references (หรือ pointers) ให้ใช้ แต่ในฐานะภาษาที่ปลอดภัย มันต้องป้องกันไม่ให้เกิดบั๊กแบบในโค้ด C++ นี้

```cpp
void foo(std::vector<int> v) {
    int *first = &v[0];
    v.push_back(42);
    *first = 1337; // This is bad!
}
```

ปัญหาคืออะไร? `first` เป็น pointer ชี้เข้าไปใน vector `v` เมื่อเรียก `push_back` อาจต้องจัดสรรหน่วยความจำใหม่ให้ vector หาก buffer เดิมเต็ม ถ้าเกิดแบบนั้น `first` ก็จะกลายเป็น dangling pointer (ตัวชี้ที่ชี้ไปยังหน่วยความจำที่ไม่ถูกต้อง) และการใช้งานมันอาจทำให้โปรแกรมล่ม (หรือแย่กว่านั้น อาจเกิด security vulnerability)

> **🔍 ทำไมถึงเกิดปัญหา?** เมื่อ `push_back` ทำงาน ถ้า vector เต็ม มันจะขอหน่วยความจำใหม่ที่ใหญ่กว่า และย้ายข้อมูลทั้งหมดไปที่ใหม่ ทำให้ `&v[0]` ที่เคยชี้ไปที่เดิม กลายเป็นตัวชี้ที่ไม่มีความหมาย (ชี้ไปที่หน่วยความจำที่ถูกปล่อยคืนไปแล้ว)

บั๊กแบบนี้เกิดขึ้นได้ก็ต่อเมื่อมีสองสิ่งเกิดขึ้นพร้อมกันเท่านั้น นั่นคือ **aliasing** (มีหลายชื่ออ้างอิงถึงข้อมูลเดียวกัน) และ **mutation** (มีการเปลี่ยนแปลงข้อมูล) ในโค้ดข้างต้น `first` กับ buffer ของ `v` เป็น aliases ของกันและกัน และเมื่อเรียก `push_back` buffer นั้นก็ถูกใช้เพื่อทำ mutation ดังนั้น หลักการสำคัญของ type system ของ Rust คือ **ห้าม mutation เมื่อมี aliasing เกิดขึ้น** เครื่องมือหลักที่ใช้บรรลุเป้าหมายนี้ก็คือแนวคิดเรื่อง **ownership**

## Ownership (การเป็นเจ้าของ)

แล้วในทางปฏิบัติหมายความว่าอย่างไร? ลองดูตัวอย่างนี้

```rust
fn work_on_vector(v: Vec<i32>) { /* do something */ }

fn ownership_demo() {
    let v = vec![1,2,3,4];
    work_on_vector(v);
    /* println!("The first element is: {}", v[0]); */ /* BAD! */
}
```

Rust ให้ความหมายพิเศษกับ argument ของ `work_on_vector` นั่นคือ ฟังก์ชันนี้สามารถถือว่าตัวเองเป็นเจ้าของ `v` ทั้งหมด และทำอะไรกับมันก็ได้ เมื่อ `work_on_vector` ทำงานเสร็จ ก็ไม่มีใครต้องการใช้ `v` อีก มันจึงถูกลบทิ้ง (รวมถึง buffer บน heap ด้วย) การส่ง `Vec<i32>` ไปให้ `work_on_vector` ถือเป็นการ **โอน ownership (move)** เดิมมีใครบางคนเป็นเจ้าของ vector นั้น แต่ตอนนี้ได้มอบให้ไปแล้ว และไม่มีสิทธิ์ใช้งานมันอีกต่อไป

> **🔍 Move vs Copy**
>
> - `Copy` = คัดลอกค่า bitwise (สำหรับข้อมูลเล็กๆ เช่น `i32`, `f32`, `bool`) ของเดิมยังใช้ได้
> - `Move` = ย้าย ownership ไปที่ใหม่ ของเดิมใช้ไม่ได้แล้ว (สำหรับข้อมูลใหญ่ เช่น `Vec`, `String`)
>
> Rust จะเลือกให้อัตโนมัติ ถ้า type มี trait `Copy` จะ copy, ถ้าไม่มีจะ move

เปรียบเหมือนเวลาคุณให้หนังสือเพื่อน คุณไม่สามารถไปบ้านเขาวันรุ่งขึ้นแล้วเอาหนังสือคืนได้หรอก มันไม่ใช่ของคุณแล้ว Rust ทำให้แน่ใจว่าคุณจะไม่ฝ่าฝืนกฎนี้ ลองเปิดบรรทัดที่คอมเมนต์ไว้ใน `ownership_demo` ดูสิ Rust จะบอกว่า `v` ถูก **moved** ไปแล้ว ซึ่งหมายความว่า ownership ถูกโอนไปที่อื่นแล้ว ในกรณีนี้ buffer ที่เก็บข้อมูลไม่มีตัวตนอยู่แล้วด้วยซ้ำ โชคดีที่ Rust จับปัญหานี้ได้ โดยพื้นฐานแล้ว ownership ช่วยกำจัด aliasing ทำให้ปัญหาประเภทที่กล่าวมาข้างต้นเกิดขึ้นไม่ได้

> **💡 สังเกต** ใน Part 02 เมื่อเราใช้ `self` (ไม่มี `&`) ใน method นั่นคือการรับ ownership แบบเต็มๆ (move) เช่นกัน เช่น `fn print(self)` ใน `NumberOrNothing`

## Borrowing: Shared References (&T)

ถ้าคุณย้อนกลับไปดูตัวอย่าง `vec_min` ในบทก่อน แล้วพยายามเรียกใช้ฟังก์ชันนั้นสองครั้ง คุณจะเจอข้อผิดพลาดเดียวกัน เพราะ `vec_min` ต้องการให้ผู้เรียกใช้โอน ownership ของ vector ให้ ดังนั้น เมื่อ `vec_min` ทำงานเสร็จ vector ทั้งหมดก็ถูกลบทิ้ง ซึ่งไม่ใช่สิ่งที่เราต้องการแน่นอน แล้วเราจะให้ `vec_min` เข้าถึง vector โดยที่ยังคง ownership ไว้ได้อย่างไร?

Rust เรียกสิ่งนี้ว่า **reference** ไปยัง vector และถือว่า references คือการ **ยืม (borrowing)** ownership มันทำงานคล้ายกับการยืมในชีวิตจริง เมื่อเพื่อนยืมหนังสือไป เพื่อนก็มีสิทธิ์ใช้มัน (ส่วนคุณไม่มี) ตราบใดที่หนังสือยังถูกยืมอยู่ เพื่อนของคุณอาจให้คนอื่นยืมต่อก็ได้ แต่ท้ายที่สุดแล้ว เพื่อนต้องคืนหนังสือให้คุณ และตอนนั้นคุณก็จะกลับมามีอำนาจเต็มที่อีกครั้ง

Rust แบ่ง references เป็นสองประเภท ประเภทแรกคือ **shared reference** อันนี้อุปมาหนังสือจะใช้ไม่ค่อยได้เท่าไหร่... คุณสามารถให้ shared reference ของข้อมูลชิ้นเดียวกันแก่หลายคนพร้อมกันได้ และทุกคนก็เข้าถึงข้อมูลได้ แน่นอนว่าสิ่งนี้ทำให้เกิด aliasing ดังนั้นเพื่อรักษาคำมั่นสัญญาเรื่องความปลอดภัย Rust จึงไม่อนุญาตให้ทำ mutation ผ่าน shared reference โดยทั่วไป

เรามาเขียน `vec_min` ใหม่ให้ทำงานกับ shared reference ไปยัง vector กันเถอะ เขียนเป็น `&Vec<i32>`

> **💡 หมายเหตุ** ตอนนี้เราเข้าใจหลักการแล้ว เราจะเปลี่ยนจาก `SomethingOrNothing` มาใช้ `Option` ตามมาตรฐานของ Rust เพื่อให้เห็นว่าโค้ดจริงเขียนอย่างไร

```rust
fn vec_min(v: &Vec<i32>) -> Option<i32> {
    use std::cmp;
    let mut min = None;
```

ครั้งนี้เราขอ iterator สำหรับ vector `v` อย่างชัดเจน method `iter` จะยืม vector ที่มันทำงานอยู่ และคืนค่าเป็น shared references ไปยังสมาชิกต่างๆ

```rust
    for e in v.iter() {
```

ในลูป `e` จะมี type เป็น `&i32` เราจึงต้อง **dereference** มันด้วย `*` เพื่อให้ได้ `i32`

```rust
        min = Some(match min {
            None => *e,
            Some(n) => cmp::min(n, *e)
        });
    }
    min
}
```

เนื่องจาก `vec_min` ไม่ได้รับ ownership ของ vector อีกต่อไป เราจึงเรียกมันหลายครั้งกับ vector เดียวกันได้ และยังทำอะไรแบบนี้ได้ด้วย

```rust
fn shared_ref_demo() {
    let v = vec![5,4,3,2,1];
    let first = &v[0];
    vec_min(&v);
    vec_min(&v);
    println!("The first element is: {}", *first);
}
```

เกิดอะไรขึ้นที่นี่? ก่อนอื่น `&` คือวิธีที่คุณให้ยืม ownership - ตัวดำเนินการนี้สร้าง shared reference `shared_ref_demo` สร้าง shared references สามตัวไปยัง `v`: reference `first` เริ่มต้นในบรรทัดที่ 2 ของฟังก์ชันและอยู่ไปจนจบ อีกสอง references ที่สร้างขึ้นเพื่อเรียก `vec_min` มีอายุเพียงแค่ระหว่างการเรียกใช้แต่ละครั้งเท่านั้น

> **💡 Lifetime** ทุก reference ใน Rust มี "อายุ" (lifetime) คอมไพเลอร์จะตรวจสอบว่า reference ไม่มีอายุยาวเกินข้อมูลที่ชี้ไป ในโค้ดส่วนใหญ่ Rust อนุมาน lifetime อัตโนมัติ ไม่ต้องเขียนเอง

ในทางเทคนิคแล้ว references ก็คือ pointers สังเกตว่าเนื่องจาก `vec_min` ได้รับเพียง shared reference Rust จึงรู้ว่ามันไม่สามารถทำ mutation กับ `v` ได้ ดังนั้น pointer ที่ชี้เข้าไปใน buffer ของ `v` ซึ่งถูกสร้างขึ้นก่อนเรียก `vec_min` จึงยังคงใช้งานได้อยู่

## Borrowing: Mutable References (&mut T)

มีวิธีที่สองในการยืม นั่นคือ reference ประเภทที่สอง **mutable reference** นี่คือ reference ที่มาพร้อมคำมั่นว่าไม่มีใครอื่นมีสิทธิ์เข้าถึงสิ่งที่ถูกอ้างอิงแต่อย่างใด - ต่างจาก shared references ที่ไม่มี aliasing กับ mutable references ดังนั้นจึงปลอดภัยเสมอที่จะทำ mutation ผ่าน reference แบบนี้ เพราะไม่มี reference อื่นชี้ไปที่ข้อมูลชิ้นเดียวกัน เราอาจเรียกมันว่า **unique reference** ก็ได้ แต่นั่นไม่ใช่ชื่อทางการ

ลองดูตัวอย่างฟังก์ชันที่เพิ่มค่าทุกสมาชิกใน vector ขึ้น 1 type `&mut Vec<i32>` คือ type ของ mutable references ไปยัง `Vec<i32>` เนื่องจาก reference นี้เปลี่ยนแปลงได้ เราจึงใช้ mutable iterator ซึ่งให้ mutable references ไปยังสมาชิกต่างๆ

```rust
fn vec_inc(v: &mut Vec<i32>) {
    for e in v.iter_mut() {
        *e += 1;  // dereference e แล้วบวก 1
    }
}
```

> **💡 สังเกต** `*e += 1` คือการ dereference pointer `e` เพื่อเข้าถึงค่าจริง แล้วเพิ่มค่าขึ้น 1 ถ้าเขียน `e += 1` จะ error เพราะ `e` เป็น `&mut i32` ไม่ใช่ `i32`

นี่คือตัวอย่างการเรียกใช้ `vec_inc`

```rust
fn mutable_ref_demo() {
    let mut v = vec![5,4,3,2,1];
    /* let first = &v[0]; */
    vec_inc(&mut v);
    vec_inc(&mut v);
    /* println!("The first element is: {}", *first); */ /* BAD! */
}
```

`&mut` คือตัวดำเนินการสร้าง mutable reference เราต้องทำเครื่องหมาย `v` ว่าเป็น `mut` เพื่อสร้าง reference แบบนี้ แม้ว่าเราจะเป็นเจ้าของ `v` ทั้งหมด Rust ก็ยังพยายามป้องกันไม่ให้เราเปลี่ยนแปลงสิ่งต่างๆ โดยไม่ตั้งใจ ดังนั้นตัวแปรที่เป็นของเราและต้องการเปลี่ยนแปลงจึงต้องมีคำว่า `mut` กำกับไว้

เนื่องจาก reference ที่ส่งไปให้ `vec_inc` มีอายุเพียงแค่ระหว่างเรียกใช้ฟังก์ชันเท่านั้น เราจึงเรียก `vec_inc` กับ vector เดียวกันได้สองครั้ง ระยะเวลาของทั้งสอง references ไม่ทับซ้อนกัน เราจึงไม่มี mutable reference มากกว่าหนึ่งตัวเลย - เราเพียงแค่ยืม `v` ทีละหนึ่งตัว

อย่างไรก็ตาม เราไม่สามารถสร้าง shared reference ที่มีอายุครอบคลุมการเรียกใช้ `vec_inc` ได้ ลองเปิดบรรทัดที่คอมเมนต์ไว้ดูสิ Rust จะบ่น นี่เป็นเพราะ `vec_inc` อาจเปลี่ยนแปลงโครงสร้างของ vector (เช่น อาจเพิ่มหรือลบสมาชิก) ทำให้ reference `first` อาจไม่ถูกต้องได้ กล่าวอีกนัยหนึ่ง Rust ปกป้องเราจากบั๊กแบบในโค้ด C++ ข้างต้น

ผมบอกไปข้างต้นว่าการมี mutable reference หมายถึงไม่มี aliasing แต่ถ้าคุณดูโค้ดข้างต้นอย่างละเอียด คุณอาจบอกว่า "เดี๋ยว `v` ใน `mutable_ref_demo` กับ `v` ใน `vec_inc` ไม่ใช่ aliases กันหรือ?" และคุณพูดถูก พวกมันเป็น aliases กัน แต่ว่า `v` ใน `mutable_ref_demo` ใช้งานไม่ได้จริงๆ มันไม่ active ตราบใดที่ `v` ถูกยืมอยู่ Rust จะไม่ยอมให้คุณทำอะไรกับมันเลย

## สรุป: กฎสามข้อของ Rust

ระบบ ownership และ borrowing ของ Rust บังคับใช้กฎสามข้อนี้:

| กฎ | คำอธิบาย |
|-----|---------|
| **1. มีเจ้าของเดียว** | ข้อมูลแต่ละชิ้นมีเจ้าของเพียงคนเดียวเสมอ |
| **2. หรือยืมหลายคน (shared)** | ถ้ามี shared reference ที่ active อยู่ การเข้าถึงข้อมูลแบบ active อื่นๆ ทั้งหมดจะต้องเป็น shared reference เช่นกัน (อ่านอย่างเดียว, หลายคน) |
| **3. หรือยืมคนเดียวที่แก้ไขได้ (mutable)** | ถ้ามี mutable reference ที่ active อยู่ ไม่มีใครอื่นสามารถเข้าถึงข้อมูลนั้นแบบ active ได้ (เขียนได้, คนเดียว) |

> **💡 จำง่ายๆ:** "หลายคนอ่าน หรือ คนเดียวเขียน" (Many readers OR one writer) - ไม่มีทางที่จะมีทั้งสองอย่างพร้อมกันได้

### ตารางสรุปการใช้งาน

| สถานการณ์ | ใช้อะไร | ตัวอย่าง |
|-----------|---------|---------|
| ต้องการ ownership เต็ม (ย้ายไปเลย) | `T` | `fn foo(v: Vec<i32>)` |
| แค่อ่านข้อมูล ไม่แก้ไข | `&T` | `fn foo(v: &Vec<i32>)` |
| ต้องการแก้ไขข้อมูล | `&mut T` | `fn foo(v: &mut Vec<i32>)` |

ปรากฏว่า เมื่อรวมกับความสามารถด้าน abstraction ของ Rust นี่คือกลไกที่ทรงพลังมากในการจัดการปัญหาหลายอย่างที่เกินกว่าความปลอดภัยของหน่วยความจำขั้นพื้นฐาน คุณจะได้เห็นตัวอย่างบางส่วนในเร็วๆ นี้

**แบบฝึกหัด 04.2:** เขียนฟังก์ชัน `vec_double` ที่รับ `&mut Vec<i32>` และคูณทุก element ด้วย 2 จากนั้นทดสอบเรียกใช้สองครั้งติดกัน

<summary>เฉลย</summary>

```rust
fn vec_double(v: &mut Vec<i32>) {
    for e in v.iter_mut() {
        *e *= 2;
    }
}

fn main() {
    let mut v = vec![1, 2, 3, 4, 5];
    vec_double(&mut v);
    vec_double(&mut v); // ใช้ได้เพราะ reference แต่ละครั้งไม่ทับซ้อนกัน
    println!("{:?}", v); // [4, 8, 12, 16, 20]
}
```

</details>

**แบบฝึกหัด 04.3 (หาข้อผิดพลาด):** โค้ดต่อไปนี้มีข้อผิดพลาดอะไร? แก้ไขอย่างไร?

```rust
fn main() {
    let v = vec![1, 2, 3, 4, 5];
    let first = &v[0];
    vec_double(&mut v); // สมมติว่ามีฟังก์ชันนี้จาก 04.2
    println!("First: {}", first);
}
```

<details>
<summary>เฉลย</summary>

```rust
// ข้อผิดพลาด: มี shared reference `first` อยู่ แต่พยายามสร้าง mutable reference
// แก้ไข: ใช้งาน `first` ให้เสร็จก่อน ค่อยเรียก vec_double

fn main() {
    let mut v = vec![1, 2, 3, 4, 5];
    {
        let first = &v[0];
        println!("First: {}", first);
    } // first หมดอายุที่นี่
    
    vec_double(&mut v); // ตอนนี้ใช้ได้แล้ว
    println!("{:?}", v);
}
```

</details>

**แบบฝึกหัด 04.4 (ท้าทาย):** เขียนฟังก์ชัน `vec_min_max` ที่รับ `&Vec<i32>` และคืนค่า tuple `(Option<i32>, Option<i32>)` แทนค่าต่ำสุดและสูงสุด โดยไม่ clone vector

<details>
<summary>เฉลย</summary>

```rust
fn vec_min_max(v: &Vec<i32>) -> (Option<i32>, Option<i32>) {
    use std::cmp;
    let mut min = None;
    let mut max = None;
    
    for e in v.iter() {
        min = Some(match min {
            None => *e,
            Some(n) => cmp::min(n, *e),
        });
        max = Some(match max {
            None => *e,
            Some(n) => cmp::max(n, *e),
        });
    }
    
    (min, max)
}
```

</details>
