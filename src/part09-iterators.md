# Rust-101, ส่วนที่ 09: ตัววนซ้ำ (Iterators)

```rust
use part05::BigInt;
```

ในส่วนต่อไปนี้ เราจะมาดูลึกเข้าไปในกลไกตัววนซ้ำ (iterator mechanism) ของ Rust และทำให้ `BigInt` ของเราเข้ากันได้กับลูป `for` แน่นอนว่า นี่คือเรื่องของการอิมพลีเมนท์เทรตบางอย่างอีกครั้ง โดยเฉพาะอย่างยิ่ง ตัววนซ้ำคือสิ่งที่อิมพลีเมนท์เทรต `Iterator` อย่างที่คุณเห็นในเอกสาร เทรตนี้บังคับฟังก์ชันเดียวคือ `next` ที่คืนค่า `Option<Self::Item>` โดยที่ `Item` คือชนิดข้อมูลสมทบ (associated type) ที่ถูกเลือกโดยการอิมพลีเมนท์ (ยังมีเมธอดอีกมากมายที่เตรียมไว้ให้สำหรับ `Iterator` แต่ทั้งหมดมีการอิมพลีเมนท์ตั้งต้นมาให้แล้ว ดังนั้นเราไม่ต้องกังวลกับพวกมันตอนนี้)

สำหรับกรณีของ `BigInt` เราต้องการให้ตัววนซ้ำของเราวนซ้ำไปบนหลักเลขในลำดับปกติทางสัญลักษณ์ (notational order): หลักที่มีนัยสำคัญมากที่สุด (most-significant digit) มาก่อน ดังนั้น เราต้องเขียนชนิดข้อมูลบางอย่าง และอิมพลีเมนท์ `Iterator` ให้มัน โดยที่ `next` จะคืนค่าเลขโดดทีละตัว ชัดเจนว่า ตัววนซ้ำต้องสามารถเข้าถึงตัวเลขที่มันวนซ้ำได้ และต้องจำตำแหน่งปัจจุบันของมันได้ อย่างไรก็ตาม มันไม่สามารถ *เป็นเจ้าของ* `BigInt` ได้ เพราะไม่อย่างนั้นตัวเลขจะหายไปหลังจากการวนซ้ำ! นั่นย่อมแย่แน่นอน ทางเลือกเดียวคือให้ตัววนซ้ำยืมตัวเลขมา ดังนั้นมันจึงรับ reference ไป

ในการเขียนสิ่งนี้ออกมา เราต้องระบุอายุขัย (lifetime) ของ reference อย่างชัดแจ้งอีกครั้ง: เรามีแค่ `Iter` เฉยๆ ไม่ได้ เราต้องมี `Iter<'a>` ที่ยืมตัวเลขมาเป็นเวลา `'a` นี่เป็นตัวอย่างแรกของเราสำหรับชนิดข้อมูล (data-type) ที่มีความเป็นพหุสัณฐาน (polymorphic) ใน lifetime ซึ่งตรงกันข้ามกับ type

`usize` ที่นี่คือชนิดข้อมูลของจำนวนนับบวกขนาดเท่า pointer (unsigned, pointer-sized numbers) โดยทั่วไปมันคือชนิดข้อมูลของ "ความยาวของสิ่งต่างๆ" โดยเฉพาะอย่างยิ่ง มันคือชนิดข้อมูลของความยาวของ `Vec` และดังนั้นจึงเป็นชนิดข้อมูลที่ถูกต้องในการเก็บออฟเซ็ตเข้าไปในเวกเตอร์ของเลขโดด

```rust
pub struct Iter<'a> {
    num: &'a BigInt,
    idx: usize, // ดัชนีของตัวเลขตัวสุดท้ายที่ถูกคืนค่าไปแล้ว
}
```

ตอนนี้เราเพียบพร้อมที่จะอิมพลีเมนท์ `Iterator` ให้กับ `Iter` แล้ว

```rust
impl<'a> Iterator for Iter<'a> {
    // เราเลือกชนิดข้อมูลของสิ่งที่เราวนซ้ำ ให้เป็นชนิดข้อมูลของเลขโดด นั่นคือ `u64`
    type Item = u64;

    fn next(&mut self) -> Option<u64> {
        // ก่อนอื่น ตรวจสอบว่ามีเลขโดดเหลือให้คืนค่าอีกหรือไม่
        if self.idx == 0 {
            // เราคืนค่าเลขโดดไปหมดแล้ว ไม่มีอะไรต้องทำ
            None
        } else {
            // มิฉะนั้น: ลดค่าดัชนี และคืนค่าเลขโดดตัวถัดไป
            self.idx = self.idx - 1;
            Some(self.num.data[self.idx])
        }
    }
}
```

สิ่งที่เราต้องการตอนนี้คือฟังก์ชันที่สร้างตัววนซ้ำเช่นนี้สำหรับ `BigInt` ที่กำหนดให้

```rust
impl BigInt {
    // สังเกตว่าเมื่อเราเขียนชนิดข้อมูลของ `iter` เราไม่ต้องให้พารามิเตอร์ lifetime
    // ของ `Iter` จริงๆ ก็เหมือนกับกรณีของฟังก์ชันที่คืนค่า reference คุณสามารถ
    // ละ lifetime (elide the lifetime) ได้ กฎสำหรับการเติม lifetime เหมือนกันเป๊ะ (ดูส่วน
    // สุดท้ายของส่วนที่ 06)
    fn iter(&self) -> Iter {
        Iter { num: self, idx: self.data.len() }
    }
}
```

ในที่สุดเราก็พร้อมที่จะวนซ้ำแล้ว! อย่าลืมแก้ไข `main.rs` เพื่อรันฟังก์ชันนี้

```rust
pub fn main() {
    let b = BigInt::new(1 << 63) + BigInt::new(1 << 16) + BigInt::new(1 << 63);
    for digit in b.iter() {
        println!("{}", digit);
    }
}
```

แน่นอนว่า เราไม่จำเป็นต้องใช้ `for` เพื่อประยุกต์ใช้ตัววนซ้ำ เรายังสามารถเรียก `next` อย่างชัดแจ้งได้ด้วย

```rust
fn print_digits_v1(b: &BigInt) {
    let mut iter = b.iter();
    // `loop` คือคีย์เวิร์ดสำหรับลูปที่ไม่มีเงื่อนไข: มันทำงานไม่รู้จบ หรือจนกว่าคุณจะกระโดด
    // ออกจากมันด้วย `break` หรือ `return`
    loop {
        // แต่ละครั้งที่เราผ่านลูป เราวิเคราะห์สมาชิกตัวถัดไปที่นำเสนอโดยตัววนซ้ำ
        // - จนกว่ามันจะหยุด
        match iter.next() {
            None => break,
            Some(digit) => println!("{}", digit)
        }
    }
}
```

ตอนนี้ ปรากฏว่าการรวมตัวกันของการทำลูปและการจับคู่รูปแบบ (pattern matching) นี้มันเจอบ่อยมาก และ Rust ก็เตรียมน้ำตาลทางไวยากรณ์ (syntactic sugar) ที่สะดวกสบายไว้ให้

```rust
fn print_digits_v2(b: &BigInt) {
    let mut iter = b.iter();
    // `while let` ทำการจับคู่รูปแบบที่กำหนดให้ในทุกรอบของลูป และยกเลิก
    // ลูปถ้ารูปแบบไม่ตรง (match ไม่ติด) ยังมี `if let` อีกด้วย ซึ่งทำงานคล้ายกัน แต่แน่นอนว่า
    // ไม่มีส่วนของการวนลูป
    while let Some(digit) = iter.next() {
        println!("{}", digit)
    }
}
```

>
> **แบบฝึกหัด 09.2**: เขียนฟังก์ชัน `iter_ldf` ที่วนซ้ำไปบนเลขโดดโดยเอาเลขโดดที่มีนัยสำคัญน้อยที่สุด (least-significant digits) ขึ้นก่อน เขียนกรณีทดสอบสำหรับมันด้วย

## การทำให้ตัววนซ้ำใช้การไม่ได้ และอายุขัย (Iterator invalidation and lifetimes)

คุณอาจจะแปลกใจที่เราต้องระบุ lifetime อย่างชัดแจ้งเมื่อเราเขียน `Iter` แน่นอนว่า ด้วยการที่ lifetime ปรากฏอยู่ในทุก reference ใน Rust นี่จึงเป็นความสม่ำเนอ (consistent) แต่เราได้อย่างน้อยอะไรกลับมาจากภาระในการระบุเพิ่มเติมนี้ไหม? (ขอบคุณที่ภาระนี้เกิดขึ้นเฉพาะตอนที่เรานิยามชนิดข้อมูล และไม่ใช่ตอนที่เรานิยามฟังก์ชัน - ซึ่งโดยทั่วไปเจอบ่อยกว่ามาก)

คำตอบของคำถามนี้คือ ได้! แง่มุมเฉพาะเจาะจงของแนวคิดเรื่อง lifetime นี้ช่วยให้ Rust กำจัดปัญหาของ **การทำให้ตัววนซ้ำใช้การไม่ได้ (iterator invalidation)** พิจารณาชิ้นส่วนของโค้ดต่อไปนี้

```rust
fn iter_invalidation_demo() {
    let mut b = BigInt::new(1 << 63) + BigInt::new(1 << 16) + BigInt::new(1 << 63);
    for digit in b.iter() {
        println!("{}", digit);
        /* b = b + BigInt::new(1); */                                 /* แย่มาก! */
    }
}
```

ถ้าคุณเปิดใช้งานบรรทัดที่แย่ Rust จะปฏิเสธโค้ดนั้น ทำไมล่ะ? ปัญหาคือเรากำลังแก้ไขตัวเลขในขณะที่กำลังวนซ้ำบนมัน ในภาษาอื่นๆ สิ่งนี้สามารถส่งผลกระทบได้สารพัด ตั้งแต่ข้อมูลไม่สอดคล้องกัน (inconsistent data) หรือการโยน exception (Java) ไปจนถึง pointer แย่ๆ ถูก dereferenced (C++) อย่างไรก็ตาม Rust สามารถตรวจจับสถานการณ์นี้ได้ เมื่อคุณเรียก `iter` คุณต้องยืม `b` สำหรับ lifetime `'a` ช่วงหนึ่ง และคุณได้รับ `Iter<'a>` นี่คือตัววนซ้ำที่ใช้งานได้แค่สำหรับ lifetime `'a` เท่านั้น ดีใจที่เรามีคำระบุนี้ไว้เพื่อสร้างคำกล่าวนั้น Rust บังคับว่า `'a` ครอบคลุมทุกการเรียก `next` ซึ่งหมายความว่ามันต้องครอบคลุมลูป ดังนั้น `b` จึงถูกยืมตลอดระยะเวลาของลูป และเราไม่สามารถแก้ไขค่า (mutate) มันได้ นี่เป็นอีกตัวอย่างหนึ่งว่าการรวมตัวกันของ mutation และ aliasing นำไปสู่ผลกระทบที่ไม่พึงประสงค์ได้อย่างไร (ไม่จำเป็นต้องโปรแกรมพังเสมอไป นึกถึง Java) ซึ่ง Rust ป้องกันได้อย่างสำเร็จงดงาม

## เทรตการแปลงเป็นตัววนซ้ำ (Iterator conversion trait)

ถ้าคุณเปรียบเทียบลูป `for` ใน `main` ข้างต้นอย่างใกล้ชิด กับลูปใน `part06::vec_min` คุณจะสังเกตเห็นว่าเราสามารถเขียน `for e in v` ได้ก่อนหน้านี้ แต่ตอนนี้เราต้องเรียก `iter` ทำไมถึงเป็นอย่างนั้น? ก็เพราะน้ำตาล `for` ไม่ได้ผูกติดกับ `Iterator` จริงๆ แต่เรียกร้องการอิมพลีเมนท์ `IntoIterator` แทน นั่นเป็นเทรตของชนิดข้อมูลที่จัดเตรียมฟังก์ชันการแปลงให้เป็นตัววนซ้ำบางชนิด เทรตการแปลง (conversion traits) เหล่านี้เป็นรูปแบบที่พบบ่อยใน Rust: แทนที่จะเรียกร้องว่าบางอย่าง *เป็น* ตัววนซ้ำ หรือสตริง หรืออะไรก็ตาม; เราเรียกร้องว่าบางอย่าง *สามารถถูกแปลงเป็น* ตัววนซ้ำ/สตริง/อะไรก็ตาม ได้ สิ่งนี้ให้ความสะดวกสบายคล้ายกับการโอเวอร์โหลดฟังก์ชัน: ฟังก์ชันสามารถถูกเรียกด้วยชนิดข้อมูลที่แตกต่างกันมากมาย โดยการอิมพลีเมนท์เทรตเช่นนี้ให้กับชนิดข้อมูลของคุณ คุณยังสามารถทำให้ชนิดข้อมูลของคุณเองทำงานได้อย่างลื่นไหลกับฟังก์ชันไลบรารีที่มีอยู่แล้ว โดยปกติสำหรับ Rust นามธรรม (abstraction) นี้มาพร้อมกับไม่มีค่าใช้จ่าย (zero cost): ถ้าข้อมูลของคุณเป็นชนิดข้อมูลที่ถูกต้องอยู่แล้ว ฟังก์ชันการแปลงจะไม่ทำอะไรเลยและถูก optimize ทิ้งไปอย่างง่ายดาย

ถ้าคุณดูที่เอกสารของ `IntoIterator` คุณจะสังเกตเห็นว่าฟังก์ชัน `into_iter` ที่มันจัดเตรียมให้ จริงๆ แล้วบริโภค (consumes) อาร์กิวเมนต์ของมัน ดังนั้นเราอิมพลีเมนท์เทรตให้กับ reference ของตัวเลข เพื่อที่ตัวเลขจะไม่หายไปหลังจากการวนซ้ำ

```rust
impl<'a> IntoIterator for &'a BigInt {
    type Item = u64;
    type IntoIter = Iter<'a>;
    fn into_iter(self) -> Iter<'a> {
        self.iter()
    }
}
```

ด้วยสิ่งนี้ ตอนนี้คุณสามารถแทนที่ `b.iter()` ใน `main` ด้วย `&b` ได้แล้ว ลองดูสิ! เดี๋ยวนะ `&b` เหรอ? ทำไมล่ะ? ก็เพราะเราอิมพลีเมนท์ `IntoIterator` ให้กับ `&BigInt` ไงล่ะ ถ้าเราอยู่ในที่ที่ `b` ถูกยืมมาอยู่แล้ว เราสามารถแค่ทำ `for digit in b` ได้เลย แต่ถ้าเราเป็นเจ้าของ `b` เราต้องสร้าง reference ไปยังมัน ทางเลือกอื่น เราอาจจะอิมพลีเมนท์ `IntoIterator` ให้กับ `BigInt` - ซึ่ง อย่างที่กล่าวไปแล้ว จะหมายความว่า `b` ถูกบริโภคโดยการวนซ้ำจริงๆ และหายไปเลย สิ่งนี้สามารถเกิดขึ้นได้ง่ายมาก ตัวอย่างเช่น กับ `Vec`: ทั้ง `Vec` และ `&Vec` (และ `&mut Vec`) อิมพลีเมนท์ `IntoIterator` ดังนั้นถ้าคุณทำ `for e in v` และ `v` มีชนิดข้อมูล `Vec` คุณก็จะได้รับความเป็นเจ้าของสมาชิกในระหว่างการวนซ้ำ - และทำลายเวกเตอร์ไปในกระบวนการ เราทำสิ่งนั้นจริงๆ ใน `part01::vec_min` แต่เราไม่แคร์ คุณสามารถเขียน `for e in &v` หรือ `for e in v.iter()` เพื่อหลีกเลี่ยงสิ่งนี้ได้
