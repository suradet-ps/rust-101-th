# Rust-101, ส่วนที่ 16: Unsafe Rust, Drop

```rust
use std::ptr;
use std::mem;
use std::marker::PhantomData;
```

ตามที่เราได้เห็น กฎที่ Rust กำหนดเพื่อรับประกันความปลอดภัยของหน่วยความจำ (memory safety) สามารถพาเราไปได้ไกลทีเดียว รูปแบบการเขียนโปรแกรม (programming patterns) จำนวนมากสามารถเขียนได้ภายใน safe Rust และที่สำคัญกว่านั้น โค้ดไลบรารีอย่างเช่น iterators หรือ threads สามารถใช้ประโยชน์จากระบบชนิดข้อมูลเพื่อให้แน่ใจถึงระดับความถูกต้องบางอย่างที่เหนือกว่าแค่ความปลอดภัยของหน่วยความจำพื้นฐาน

อย่างไรก็ตาม ก็ยังจะมีโปรแกรมที่คนเราไม่สามารถเขียนให้สอดคล้องกับ borrow checker ได้ และจะมีกรณีที่อาจจะเป็นไปได้ที่จะทำให้คอมไพเลอร์พอใจ แต่ต้องแลกมาด้วยต้นทุนรันไทม์ (run-time overhead) บางอย่าง อย่างที่เราเห็นกับ `RefCell` - ต้นทุนซึ่งอาจจะยอมรับไม่ได้ ในสถานการณ์เช่นนั้น เป็นไปได้ที่จะใช้ **unsafe Rust**: นั่นคือส่วนของภาษาที่รู้กันว่าเปิดประตูสู่การเข้าถึง pointer ที่ไม่ถูกต้อง (invalid pointer access) และปัญหาความปลอดภัยของหน่วยความจำแบบอื่นๆ ทั้งหมด แน่นอนว่า unsafe ยังหมายถึง "ที่นี่มีมังกร (Here Be Dragons)": คุณต้องพึ่งพาตัวเองเมื่อพูดถึงการทำให้มั่นใจว่าคุณจะไม่ทำอะไรพัง

เป้าหมายในกรณีเหล่านี้คือการจำกัดความไม่ปลอดภัย (unsafety) ไว้ในโมดูล local ชนิดข้อมูลอย่าง `Rc` และ `Vec` ถูกอิมพลีเมนท์โดยใช้ unsafe Rust แต่การใช้พวกมันอย่างที่เราทำนั้น (เชื่อว่า) ปลอดภัยอย่างสมบูรณ์

---

## Unsafe Code

เป็นตัวอย่าง ลองมาเขียนรายการโยงสองทาง (doubly-linked list) กัน ชัดเจนว่า โครงสร้างข้อมูลแบบนี้เกี่ยวข้องกับ aliasing และ mutation: ทุกโหนดในรายการถูกชี้โดยเพื่อนบ้านทางซ้ายและขวาของมัน แต่เราก็ยังต้องการแก้ไขโหนดเหล่านั้น เราอาจจะลองใช้การผสมผสานที่ชาญฉลาดของ `Rc` และ `RefCell` แต่นั่นจะลงเอยด้วยการน่ารำคาญใจทีเดียว - และมันจะก่อให้เกิดต้นทุนบางอย่าง สำหรับโครงสร้างข้อมูลระดับต่ำ (low-level) อย่าง doubly-linked list มันสมเหตุสมผลที่จะอิมพลีเมนท์เวอร์ชันที่มีประสิทธิภาพสักครั้งหนึ่ง ไว้อย่าง unsafe ภายใน แต่สามารถถูกใช้ได้โดยไม่มีความเสี่ยงใดๆ โดยโค้ดลูกค้าที่ปลอดภัย (safe client code)

ตามปกติ เราเริ่มโดยการนิยามชนิดข้อมูล ทุกอย่างถูกกำหนดพารามิเตอร์ (parameterized) โดยชนิดข้อมูล `T` ของข้อมูลที่ถูกเก็บในรายการ โหนดของรายการประกอบด้วยข้อมูล, และตัวชี้โหนด (node pointers) สองตัวสำหรับผู้มาก่อน (predecessor) และผู้สืบทอด (successor)

```rust
struct Node<T> {
    next: NodePtr<T>,
    prev: NodePtr<T>,
    data: T,
}
```

ตัวชี้โหนดคือ *mutable raw pointer* (พอยน์เตอร์ดิบแบบแก้ไขได้) ไปยังโหนด Raw pointers (`*mut T` และ `*const T`) คือสิ่งที่เทียบเท่าใน Rust ของ pointers ใน C ต่างจาก references พวกมันไม่ได้มาพร้อมกับการรับประกันใดๆ: Raw pointers สามารถเป็น null, หรือพวกมันสามารถชี้ไปยังขยะได้ พวกมันไม่มี lifetime

```rust
type NodePtr<T> = *mut Node<T>;
```

ตัว linked list เองเก็บ pointers ไปยังโหนดแรกและโหนดสุดท้าย นอกจากนี้ เราบอก Rust ว่าชนิดข้อมูลนี้จะเป็นเจ้าของข้อมูลของชนิดข้อมูล `T` ชนิดข้อมูล `PhantomData<T>` ไม่ได้เก็บอะไรในหน่วยความจำจริงๆ - มันมีขนาดเป็นศูนย์ อย่างไรก็ตาม ในทางตรรกะ Rust จะพิจารณาว่า `T` นั้นปรากฏอยู่ ในกรณีนี้ Rust รู้ว่าข้อมูลของชนิดข้อมูล `T` อาจจะถูก drop เมื่อไหร่ก็ตามที่ `LinkedList<T>` ถูก drop การ Dropping มีการตรวจสอบที่ละเอียดอ่อนมากมาย เพื่อให้แน่ใจว่าสิ่งต่างๆ จะไม่ผิดพลาด เพื่อให้สิ่งนี้ทำงานได้ Rust จำเป็นต้องรู้ว่าชนิดข้อมูลไหนมีศักยภาพที่จะถูก drop ใน safe Rust ทั้งหมดนี้ถูกอนุมาน (inferred) ได้โดยอัตโนมัติ แต่ที่นี่ เรามีแค่ `*mut Node<T>` และเราจำเป็นต้องบอก Rust ว่าเราเป็นเจ้าของข้อมูลดังกล่าวจริงๆ และจะทำการ drop มัน (สำหรับรายละเอียดที่รุ่งโรจน์กว่านี้ ดู blog post นี้)

```rust
pub struct LinkedList<T> {
    first: NodePtr<T>,
    last:  NodePtr<T>,
    _marker: PhantomData<T>,
}
```

ก่อนที่เราจะไปถึงเมธอดของ linked-list จริงๆ เราเขียนฟังก์ชันตัวช่วยสั้นๆ สองตัวที่แปลงระหว่าง mutable raw pointers และ boxed data ทั้งสองตัวใช้ `mem::transmute` ซึ่งสามารถแปลงอะไรก็ได้ไปเป็นอะไรก็ได้ โดยการแค่ตีความไบต์ใหม่ (re-interpreting the bytes) ชัดเจนว่า นั่นเป็นการปฏิบัติการที่ unsafe และต้องถูกใช้ด้วยความระมัดระวังอย่างยิ่ง - หรือจะให้ดีกว่านั้น ไม่ใช้เลย เอาจริงนะ ถ้าเป็นไปได้ทั้งหมด คุณไม่ควรใช้ `transmute` เลย

เรากำลังตั้งสมมติฐานที่นี่ว่า `Box` และ raw pointer มีการแทนค่าในหน่วยความจำเหมือนกัน ในอนาคต Rust จะจัดเตรียมการปฏิบัติการเช่นนี้ในไลบรารีมาตรฐาน แต่ API ที่แน่นอนยังคงอยู่ระหว่างการขัดเกลา

เราประกาศ `raw_into_box` ให้เป็น **unsafe function** เป็นการบอก Rust ว่าการเรียกฟังก์ชันนี้โดยทั่วไปไม่ปลอดภัย สิ่งนี้มอบพลัง unsafe ให้เราสำหรับเนื้อหา (body) ของฟังก์ชัน: เราสามารถ dereference (ยกเลิกการอ้างอิง) raw pointers, และ - สำคัญที่สุด - เราสามารถเรียก unsafe functions อื่นๆ (พลัง unsafe อื่นๆ จะไม่เกี่ยวข้องที่นี่ ไปอ่าน The Rustonomicon ถ้าคุณต้องการเรียนรู้ทั้งหมดเกี่ยวกับเรื่องนี้ แต่ขอเตือนไว้ก่อน - หนทางนั้นสู่ความบ้าคลั่ง (That Way Lies Madness))

ที่นี่ ผู้เรียก (caller) จะต้องรับรองว่า `r` เป็น pointer ที่ถูกต้อง และไม่มีใครอื่นที่มี pointer ไปยังข้อมูลนี้

```rust
unsafe fn raw_into_box<T>(r: *mut T) -> Box<T> {
    mem::transmute(r)
}
```

กรณีนี้แตกต่างเล็กน้อยสำหรับ `box_into_raw`: การแปลง `Box` ไปเป็น raw pointer นั้นปลอดภัยเสมอ มันแค่ทิ้งข้อมูลบางอย่าง (พวก ownership) ดังนั้นเราเก็บฟังก์ชันนี้ไว้เป็น safe และใช้ **unsafe block** ภายในฟังก์ชัน นี่คือคำสัญญา (ที่ไม่ได้ตรวจสอบ) ต่อคอมไพเลอร์ Rust ที่บอกว่าการเรียก `box_into_raw` แบบปลอดภัยไม่สามารถผิดพลาดได้ เรายังมีพลัง unsafe ใน unsafe block ด้วย

```rust
fn box_into_raw<T>(b: Box<T>) -> *mut T {
    unsafe { mem::transmute(b) }
}
```

ตอนนี้เราสามารถอิมพลีเมนท์เมธอดของ linked list ได้แล้ว

```rust
impl<T> LinkedList<T> {
    // linked list ใหม่จะประกอบด้วย null pointers เท่านั้น `PhantomData` คือวิธีที่เราสร้าง
    // `PhantomData<T>` ใดๆ
    pub fn new() -> Self {
        LinkedList { first: ptr::null_mut(), last: ptr::null_mut(), _marker: PhantomData }
    }

    // ฟังก์ชันนี้เพิ่มโหนดใหม่ไปยังตอนท้ายของรายการ
    pub fn push_back(&mut self, t: T) {
        // สร้างโหนดใหม่ และทำให้มันเป็น raw pointer การเรียก `box_into_raw` สละความเป็นเจ้าของ
        // ของ box ซึ่งสำคัญมาก: เราไม่ต้องการให้หน่วยความจำที่มันชี้ไปถูกคืน (deallocated)!
        let new = Box::new( Node { data: t, next: ptr::null_mut(), prev: self.last } );
        let new = box_into_raw(new);
        // อัปเดต pointers อื่นๆ ไปยังโหนดนี้
        if self.last.is_null() {
            debug_assert!(self.first.is_null());
            // รายการว่างอยู่ในขณะนี้ ดังนั้นเราต้องอัปเดต head pointer
            self.first = new;
        } else {
            debug_assert!(!self.first.is_null());
            // เราต้องอัปเดต `next` pointer ของโหนดหาง (tail node) เนื่องจาก Rust ไม่รู้ว่า
            // raw pointer ชี้ไปยังอะไรจริงๆ หรือเปล่า การ dereferencing pointer ดังกล่าวจึงเป็นการปฏิบัติการที่
            // unsafe ดังนั้น unsafe block นี้สัญญาว่า pointer จะถูกต้องจริงๆ
            unsafe { (*self.last).next = new; }
        }
        // ทำให้สิ่งนี้เป็นโหนดสุดท้าย
        self.last = new;
    }

    // **แบบฝึกหัด 16.1**: เพิ่มการปฏิบัติการอื่นๆ ให้กับ `LinkedList`: `pop_back`, `push_front` และ
    // `pop_front` เพิ่ม testcases สำหรับ `push_back` และฟังก์ชันทั้งหมดของคุณ ฟังก์ชัน `pop`
    // ควรรับ `&mut self` และคืนค่า `Option<T>`

    // ถัดไป เราจะจัดเตรียม iterator ฟังก์ชันนี้แค่สร้าง instance ของ
    // `IterMut` ซึ่งเป็นชนิดข้อมูล iterator ที่ทำงานจริงๆ
    pub fn iter_mut(&mut self) -> IterMut<T> {
        IterMut { next: self.first, _marker: PhantomData  }
    }
}
```

Iterator จำเป็นต้องเก็บอะไร? พูดอย่างเคร่งครัด ทั้งหมดที่มันต้องการคือ pointer ไปยังโหนดถัดไปที่มันกำลังจะไปเยี่ยม อย่างไรก็ตาม เราจะแน่ใจได้อย่างไรว่า pointer นี้ยังคงถูกต้อง? เราต้องจัดการเรื่องนี้ให้ถูกต้องด้วยตัวเราเอง เพราะเราได้ออกจากอาณาจักรที่ปลอดภัยของการยืมและความเป็นเจ้าของแล้ว จำได้ไหมว่าส่วนประกอบสำคัญสำหรับความปลอดภัยของ iterator คือการผูก lifetime ของ iterator เข้ากับ lifetime ของ reference ที่ใช้สำหรับ `iter_mut` เราจึงจะให้ `IterMut` มีพารามิเตอร์สองตัว: พารามิเตอร์ชนิดข้อมูลที่ระบุชนิดของข้อมูล, และพารามิเตอร์ lifetime ที่ระบุว่ารายการถูกยืมมาให้ iterator นานเท่าไหร่

เพื่อให้ Rust ยอมรับชนิดข้อมูลนี้ เราต้องเพิ่ม annotations อีกสองอย่าง ก่อนอื่นเลย เราต้องแน่ใจว่าข้อมูลในรายการมีชีวิตอยู่อย่างน้อยก็นานเท่ากับ iterator: ถ้าคุณเอา `T: 'a` ออก Rust จะบอกให้คุณใส่มันกลับเข้าไป และอย่างที่สอง Rust จะบ่นถ้า `'a` ไม่ได้ถูกใช้จริงๆ ใน struct มันไม่รู้ว่ามันควรจะทำอะไรกับ lifetime นั้น ดังนั้นเราใช้ `PhantomData` อีกครั้งเพื่อบอกมันว่าในแง่ของความเป็นเจ้าของ ชนิดข้อมูลนี้ทำการยืม (อย่างไม่ซ้ำใคร/uniquely) linked list จริงๆ สิ่งนี้ไม่มีผลทางการปฏิบัติงาน (operational effect) แต่มันหมายความว่า Rust สามารถอนุมานเจตนาที่เรามีตอนที่เพิ่มพารามิเตอร์ lifetime ที่ดูเหมือนไร้ประโยชน์นั้นได้

```rust
pub struct IterMut<'a, T> where T: 'a {
    next: NodePtr<T>,
    _marker: PhantomData<&'a mut LinkedList<T>>,
}
```

เมื่ออิมพลีเมนท์ `Iterator` สำหรับ `IterMut` ความจริงที่ว่าเรามี lifetime `'a` แปะอยู่ด้วยนั้นให้ผลตอบแทนทันที: เราคงจะไม่สามารถแม้แต่จะเขียนชนิดข้อมูล `Item` ลงไปได้ถ้าไม่มี lifetime นั้น

```rust
impl<'a, T> Iterator for IterMut<'a, T> {
    type Item = &'a mut T;

    fn next(&mut self) -> Option<Self::Item> {
        // การวนซ้ำจริงๆ นั้นตรงไปตรงมา: เมื่อเราไปถึง null pointer เราก็เสร็จสิ้น
        if self.next.is_null() {
            None
        } else {
            // ไม่อย่างนั้น เราสามารถแปลง pointer ถัดไปเป็น reference, รับ reference ไปยังข้อมูล
            // และอัปเดต iterator
            let next = unsafe { &mut *self.next };
            let ret = &mut next.data;
            self.next = next.next;
            Some(ret)
        }
    }
}
```

ใน `next` ข้างบน เราใช้ประโยชน์อย่างสำคัญจากสมมติฐานที่ว่า `self.next` เป็น null หรือไม่ก็เป็น pointer ที่ถูกต้อง สิ่งนี้ทำงานได้ก็เพราะถ้ามีใครพยายามลบสมาชิกออกจากรายการในระหว่างการ iteration เรารู้ว่า borrow checker จะจับพวกเขา: ถ้าพวกเขาเรียก `next` lifetime `'a` ที่เราเติมให้กับ iterator อย่างเทียมๆ นั้นจะต้องยังคง active อยู่ ซึ่งหมายความว่า mutable reference ที่ส่งให้กับ `iter_mut` ยังคง active ซึ่งหมายความว่าไม่มีใครสามารถลบอะไรออกจากรายการได้ พูดอีกอย่างคือ เราใช้ระบบชนิดข้อมูลที่สื่อความหมายได้มากของ Rust เพื่อประดับตกแต่ง unsafe implementation ของเราด้วยข้อมูลที่เพียงพอ เพื่อที่ Rust จะสามารถตรวจสอบการใช้งานของ linked-list ได้ ถ้าระบบชนิดข้อมูลอ่อนแอกว่านี้ เราคงไม่สามารถเขียน linked-list แบบข้างบนด้วยอินเทอร์เฟซที่ปลอดภัยได้!

> **แบบฝึกหัด 16.2**: เพิ่มเมธอด `iter` และชนิดข้อมูล `Iter` ที่จัดเตรียมการ iteration สำหรับ shared references เพิ่ม testcases สำหรับ iterators ทั้งสองชนิด

## `Drop`

Linked list ที่เราเขียนทำงานได้ค่อนข้างดีทีเดียวแล้ว แต่มีปัญหาอยู่อย่างหนึ่ง: เมื่อรายการถูก drop ไม่มีใครใส่ใจที่จะคืนหน่วยความจำ (deallocate) โหนดที่เหลืออยู่ ที่แย่กว่านั้น ถ้า `T` เองมี destructor ที่จำเป็นต้องทำความสะอาด มันจะไม่ถูกเรียกสำหรับสมาชิกที่เหลืออยู่ในรายการ เราจำเป็นต้องจัดการเรื่องนั้น

ใน Rust การเพิ่ม destructor สำหรับชนิดข้อมูลทำได้โดยการอิมพลีเมนท์เทรต `Drop` นี่คือเทรตที่พิเศษมาก มันสามารถถูกอิมพลีเมนท์สำหรับ nominal types (ชนิดข้อมูลที่มีชื่อ) เท่านั้น กล่าวคือ คุณไม่สามารถอิมพลีเมนท์ `Drop` สำหรับ `&mut T` ได้ คุณยังไม่สามารถจำกัดพารามิเตอร์ชนิดข้อมูลและ lifetime ได้มากไปกว่าที่ตัวชนิดข้อมูลทำ - การอิมพลีเมนท์ `Drop` ต้อง ينطبق (ใช้ได้) กับ instances ของ `LinkedList` ทั้งหมด

```rust
impl<T> Drop for LinkedList<T> {
    // ตัว destructor เองเป็นเมธอดที่รับ `self` ในรูปแบบที่ถูกยืมแบบแก้ไขได้ (mutably borrowed) มันไม่สามารถเป็นเจ้าของ
    // `self` ได้ เพราะไม่อย่างนั้น destructor ของ `self` จะถูกเรียกที่ตอนจบของฟังก์ชัน
    // ส่งผลให้เกิด infinite recursion (การเรียกซ้ำไม่สิ้นสุด)
    fn drop(&mut self) {
        let mut cur_ptr = self.first;
        while !cur_ptr.is_null() {
            // ใน destructor เราแค่วนซ้ำไปทั่วทั้งรายการ รับเอา (obtaining)
            // ความเป็นเจ้าของ (`Box`) ของทุกโหนดทีละตัว เมื่อ box ถูก drop มันจะเรียก destructor
            // บน `data` ถ้าจำเป็น และต่อมาปล่อย (free) โหนดบนฮีป เราเรียก `drop`
            // อย่างชัดแจ้งที่นี่เพื่อจุดประสงค์ในการทำเอกสารเท่านั้น
            let cur = unsafe { raw_into_box(cur_ptr) };
            cur_ptr = cur.next;
            drop(cur);
        }
    }
}
```
