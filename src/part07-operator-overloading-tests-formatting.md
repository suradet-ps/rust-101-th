# Rust-101, ส่วนที่ 07: การโอเวอร์โหลดตัวดำเนินการ, การทดสอบ, การจัดรูปแบบ

```rust
pub use part05::BigInt;
```

ด้วยความรู้ใหม่ของเราเรื่อง lifetimes ตอนนี้เราสามารถเขียนชนิดข้อมูลที่ต้องการของ `min` ได้แล้ว: เราต้องการให้ฟังก์ชันรับ reference สองตัวที่มี lifetime เดียวกัน และจากนั้นคืนค่า reference ที่มี lifetime นั้น ถ้า lifetime ขาเข้าทั้งสองแตกต่างกัน เราจะไม่รู้ว่าจะใช้ lifetime ไหนสำหรับผลลัพธ์

```rust
pub trait Minimum {
    fn min<'a>(&'a self, other: &'a Self) -> &'a Self;
}
```

ตอนนี้เราสามารถอิมพลีเมนท์ฟังก์ชันเจเนอริก `vec_min` ที่ทำงานบนเทรตข้างต้นได้ โค้ดค่อนข้างตรงไปตรงมา และ Rust ตรวจสอบว่า lifetime ทั้งหมดทำงานได้จริง สังเกตว่าเราไม่ต้องทำการคัดลอกใดๆ เลย!

```rust
pub fn vec_min<T: Minimum>(v: &Vec<T>) -> Option<&T> {
    let mut min: Option<&T> = None;
    for e in v {
        min = Some(match min {
            None => e,
            Some(n) => n.min(e)
        });
    }
    min
}
```

สังเกตว่าชนิดข้อมูลที่ส่งกลับ `Option<&T>` ในทางเทคนิค (พักเรื่องการยืมไว้ก่อน) คือ pointer ไปยัง `T` ที่อาจจะใช้การไม่ได้ (invalid) ก็ได้ กล่าวอีกนัยหนึ่ง มันเหมือนกับ pointer ใน C(++) หรือ Java ที่สามารถเป็น `NULL` ได้ อย่างไรก็ตาม ต้องขอบคุณที่ `Option` เป็น `enum` เราจึงไม่สามารถลืมตรวจสอบความถูกต้องของ pointer ได้ หลีกเลี่ยงปัญหาความปลอดภัยของ C(++) นอกจากนี้ ถ้าคุณกังวลเรื่องการเปลืองพื้นที่ สังเกตว่า Rust รู้ว่า `&T` ไม่มีทางเป็น `NULL` ได้ และดังนั้นจึงปรับแต่ง (optimizes) `Option<&T>` ให้มีขนาดไม่ใหญ่ไปกว่า `&T` กรณี `None` ถูกแทนด้วย `NULL` นี่เป็นอีกตัวอย่างที่ยอดเยี่ยมของ **Zero-cost Abstraction (นามธรรมที่ไม่มีค่าใช้จ่าย)**: `Option<&T>` เหมือนกับ pointer ใน C(++) เป๊ะๆ ถ้าคุณดูสิ่งที่เกิดขึ้นระหว่างการประมวลผล - แต่มันปลอดภัยกว่าในการใช้งานมาก

> **แบบฝึกหัด 07.1**: เพื่อให้ `vec_min` ของเราใช้งานได้กับ `BigInt` คุณจะต้องจัดเตรียมการอิมพลีเมนท์ของ `Minimum` คุณน่าจะสามารถคัดลอกโค้ดที่คุณเขียนสำหรับแบบฝึกหัด 06.1 มาได้เกือบหมด คุณ *ไม่ควร* ทำการคัดลอก `BigInt` ใดๆ!

```rust
impl Minimum for BigInt {
    fn min<'a>(&'a self, other: &'a Self) -> &'a Self {
        unimplemented!()
    }
}
```

## การโอเวอร์โหลดตัวดำเนินการ (Operator Overloading)

เราจะรู้ได้อย่างไรว่าฟังก์ชัน `min` ของเราทำในสิ่งที่เราต้องการให้มันทำจริงๆ? ความเป็นไปได้อย่างหนึ่งคือการทำการทดสอบ Rust มาพร้อมกับการสนับสนุนในตัวที่ดีเยี่ยมสำหรับทั้งการทดสอบหน่วย (unit tests) และการทดสอบบูรณาการ (integration tests) อย่างไรก็ตาม ก่อนที่เราจะไปถึงตรงนั้น เราจำเป็นต้องมีวิธีเช็คว่าผลลัพธ์ของการเรียกฟังก์ชันนั้นถูกต้อง กล่าวอีกนัยหนึ่ง เราต้องนิยามวิธีการทดสอบความเท่ากันของ `BigInt` การที่สามารถทดสอบความเท่ากันได้เป็นคุณสมบัติของชนิดข้อมูล ซึ่ง - คุณเดาถูกแล้ว - Rust แสดงออกในรูปแบบของเทรต: `PartialEq`

การทำสิ่งนี้สำหรับ `BigInt` ค่อนข้างง่าย ขอบคุณข้อกำหนดของเราที่ว่าต้องไม่มีศูนย์ต่อท้าย เราเพียงแค่นำการทดสอบความเท่ากันบนเวกเตอร์มาใช้ซ้ำ ซึ่งจะเปรียบสมาชิกทุกตัวทีละตัว Attribute `inline` บอก Rust ว่าโดยทั่วไปเราต้องการให้ฟังก์ชันนี้ถูก inline (แทรกโค้ดเข้าไปในจุดที่เรียกใช้)

```rust
impl PartialEq for BigInt {
    #[inline]
    fn eq(&self, other: &BigInt) -> bool {
        debug_assert!(self.test_invariant() && other.test_invariant());
        self.data == other.data
    }
}
```

เนื่องจากการอิมพลีเมนท์ `PartialEq` เป็นเรื่องของกลไกค่อนข้างมาก คุณสามารถปล่อยให้ Rust จัดการสิ่งนี้อัตโนมัติได้โดยการเติม attribute `derive(PartialEq)` ไปที่การนิยามชนิดข้อมูล ในกรณีที่คุณสงสัยเกี่ยวกับ "partial" (บางส่วน) ผมแนะนำให้คุณเช็คเอกสารของ `PartialEq` และ `Eq` ดู `Eq` สามารถถูกสร้างอัตโนมัติ (derived) ได้เช่นกัน

ตอนนี้เราสามารถเปรียบเทียบ `BigInt` ได้แล้ว Rust ปฏิบัติกับ `PartialEq` เป็นพิเศษโดยที่มันถูกเชื่อมโยงเข้ากับตัวดำเนินการ `==`: ตัวดำเนินการนั้นสามารถถูกใช้บนตัวเลขของเราได้แล้ว! พูดในภาษา C++ ก็คือ เราเพิ่งจะทำการโอเวอร์โหลด (overloaded) ตัวดำเนินการ `==` สำหรับ `BigInt` Rust ไม่มีการโอเวอร์โหลดฟังก์ชัน (นั่นคือ มันจะไม่ dispatch ไปยังฟังก์ชันที่แตกต่างกันขึ้นอยู่กับชนิดข้อมูลของอาร์กิวเมนต์) แต่โดยปกติแล้วเราจะหา (หรือนิยาม) เทรตที่จับเอาลักษณะสำคัญที่เหมือนกันของการโอเวอร์โหลดทั้งหมด และเขียนฟังก์ชันเดียวที่เป็นเจเนอริกในเทรตนั้น ตัวอย่างเช่น แทนที่จะโอเวอร์โหลดฟังก์ชันสำหรับทุกวิธีการที่สตริงสามารถถูกแทนค่าได้ เราเขียนฟังก์ชันเจเนอริกบน `ToString` โดยปกติ จะมีเทรตแบบนี้ที่ตรงกับวัตถุประสงค์ - และถ้ามี สิ่งนี้มีข้อดีมหาศาลตรงที่ชนิดข้อมูลใดๆ ที่คุณเขียนที่สามารถแปลงเป็นสตริงได้ เพียงแค่ต้องอิมพลีเมนท์เทรตนั้นเพื่อที่จะใช้งานได้ทันทีกับฟังก์ชันทั้งหมดที่มีอยู่แล้ว ที่เขียนไว้แบบทั่วไปบน `ToString` เปรียบเทียบสิ่งนี้กับ C++ หรือ Java ที่ซึ่งโอกาสเดียวที่จะเพิ่ม variant การโอเวอร์โหลดใหม่คือการแก้ไขคลาสของผู้รับ (receiver)

ทำไมเราถึงใช้ `!=` ได้ด้วย ถึงแม้ว่าเราเพิ่งจะโอเวอร์โหลดแค่ `==`? คำตอบอยู่ที่สิ่งที่เรียกว่า **default implementation (การอิมพลีเมนท์ตั้งต้น)** ถ้าคุณเช็คเอกสารของ `PartialEq` ที่ผมลิงก์ไว้ด้านบน คุณจะเห็นว่าเทรตนี้จริงๆ แล้วจัดเตรียมเมธอดไว้สองตัว: `eq` เพื่อทดสอบความเท่ากัน และ `ne` เพื่อทดสอบความไม่เท่ากัน อย่างที่คุณน่าจะเดาได้ `!=` ถูกเชื่อมโยงกับ `ne` การนิยามเทรตยังจัดเตรียม default implementation ของ `ne` ไว้ให้เป็นนิเสธของ `eq` ดังนั้นคุณสามารถแค่จัดเตรียม `eq` และ `!=` ก็จะทำงานได้ดี หรือถ้าคุณมีวิธีที่มีประสิทธิภาพกว่าในการตัดสินความไม่เท่ากัน คุณก็สามารถจัดเตรียม `ne` สำหรับชนิดข้อมูลของคุณเองได้

```rust
fn compare_big_ints() {
    let b1 = BigInt::new(13);
    let b2 = BigInt::new(37);
    println!("b1 == b1: {} ; b1 == b2: {}; b1 != b2: {}", b1 == b1, b1 == b2, b1 != b2);
}
```

## การทดสอบ (Testing)

เมื่อเขียนการทดสอบความเท่ากันเสร็จแล้ว เราก็พร้อมที่จะเขียนกรณีทดสอบ (testcase) แรกของเรา มันง่ายมาก: คุณแค่เขียนฟังก์ชัน (ที่ไม่มีอาร์กิวเมนต์หรือค่าส่งกลับ) และให้ attribute `test` กับมัน `assert!` เหมือนกับ `debug_assert!` แต่จะไม่ถูกลบออกไปใน release build

```rust
#[test]
fn test_min() {
    let b1 = BigInt::new(1);
    let b2 = BigInt::new(42);
    let b3 = BigInt::from_vec(vec![0, 1]);

    assert!(*b1.min(&b2) == b1);
    assert!(*b3.min(&b2) == b2);
}
```

ตอนนี้รัน `cargo test` เพื่อประมวลผลการทดสอบ ถ้าคุณอิมพลีเมนท์ `min` ถูกต้อง มันควรจะทำงานได้ทั้งหมด!

## การจัดรูปแบบ (Formatting)

ยังมีแมโคร `assert_eq!` ที่ถูกปรับแต่งมาเพื่อทดสอบความเท่ากัน และจะพิมพ์ค่าสองค่า (ซ้ายและขวา) ออกมาถ้าพวกมันแตกต่างกัน เพื่อที่จะสามารถทำสิ่งนั้นได้ แมโครจำเป็นต้องรู้วิธีจัดรูปแบบค่าเพื่อการพิมพ์ นี่หมายความว่าเรา - เดาสิ? - ต้องอิมพลีเมนท์เทรตที่เหมาะสม Rust รู้จักวิธีการจัดรูปแบบค่าสองวิธี: `Display` สำหรับการพิมพ์สวยงามในแบบที่ผู้ใช้เข้าใจได้ ในขณะที่ `Debug` มีไว้เพื่อแสดงสถานะภายในของข้อมูลและพุ่งเป้าไปที่โปรแกรมเมอร์ อย่างหลังนี้คือสิ่งที่เราต้องการสำหรับ `assert_eq!` ดังนั้นมาเริ่มกันเลย

การจัดรูปแบบทั้งหมดถูกจัดการโดย [`std::fmt`](https://doc.rust-lang.org/std/fmt/index.html) ผมจะไม่อธิบายรายละเอียดทั้งหมด และอ้างอิงคุณไปที่เอกสารแทน

```rust
use std::fmt;
```

ในกรณีของ `BigInt` เราอยากจะแค่แสดงผลอาร์เรย์ข้อมูลภายในของเรา ดังนั้นเราเพียงแค่เรียกฟังก์ชันจัดรูปแบบของ `Vec<u64>`:

```rust
impl fmt::Debug for BigInt {
    fn fmt(&self, f: &mut fmt::Formatter) -> fmt::Result {
        self.data.fmt(f)
    }
}
```

การอิมพลีเมนท์ `Debug` สามารถถูกสร้างขึ้นอัตโนมัติโดยใช้ attribute `derive(Debug)`

ตอนนี้เราพร้อมที่จะใช้ `assert_eq!` เพื่อทดสอบ `vec_min` แล้ว

```rust
/*#[test]*/
fn test_vec_min() {
    let b1 = BigInt::new(1);
    let b2 = BigInt::new(42);
    let b3 = BigInt::from_vec(vec![0, 1]);

    let v1 = vec![b2.clone(), b1.clone(), b3.clone()];
    let v2 = vec![b2.clone(), b3.clone()];
    
    assert_eq!(vec_min(&v1), Some(&b1));
    assert_eq!(vec_min(&v2), Some(&b2));
}
```
