# Rust-101, ส่วนที่ 12: Rc, ความสามารถในการแก้ไขภายใน (Interior Mutability), Cell, RefCell

```rust
use std::rc::Rc;
use std::cell::{Cell, RefCell};
```

กลไก callback แบบเจเนอริกของเราทำงานได้ค่อนข้างดีทีเดียว อย่างไรก็ตาม มีจุดหนึ่งที่เราอาจจะอยากแก้ไข: `Callbacks` ไม่ได้อิมพลีเมนท์ `Clone` ปัญหาคือ closures (หรือพูดให้ถูกคือ สภาพแวดล้อมของพวกมัน) ไม่สามารถถูกโคลนได้เลย (ไม่มีแม้กระทั่งการสร้างอัตโนมัติ (automatic derivation) สำหรับกรณีที่มันน่าจะเป็นไปได้) ข้อจำกัดนี้แพร่กระจายขึ้นไปถึงตัว `Callbacks` เอง เราจะทำอะไรกับเรื่องนี้ได้บ้าง?

## Rc

ทางแก้คือการหาวิธีในการโคลน `Callbacks` โดยไม่ต้องโคลนสภาพแวดล้อม สิ่งนี้สามารถบรรลุผลได้ด้วย `Rc<T>` ซึ่งเป็น pointer ที่มีการนับ reference (reference-counted pointer) นี่เป็นอีกตัวอย่างหนึ่งของ smart pointer คุณสามารถโคลน `Rc` บ่อยแค่ไหนก็ได้ตามที่คุณต้องการ นั่นไม่ส่งผลกระทบต่อข้อมูลที่มันบรรจุอยู่ มันเพียงแค่สร้าง references เพิ่มเติมไปยังข้อมูลเดียวกัน เมื่อ references ทั้งหมดหายไป ข้อมูลจะถูกคืนหน่วยความจำ

เดี๋ยวก่อน คุณอาจจะพูดขึ้นมา References หลายตัวไปยังข้อมูลเดียวกัน? นั่นมัน Aliasing! จริงด้วย: เมื่อข้อมูลถูกเก็บใน `Rc` มันจะเป็นแบบอ่านได้อย่างเดียว (read-only) และคุณสามารถทำได้เพียงแค่รับ shared reference ไปยังข้อมูลเท่านั้น

เนื่องจากข้อจำกัดแบบอ่านได้อย่างเดียวนี้ เราไม่สามารถใช้ `FnMut` ที่นี่ได้: เราจะไม่สามารถเรียกฟังก์ชันด้วย mutable reference ไปยังสภาพแวดล้อมของมันได้! ดังนั้นเราต้องไปกับ `Fn` เราห่อมันใน `Rc` และจากนั้น Rust ก็สร้าง `Clone` ให้เราอย่างมีความสุข

```rust
#[derive(Clone)]
struct Callbacks {
    callbacks: Vec<Rc<Fn(i32)>>,
}

impl Callbacks {
    pub fn new() -> Self {
        Callbacks { callbacks: Vec::new() }
    }

    // การลงทะเบียนทำงานเหมือนกับครั้งที่แล้ว ยกเว้นว่าตอนนี้เรากำลังสร้าง `Rc`
    pub fn register<F: Fn(i32) + 'static>(&mut self, callback: F) {
        self.callbacks.push(Rc::new(callback));
    }

    pub fn call(&self, val: i32) {
        // เราต้องการแค่ตัววนซ้ำแบบแบ่งปัน (shared iterator) ที่นี่ เนื่องจาก `Rc` เป็น smart pointer เราจึงสามารถเรียก callback ได้โดยตรง
        for callback in self.callbacks.iter() {
            callback(val);
        }
    }
}
```

ได้เวลาสาธิต!

```rust
fn demo(c: &mut Callbacks) {
    c.register(|val| println!("Callback 1: {}", val));
    c.call(0);
    c.clone().call(1);
}

pub fn main() {
    let mut c = Callbacks::new();
    demo(&mut c);
}
```

## ความสามารถในการแก้ไขภายใน (Interior Mutability)

แน่นอนว่า ตัวอย่างการนับเลขจากครั้งที่แล้วใช้ไม่ได้อีกต่อไป: มันจำเป็นต้องแก้ไขสภาพแวดล้อม ซึ่ง `Fn` ทำไม่ได้ กฎการยืมที่เข้มงวดของ Rust กำลังเข้ามาขวางทางเรา

อย่างไรก็ตาม เมื่อพูดถึงการแก้ไขตัวเลขธรรมดาๆ (`usize`) มันแทบไม่มีโอกาสที่จะเกิดปัญหาอะไรขึ้น ทุกคนสามารถอ่านและเขียนตัวแปรนั้นได้ตามที่พวกเขาต้องการ ดังนั้นมันคงน่าเศร้ามากถ้าเราไม่สามารถเขียนโปรแกรมนี้ได้ โชคดีพอ ไลบรารีมาตรฐานของ Rust จัดเตรียมทางแก้ไว้ในรูปแบบของ `Cell<T>` ชนิดข้อมูลนี้เป็นตัวแทนของเซลล์หน่วยความจำของชนิดข้อมูล `T` บางอย่าง โดยจัดเตรียมการกระทำพื้นฐานสองอย่างคือ `get` และ `set` `get` คืนค่าสำเนาของเนื้อหาของเซลล์ ดังนั้นทั้งหมดนี้ทำงานได้เฉพาะถ้า `T` เป็น `Copy` ส่วน `set` ซึ่งเขียนทับเนื้อหา ต้องการเพียงแค่ shared reference ไปยังเซลล์

ปรากฏการณ์ของชนิดข้อมูลที่อนุญาตให้เกิดการแก้ไขผ่าน shared references (นั่นคือ การแก้ไขทั้งๆ ที่มีความเป็นไปได้ที่จะเกิด aliasing) เรียกว่า **ความสามารถในการแก้ไขภายใน (interior mutability)** คุณสามารถคิดว่า `set` เปลี่ยนแปลงแค่เนื้อหาของเซลล์ ไม่ใช่ตัวตน (identity) ของมัน ในทางตรงกันข้าม ชนิดของการแก้ไขที่เราเห็นมาตลอดคือการแทนที่ข้อมูลชิ้นหนึ่งด้วยอย่างอื่นที่มีชนิดข้อมูลเดียวกัน สิ่งนี้เรียกว่า **ความสามารถในการแก้ไขที่สืบทอดมา (inherited mutability)**

สังเกตว่าเป็นไปไม่ได้ที่จะยืมเนื้อหาของเซลล์ และนั่นคือกุญแจสำคัญจริงๆ ว่าทำไมสิ่งนี้ถึงปลอดภัย

ดังนั้น ลองเอาตัวนับของเราไปใส่ใน `Cell` และจำลองตัวอย่างจากส่วนที่แล้วกัน

```rust
fn demo_cell(c: &mut Callbacks) {
    {
        let count = Cell::new(0);
        // อีกครั้ง เราต้องย้ายความเจ้าของของ `count` เข้าไปใน closure สภาพแวดล้อม
        c.register(move |val| {
            // ในนี้ ทั้งหมดที่เรามีคือ shared reference ของสภาพแวดล้อมของเรา แต่นั่นก็ดีพอแล้ว
            // สำหรับ `get` และ `set` ของเซลล์! ณ รันไทม์ `Cell` จะเกือบถูก
            // คอมไพล์ทิ้งไปทั้งหมด (compiled away) ดังนั้นสิ่งนี้จึงกลายเป็นสิ่งที่เทียบเท่ากับเวอร์ชันที่เราเขียนใน
            // ส่วนที่แล้ว
            let new_count = count.get() + 1;
            count.set(new_count);
            println!("Callback 2: {} ({}. time)", val, new_count);
        });
    }

    c.call(2);
    c.clone().call(3);
}
```

มันคุ้มค่าที่จะกล่าวถึงว่า `Rc` เองก็ต้องใช้ประโยชน์จาก interior mutability เช่นกัน: เมื่อคุณโคลน `Rc` ทั้งหมดที่มันมีให้ใช้คือ shared reference อย่างไรก็ตาม มันต้องเพิ่มตัวนับ reference! ภายใน `Rc` ใช้ `Cell` สำหรับตัวนับ เพื่อที่มันจะสามารถถูกอัปเดตได้ในระหว่าง `clone`

เมื่อเอาทั้งหมดมารวมกัน เรื่องราวเกี่ยวกับการแก้ไขและความเป็นเจ้าของผ่าน references ดูจะเป็นดังนี้: มี unique references ซึ่ง - เนื่องจากความพิเศษเฉพาะตัวของพวกมัน - ปลอดภัยเสมอที่จะแก้ไขค่าผ่านมัน และมี shared references ซึ่งคอมไพเลอร์ไม่สามารถสัญญาได้โดยทั่วไปว่าการแก้ไขนั้นปลอดภัย อย่างไรก็ตาม ถ้ามีสถานการณ์พิเศษที่รับประกันว่าการแก้ไขนั้นปลอดภัย มันก็สามารถเกิดขึ้นได้แม้กระทั่งผ่าน shared reference - อย่างที่เราเห็นกับ `Cell`

## RefCell

เป็นขั้นตอนถัดไปในวิวัฒนาการของ `Callbacks` เราอาจจะลองแก้ปัญหาเรื่องความสามารถในการแก้ไขนี้แบบครั้งเดียวจบ โดยการเติม `Cell` เข้าไปใน `Callbacks` เพื่อที่ลูกค้า (clients) จะได้ไม่ต้องกังวลเกี่ยวกับเรื่องนี้ อย่างไรก็ตาม นั่นจะจบลงด้วยการทำงานไม่ได้: จำไว้ว่า `Cell` ทำงานได้เฉพาะกับชนิดข้อมูลที่เป็น `Copy` ซึ่งสภาพแวดล้อมของ closure ไม่มีทางเป็นอย่างนั้น เราต้องการ `Cell` อีกแบบที่อนุญาตให้ยืมเนื้อหาของมันได้ เพื่อที่เราจะสามารถมอบสภาพแวดล้อมให้กับ `FnMut` ได้ แต่ถ้า `Cell` อนุญาตสิ่งนั้น เราก็จะสามารถเขียนโปรแกรม C++ พังๆ เหล่านั้นที่เราอยากจะกำจัดทิ้งได้

นี่คือจุดที่โปรแกรมของเราซับซ้อนเกินกว่าที่ Rust จะรับประกัน ณ เวลาคอมไพล์ (compile-time) ว่าจะไม่มีเรื่องร้ายแรงเกิดขึ้น เนื่องจากเราไม่อยากทิ้งการรับประกันความปลอดภัย เราจึงจะต้องมีโค้ดบางส่วนที่คอยตรวจสอบจริงๆ ณ รันไทม์ (run-time) ว่ากฎการยืมไม่ถูกละเมิด การตรวจสอบดังกล่าวถูกจัดเตรียมให้โดย `RefCell<T>`: แตกต่างจาก `Cell<T>` สิ่งนี้ยอมให้เรายืมเนื้อหา และมันทำงานได้กับ `T` ที่ไม่ใช่ `Copy` แต่ อย่างที่เราจะเห็น มันมีค่าใช้จ่ายตอนรันไทม์เกิดขึ้น

เวอร์ชันสุดท้ายของ `Callbacks` ของเราเอาสภาพแวดล้อมของ closure ใส่เข้าไปใน `RefCell`

```rust
#[derive(Clone)]
struct CallbacksMut {
    callbacks: Vec<Rc<RefCell<FnMut(i32)>>>,
}

impl CallbacksMut {
    pub fn new() -> Self {
        CallbacksMut { callbacks: Vec::new() }
    }

    pub fn register<F: FnMut(i32) + 'static>(&mut self, callback: F) {
        let cell = Rc::new(RefCell::new(callback));
        self.callbacks.push(cell);
    }

    pub fn call(&mut self, val: i32) {
        for callback in self.callbacks.iter() {
            // เราต้องยืมเนื้อหาของ `RefCell` *อย่างชัดแจ้ง* โดยการเรียก `borrow` หรือ
            // `borrow_mut` ณ รันไทม์ เซลล์จะคอยติดตามจำนวนของ
            // shared และ mutable references ที่ค้างอยู่ และ panic ถ้ากฎถูกละเมิด
            let mut closure = callback.borrow_mut();
            // เพื่อที่จะให้การตรวจสอบนี้ถูกดำเนินการ `closure` คือยาม (guard): แทนที่จะเป็น reference ปกติ
            // `borrow_mut` คืนค่า smart pointer (`RefMut` ในกรณีนี้) ที่รอจนกว่ามันจะ
            // หลุดขอบเขต (goes out of scope) และจากนั้นจึงอัปเดตจำนวนของ references ที่ active อยู่อย่างเหมาะสม
            // 
            // โชคร้ายที่ การ auto-dereference ของ pointers ใน Rust ไม่ฉลาดพอตรงนี้ เรา
            // จึงต้อง dereference ตัว smart pointer อย่างชัดแจ้ง และรับ mutable reference
            // ไปยังเนื้อหา
            (&mut *closure)(val);
        }
    }
}
```

เนื่องจาก `call` เป็นที่เดียวที่ยืมสภาพแวดล้อมของ closures เราควรคาดหวังว่าการตรวจสอบจะผ่านเสมอ และจริงๆ แล้วไร้ประโยชน์โดยสิ้นเชิง อย่างไรก็ตาม เรื่องนี้ไม่จริงซะทีเดียว `CallbacksMut` ที่แตกต่างกันหลายตัวอาจจะแบ่งปัน callback ตัวเดียวกัน (เพราะพวกมันถูกสร้างด้วย `clone`) และการเรียก callback ตัวหนึ่งที่นี่อาจจะกระตุ้นการเรียก callbacks ทั้งหมดของ `CallbacksMut` ตัวอื่น ซึ่งจะจบลงด้วยการเรียก callback เริ่มต้นซ้ำอีกครั้ง ปัญหานี้ของการที่ฟังก์ชันเผลอเรียกตัวเองซ้ำ (recursively calling themselves) เรียกว่า **Reentrancy (การเข้าทำงานซ้ำ)** และมันสามารถนำไปสู่บั๊กที่แนบเนียน ที่นี่ มันจะหมายความว่า closure รันสองครั้ง แต่ละครั้งคิดว่ามันมี unique, mutable reference ไปยังสภาพแวดล้อมของมัน - ดังนั้นมันอาจจะจบลงด้วยการ dereference pointer ที่ห้อยต่องแต่ง (dangling pointer) โอ๊ย! โชคดีพอ Rust ตรวจจับสิ่งนี้ได้ ณ รันไทม์และ panic ทันทีที่เราพยายามจะยืมสภาพแวดล้อมเดิมอีกครั้ง ผมหวังว่าสิ่งนี้จะช่วยทำให้ชัดเจนด้วยว่าไม่มีความหวังเลยที่ Rust จะทำการตรวจสอบเหล่านี้แบบ statically ณ compile-time: มันจะต้องตรวจจับ reentrancy ให้ได้!

ตอนนี้เราสามารถทำซ้ำการสาธิตจากส่วนที่แล้ว - แต่ครั้งนี้ ชนิดข้อมูล `CallbacksMut` ของเราสามารถโคลนได้

```rust
fn demo_mut(c: &mut CallbacksMut) {
    c.register(|val| println!("Callback 1: {}", val));
    c.call(0);

    {
        let mut count: usize = 0;
        c.register(move |val| {
            count = count + 1;
            println!("Callback 2: {} ({}. time)", val, count);
        });
    }
    c.call(1);
    c.clone().call(2);
}
```
