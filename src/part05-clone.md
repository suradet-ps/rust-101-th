## Part 05: Clone

## จำนวนขนาดใหญ่

ในช่วงหลายตอนถัดไป เราจะสร้างโครงสร้างข้อมูลสำหรับการคำนวณกับจำนวนขนาดใหญ่ เราต้องการให้ตัวเลขเหล่านี้ใหญ่ได้ไม่จำกัด โดยมีเพียงหน่วยความจำของเครื่องเท่านั้นที่เป็นข้อจำกัด

เราเริ่มจากการตัดสินใจว่าจะแทนจำนวนขนาดใหญ่เหล่านี้อย่างไร วิธีหนึ่งคือใช้เวกเตอร์เก็บ "หลัก" ของตัวเลข ก็เหมือนกับที่ "1337" เป็นเวกเตอร์ของสี่หลัก (1, 3, 3, 7) แต่เราจะใช้ `u64` เป็นชนิดข้อมูลของแต่ละหลัก หมายความว่าเรามีหลักตัวเลขได้ถึง 2^64 หลัก ตอนนี้เราแค่ต้องตัดสินใจว่าจะเรียงลำดับตัวเลขอย่างไร ผมเลือกที่จะเก็บหลักที่มีนัยสำคัญน้อยที่สุดไว้ก่อน ซึ่งหมายความว่า "1337" จะกลายเป็น (7, 3, 3, 1)

นอกจากนี้ เรากำหนดว่าห้ามมีเลขศูนย์ต่อท้าย (ซึ่งก็เหมือนกับเลขศูนย์นำหน้าที่ไม่มีประโยชน์ในการเขียนตัวเลขปกติ) เงื่อนไขนี้จะช่วยให้แน่ใจว่าตัวเลขเดียวกันจะถูกเก็บได้เพียงแบบเดียวเท่านั้น

ในการเขียนสิ่งนี้ลงใน Rust เราใช้ `struct` (โครงสร้างข้อมูล) ซึ่งคล้ายกับ `struct` ใน C มาก คือชุดของฟิลด์ที่มีชื่อ แต่ละฟิลด์สามารถเป็น private ของโมดูลปัจจุบัน (ค่าเริ่มต้น) หรือเป็น public (โดยใส่ `pub` ไว้หน้าชื่อ) เพื่อประโยชน์ของบทเรียนนี้ เราทำให้ `data` เป็น public เพราะไม่เช่นนั้นตอนถัดไปจะไม่สามารถทำงานกับ `BigInt` ได้ แต่ในโปรแกรมจริง ควรทำให้ฟิลด์เป็น private เพื่อรักษาเงื่อนไข (ห้ามมีเลขศูนย์ต่อท้าย) ไว้

```rust
pub struct BigInt {
    pub data: Vec<u64>, // least significant digit first, no trailing zeros
}
```

เมื่อกำหนดการแทนข้อมูลเสร็จแล้ว เราก็สามารถเริ่มสร้างเมธอดต่างๆ ได้

```rust
impl BigInt {
```

มาเริ่มด้วย constructor ที่สร้าง `BigInt` จากจำนวนเต็มธรรมดา การสร้างอินสแตนซ์ของ `struct` ทำได้โดยเขียนชื่อของมันตามด้วยรายการฟิลด์และค่าเริ่มต้น

```rust
    pub fn new(x: u64) -> Self {
        if x == 0 {
            BigInt { data: vec![] }
        } else {
            BigInt { data: vec![x] }
        }
    }
```

มักจะเป็นประโยชน์ที่จะเข้ารหัสเงื่อนไขของโครงสร้างข้อมูลลงในโค้ด ดังนั้นนี่คือฟังก์ชันตรวจสอบที่จะจับเลขศูนย์ต่อท้ายที่ไม่จำเป็น

```rust
    pub fn test_invariant(&self) -> bool {
        if self.data.len() == 0 {
            true
        } else {
            self.data[self.data.len() - 1] != 0
        }
    }
```

เราสามารถแปลงเวกเตอร์ little-endian ของหลักใดๆ (คือเรียงหลักที่มีนัยสำคัญน้อยที่สุดไว้ก่อน) เป็นตัวเลข โดยการตัดเลขศูนย์ต่อท้ายออก การประกาศ `mut` สำหรับ `v` ที่นี่ก็เหมือนกับใน `let mut ...`: เราเป็นเจ้าของ `v` อย่างสมบูรณ์ แต่ Rust ยังคงขอให้เราระบุเจตนาที่จะแก้ไขมันให้ชัดเจน อย่างไรก็ตาม `mut` นี้ไม่ใช่ส่วนหนึ่งของไทป์ของ `from_vec` เพราะผู้เรียกใช้ต้องยกเลิก ownership ของ `v` อยู่แล้ว จึงไม่สนใจว่าคุณจะทำอะไรกับมันต่อ

**คำใบ้:** คุณสามารถใช้ `pop` เพื่อลบสมาชิกตัวสุดท้ายของเวกเตอร์ได้

```rust
    pub fn from_vec(mut v: Vec<u64>) -> Self {
        unimplemented!()
    }
}
```

## การ Clone

ถ้าคุณสังเกตไทป์ของ `BigInt::from_vec` ให้ดี จะเห็นว่ามันจะ consume เวกเตอร์ `v` ดังนั้นผู้เรียกใช้จะเข้าถึงเวกเตอร์ของตนไม่ได้อีก แต่เรามีทางแก้ถ้าไม่ต้องการให้เป็นเช่นนั้น: เราสามารถ clone เวกเตอร์อย่างชัดเจน ซึ่งหมายความว่าจะมีการคัดลอกแบบเต็มรูปแบบ (หรือแบบลึก) เกิดขึ้น ในทางเทคนิค `clone` จะรับเวกเตอร์ที่ถูก borrow ในรูปแบบของ shared reference และคืนค่าที่เป็น owned อย่างสมบูรณ์

```rust
fn clone_demo() {
    let v = vec![0, 1 << 16];
    let b1 = BigInt::from_vec((&v).clone());
    let b2 = BigInt::from_vec(v);
}
```

Rust มีการจัดการพิเศษสำหรับเมธอดที่ borrow อาร์กิวเมนต์ `self` (เช่น `clone` หรือ `test_invariant` ข้างต้น): ไม่จำเป็นต้อง borrow ตัวรับของเมธอดอย่างชัดเจน ดังนั้นคุณสามารถเขียน `v.clone()` แทน `(&v).clone()` ข้างต้นได้ ลองดูสิ!

การเป็น clonable เป็นคุณสมบัติของชนิดข้อมูล และจึงแสดงออกผ่าน trait อย่างเป็นธรรมชาติ ที่จริงแล้ว Rust มี trait `Clone` สำหรับจุดประสงค์นี้อยู่แล้ว เราจึงสามารถทำให้ `BigInt` ของเราเป็น clonable ได้ด้วย

```rust
impl Clone for BigInt {
    fn clone(&self) -> Self {
        BigInt { data: self.data.clone() }
    }
}
```

การทำให้ชนิดข้อมูลเป็น clonable เป็นเรื่องที่ทำกันบ่อยจนกระทั่ง Rust สามารถช่วยคุณทำมันได้: ถ้าคุณเพิ่ม `#[derive(Clone)]` ไว้หน้าคำจำกัดความของ `BigInt` Rust จะสร้าง implementation ของ `Clone` ที่ clone ทุกฟิลด์ให้โดยอัตโนมัติ ลองดูสิ! คำอธิบายประกอบ `#[...]` เหล่านี้ที่ใช้กับชนิดข้อมูล (และฟังก์ชัน โมดูล crates) เรียกว่า **attributes** เราจะเห็นตัวอย่าง attributes เพิ่มเติมในภายหลัง

เรายังสามารถทำให้ชนิดข้อมูล `SomethingOrNothing<T>` มี `Clone` ได้ด้วย แต่จะทำได้ก็ต่อเมื่อ `T` เป็น `Clone` เท่านั้น! ดังนั้นเราต้องเพิ่มข้อจำกัดนี้ให้กับ `T` ตอนที่ประกาศตัวแปรไทป์

```rust
use part02::{SomethingOrNothing, Something, Nothing};

impl<T: Clone> Clone for SomethingOrNothing<T> {
    fn clone(&self) -> Self {
        match *self {
            Nothing => Nothing,
```

ในแขนที่สองของ `match` เราต้องอ้างถึงค่า `v` ที่เก็บอยู่ใน `self` แต่ถ้าเราเขียน pattern เป็น `Something(v)` นั่นจะหมายความว่าเราเป็นเจ้าของ `v` ในโค้ดหลังลูกศร ซึ่งทำไม่ได้ เพราะเราต้องปล่อยให้ `v` เป็นของผู้ที่เรียกเราอยู่ นอกจากนี้เราไม่ได้เป็นเจ้าของ `self` ด้วยซ้ำ แค่ borrow มา ด้วยการเขียน `Something(ref v)` เราจะ borrow `v` เฉพาะภายในแขน match นั้น ซึ่งก็เพียงพอสำหรับการ clone มัน

```rust
            Something(ref v) => Something(v.clone()),
        }
    }
}
```

อีกครั้ง Rust จะสร้าง implementation นี้ให้โดยอัตโนมัติ ถ้าคุณเพิ่ม `#[derive(Clone)]` ไว้หน้าคำจำกัดความของ `SomethingOrNothing`

**แบบฝึกหัด 05.2:** เขียนฟังก์ชันเพิ่มเติมสำหรับ `BigInt` ลองทำฟังก์ชันที่คืนจำนวนหลักดูสิ? หรือจำนวนหลักที่ไม่ใช่ศูนย์? หลักที่เล็กที่สุด/ใหญ่ที่สุด? แน่นอนว่าฟังก์ชันเหล่านี้ควรรับ `self` เป็น shared reference ทั้งหมด (คือในรูปแบบที่ borrow)

## Mutation + aliasing considered harmful (ตอนที่ 2)

> **⚠️ ตัวอย่างนี้ซับซ้อนและอันตราย** แสดงให้เห็นว่าทำไม Rust ต้องห้าม mutation เมื่อมี aliasing อย่างเข้มงวด แม้จะดูยาก แต่สำคัญในการเข้าใจระบบความปลอดภัยของ Rust ถ้าอ่านไม่เข้าใจตอนแรก ไม่เป็นไร สามารถกลับมาอ่านภายหลังได้

เมื่อเรารู้วิธีสร้าง reference ไปยังเนื้อหาของ enum (เช่น `v` ข้างต้น) แล้ว เรามาดูอีกตัวอย่างหนึ่งว่าทำไมเราต้องห้าม mutation เมื่อมี aliasing เกิดขึ้น ก่อนอื่นเรากำหนด enum ที่สามารถเก็บได้ทั้งตัวเลขหรือสตริง

```rust
enum Variant {
    Number(i32),
    Text(String),
}
```

ตอนนี้มาดูโค้ดส่วนนี้ เช่นเดียวกับข้างต้น `n` จะเป็น reference ไปยังส่วนหนึ่งของ `var` และเนื่องจากเราเขียน `ref mut` reference นี้จะเป็นแบบ unique และ mutable กล่าวอีกนัยหนึ่ง หลังจาก match แล้ว `ptr` จะชี้ไปที่ตัวเลขที่เก็บอยู่ใน `var` ซึ่งในกรณีนี้ `var` เป็น `Number` จำไว้ว่า `_` หมายความว่า "เราไม่สนใจ"

```rust
fn work_on_variant(mut var: Variant, text: String) {
    let mut ptr: &mut i32;
    match var {
        Variant::Number(ref mut n) => ptr = n,
        Variant::Text(_) => return,
    }
    /* var = Variant::Text(text); */ /* BAD! */
    *ptr = 1337;
}
```

ตอนนี้ลองจินตนาการว่าจะเกิดอะไรขึ้นถ้าเราได้รับอนุญาตให้แก้ไข `var` ด้วย เราอาจทำให้มันกลายเป็น `Text` ได้ แต่ `ptr` ยังคงชี้ไปที่ตำแหน่งเดิม! ดังนั้น `ptr` ตอนนี้จะชี้ไปที่ไหนสักแห่งภายในส่วนข้อมูลของ `String` การเปลี่ยน `ptr` จะจัดการกับสตริงในวิธีที่คาดเดาไม่ได้อย่างสมบูรณ์ และอะไรก็สามารถเกิดขึ้นได้ถ้าเราใช้มันอีกครั้ง! (เจาะลึกหน่อย ฟิลด์แรกของ `String` เป็น pointer ไปยังข้อมูลตัวอักษร ดังนั้นเมื่อเขียนทับ pointer นั้นด้วยจำนวนเต็ม เราจะทำให้มันกลายเป็นแอดเดรสที่ใช้ไม่ได้โดยสิ้นเชิง เมื่อ destructor ของ `var` ทำงาน มันจะพยายามคืนหน่วยความจำที่แอดเดรสนั้น และ Rust ก็จะกินผ้าซักรีดของคุณ - หรืออะไรก็ตาม)

ผมหวังว่าตัวอย่างนี้จะทำให้เห็นชัดเจนว่าทำไม Rust ต้องห้าม mutation เมื่อมี aliasing เกิดขึ้นโดยทั่วไป ไม่ใช่แค่กรณีเฉพาะของบัฟเฟอร์ที่ถูกจัดสรรใหม่และ pointers เก่าที่กลายเป็นใช้ไม่ได้เท่านั้น
