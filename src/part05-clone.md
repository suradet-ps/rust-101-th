# Rust-101, ส่วนที่ 05: การโคลน (Clone)

## ตัวเลขขนาดใหญ่ (Big Numbers)

ในอีกไม่กี่ส่วนถัดไป เราจะสร้างโครงสร้างข้อมูลสำหรับการคำนวณกับตัวเลขขนาดใหญ่ เราไม่อยากจะให้มีขอบเขตบน (upper bound) ว่าตัวเลขเหล่านี้จะใหญ่ได้แค่ไหน โดยให้หน่วยความจำของเครื่องเป็นขีดจำกัดเพียงอย่างเดียว

เราเริ่มโดยการตัดสินใจว่าจะแทนค่าตัวเลขขนาดใหญ่เหล่านี้อย่างไรทางเลือกหนึ่งคือการใช้เวกเตอร์ของ "หลักเลข" ของตัวเลขนั้น นี่เหมือนกับว่า "1337" เป็นเวกเตอร์ของเลขโดดสี่ตัว (1, 3, 3, 7) ยกเว้นแต่ว่าเราจะใช้ `u64` เป็นชนิดข้อมูลของหลักเลขของเรา ซึ่งหมายความว่าเรามีเลขโดดที่แตกต่างกันได้ 2^64 ตัว ตอนนี้เราแค่ต้องตัดสินใจลำดับที่เราจะเก็บตัวเลข ผมตัดสินใจว่าเราจะเก็บหลักที่มีนัยสำคัญน้อยที่สุด (least significant digit) ก่อน ซึ่งหมายความว่า "1337" จะกลายเป็น (7, 3, 3, 1) จริงๆ

สุดท้าย เราประกาศว่าจะต้องไม่มีศูนย์ต่อท้าย (trailing zeros) ใดๆ (ซึ่งตรงกับศูนย์นำหน้าในวิธีการเขียนตัวเลขปกติของเราที่ไม่มีประโยชน์) นี่เพื่อให้มั่นใจว่าตัวเลขเดียวกันจะถูกเก็บได้ในรูปแบบเดียวเท่านั้น

ในการเขียนสิ่งนี้เป็น Rust เราใช้ `struct` ซึ่งเหมือนกับ struct ใน C มาก: ก็แค่กลุ่มของฟิลด์ที่มีชื่อ ทุกฟิลด์สามารถเป็นส่วนตัว (private) ต่อโมดูลปัจจุบัน (ซึ่งเป็นค่าตั้งต้น) หรือเป็นสาธารณะ (public) (ซึ่งระบุโดย `pub` ข้างหน้าชื่อ) เพื่อประโยชน์ของบทเรียนนี้ เราจะทำให้ `data` เป็นสาธารณะ - มิฉะนั้น ส่วนถัดๆ ไปของคอร์สนี้จะไม่สามารถทำงานกับ `BigInt` ได้ แน่นอนว่า ในโปรแกรมจริง เรามักจะทำให้ฟิลด์เป็นส่วนตัวเพื่อประกันว่า invariant (การไม่มีศูนย์ต่อท้าย) จะถูกรักษาไว้

```rust
pub struct BigInt {
    pub data: Vec<u64>, // หลักที่มีนัยสำคัญน้อยที่สุดขึ้นก่อน, ไม่มีศูนย์ต่อท้าย
}
```

ตอนนี้เรากำหนดรูปแบบข้อมูลได้แล้ว เราสามารถเริ่มอิมพลีเมนท์เมธอดให้กับมัน

```rust
impl BigInt {
    // มาเริ่มกันที่ตัวสร้าง (constructor) สร้าง BigInt จากจำนวนเต็มธรรมดา
    // ในการสร้างอินสแตนซ์ของ struct เราเขียนชื่อของมันตามด้วยรายการของฟิลด์
    // และค่าเริ่มต้นที่กำหนดให้พวกมัน
    pub fn new(x: u64) -> Self {
        if x == 0 {
            BigInt { data: vec![] }
        } else {
            BigInt { data: vec![x] }
        }
    }

    // บ่อยครั้งที่เป็นประโยชน์ที่จะเข้ารหัส invariant ของโครงสร้างข้อมูลไว้ในโค้ด
    // ดังนั้นนี่คือการตรวจสอบที่ตรวจหาศูนย์ต่อท้ายที่ไร้ประโยชน์
    pub fn test_invariant(&self) -> bool {
        if self.data.len() == 0 {
            true
        } else {
            self.data[self.data.len() - 1] != 0
        }
    }

    // เราสามารถแปลงเวกเตอร์ของหลักเลขแบบ little-endian (นั่นคือ หลักที่มีนัยสำคัญน้อยที่สุดขึ้นก่อน)
    // ใดๆ ให้เป็นตัวเลขได้ โดยการลบศูนย์ต่อท้ายออก การประกาศ `mut` สำหรับ `v` ที่นี่ก็เหมือนกับ
    // ใน `let mut ...`: เราเป็นเจ้าของ `v` อย่างสมบูรณ์ แต่ Rust ยังคงขอให้เราแสดงเจตนาที่จะ
    // แก้ไขมันอย่างชัดแจ้ง `mut` ตรงนี้ *ไม่ใช่* ส่วนหนึ่งของชนิดข้อมูลของ `from_vec` - ผู้เรียก
    // ต้องสละความเป็นเจ้าของ `v` ให้อยู่แล้ว ดังนั้นพวกเขาไม่สนใจอีกต่อไปว่าคุณจะทำอะไรกับมัน
    // 
    // **แบบฝึกหัด 05.1**: อิมพลีเมนท์ฟังก์ชันนี้
    // 
    // *คำใบ้*: คุณสามารถใช้ `pop` เพื่อลบสมาชิกตัวสุดท้ายของเวกเตอร์ได้
    pub fn from_vec(mut v: Vec<u64>) -> Self {
        unimplemented!()
    }
}
```

## การโคลน (Cloning)

ถ้าคุณพิจารณาชนิดข้อมูลของ `BigInt::from_vec` อย่างใกล้ชิด คุณจะสังเกตเห็นว่ามันบริโภค (consumes) เวกเตอร์ `v` ผู้เรียกจึงสูญเสียการเข้าถึงเวกเตอร์ของตน อย่างไรก็ตาม มีบางอย่างที่เราทำได้ถ้าเราไม่อยากให้เหตุการณ์นั้นเกิดขึ้น: เราสามารถสั่งโคลนเวกเตอร์นั้นอย่างชัดแจ้ง ซึ่งหมายความว่าจะมีการทำสำเนาข้อมูลทั้งหมด (full or deep copy) ในทางเทคนิคแล้ว `clone` รับเวกเตอร์ที่ถูกยืมในรูปแบบของ shared reference และส่งคืนสำเนาที่มีความเป็นเจ้าของอย่างสมบูรณ์

```rust
fn clone_demo() {
    let v = vec![0, 1 << 16];
    let b1 = BigInt::from_vec((&v).clone());
    let b2 = BigInt::from_vec(v);
}
```

Rust มีการปฏิบัติเป็นพิเศษสำหรับเมธอดที่ยืมอาร์กิวเมนต์ `self` ของมัน (เหมือน `clone` หรือเหมือน `test_invariant` ข้างบน): มันไม่จำเป็นต้องสั่งยืมผู้รับ (receiver) ของเมธอดอย่างชัดแจ้ง ดังนั้นคุณสามารถแทนที่ `(&v).clone()` ด้วย `v.clone()` ได้เลย ลองดูสิ!

การที่จะสามารถโคลนได้เป็นคุณสมบัติของชนิดข้อมูล และในฐานะที่เป็นเช่นนั้น จึงถูกแสดงออกมาด้วยเทรตโดยธรรมชาติ อันที่จริง Rust มาพร้อมกับเทรต `Clone` เพื่อวัตถุประสงค์นี้โดยเฉพาะ ดังนั้นเราสามารถทำให้ `BigInt` ของเราโคลนได้ดังนี้:

```rust
impl Clone for BigInt {
    fn clone(&self) -> Self {
        BigInt { data: self.data.clone() }
    }
}
```

การทำให้ชนิดข้อมูลโคลนได้เป็นแบบฝึกหัดที่พบบ่อยมากจน Rust สามารถช่วยคุณทำมันได้: ถ้าคุณเติม `#[derive(Clone)]` ไว้หน้าการนิยาม `BigInt` เลย Rust จะสร้างการอิมพลีเมนท์ของ `Clone` ที่เพียงแค่โคลนฟิลด์ทุกตัวให้ ลองดูสิ! การระบุ `#[...]` ที่ชนิดข้อมูล (และฟังก์ชัน, โมดูล, เครต) เหล่านี้เรียกว่า **attributes** เราจะเห็นตัวอย่างของ attributes เพิ่มเติมในภายหลัง

เรายังสามารถทำให้ชนิดข้อมูล `SomethingOrNothing<T>` อิมพลีเมนท์ `Clone` ได้ด้วย อย่างไรก็ตาม นั่นจะทำงานได้ก็ต่อเมื่อ `T` เป็น `Clone` เท่านั้น! ดังนั้นเราต้องเพิ่มขอบเขต (bound) นี้ให้กับ `T` เมื่อเราแนะนำตัวแปรชนิดข้อมูลนี้

```rust
use part02::{SomethingOrNothing, Something, Nothing};

impl<T: Clone> Clone for SomethingOrNothing<T> {
    fn clone(&self) -> Self {
        match *self {
            Nothing => Nothing,
            // ในแขนที่สองของ match เราจำเป็นต้องพูดถึงค่า v ที่ถูกเก็บอยู่ใน self
            // อย่างไรก็ตาม ถ้าเราเขียนรูปแบบเป็น Something(v) นั่นจะบ่งบอกว่าเรา
            // เป็นเจ้าของ v ในโค้ดหลังลูกศร นั่นไม่สามารถทำงานได้ เพราะเราต้องปล่อยให้ v ถูกเป็นเจ้าของโดย
            // ใครก็ตามที่เรียกเรา - ท้ายที่สุดแล้ว เราไม่ได้เป็นเจ้าของ self ด้วยซ้ำ เราแค่ยืมมันมา
            // โดยการเขียน Something(ref v) เราทำการยืม v สำหรับช่วงเวลาของ match arm นั้น
            // นั่นดีพอสำหรับการโคลนมัน
            Something(ref v) => Something(v.clone()),
        }
    }
}
```

อีกครั้ง Rust จะสร้างการอิมพลีเมนท์นี้ให้อัตโนมัติถ้าคุณเติม `#[derive(Clone)]` ไว้หน้าการนิยาม `SomethingOrNothing`

> **แบบฝึกหัด 05.2**: เขียนฟังก์ชันเพิ่มเติมให้กับ `BigInt` หน่อยสิ เอาเป็นฟังก์ชันที่คืนค่าจำนวนหลักเลข? จำนวนหลักที่ไม่ใช่ศูนย์? เลขโดดที่น้อยที่สุด/มากที่สุด? แน่นอนว่า ทั้งหมดนี้ควรรับ `self` เป็น shared reference (นั่นคือ ในรูปแบบที่ถูกยืม)

## การเปลี่ยนแปลงค่า + การมีชื่อเรียกซ้ำซ้อน ถือเป็นสิ่งอันตราย (Mutation + aliasing considered harmful) (ตอนที่ 2)

ตอนนี้ที่เรารู้วิธีสร้าง reference ไปยังเนื้อหาของ enum แล้ว (เหมือน `v` ข้างบน) มีอีกตัวอย่างหนึ่งที่เราสามารถดูได้ว่าทำไมเราต้องกำจัดการเปลี่ยนแปลงค่าเมื่อมีการใช้ชื่อเรียกซ้ำซ้อน ก่อนอื่น เรานิยาม enum ที่สามารถเก็บตัวเลข หรือไม่ก็สตริง

```rust
enum Variant {
    Number(i32),
    Text(String),
}
```

ตอนนี้พิจารณาโค้ดส่วนต่อไปนี้ เช่นเดียวกับข้างบน `n` จะเป็น reference ไปยังส่วนหนึ่งของ `var` และเนื่องจากเราเขียน `ref mut` reference นี้จะเป็นแบบแก้ไขได้และไม่ซ้ำใคร (unique and mutable) กล่าวอีกนัยหนึ่ง ทันทีหลังจาก match `ptr` ชี้ไปยังตัวเลขที่ถูกเก็บใน `var` โดยที่ `var` คือ `Variant::Number(_)` จำไว้ว่า `_` หมายถึง "เราไม่สนใจ"

```rust
fn work_on_variant(mut var: Variant, text: String) {
    let mut ptr: &mut i32;
    match var {
        Variant::Number(ref mut n) => ptr = n,
        Variant::Text(_) => return,
    }
    /* var = Variant::Text(text); */                                /* แย่มาก! */
    *ptr = 1337;
}
```

ทีนี้ จินตนาการว่าจะเกิดอะไรขึ้นถ้าเราได้รับอนุญาตให้แก้ไข `var` ได้ด้วย ตัวอย่างเช่น เราอาจทำให้มันกลายเป็น `Text` อย่างไรก็ตาม `ptr` ยังคงชี้ไปที่ตำแหน่งเดิม! ดังนั้น `ptr` ตอนนี้จึงชี้ไปตรงไหนสักแห่งในการแทนข้อมูลของ `String` โดยการเปลี่ยนค่า `ptr` เรากำลังจัดการกับสตริงในรูปแบบที่คาดเดาไม่ได้อย่างสิ้นเชิง และอะไรก็เกิดขึ้นได้ถ้าเรานำมันมาใช้อีก! (ในทางเทคนิค ฟิลด์แรกของ `String` คือ pointer ไปยังข้อมูลตัวอักษรของมัน ดังนั้นโดยการเขียนทับ pointer นั้นด้วยจำนวนเต็ม เรากำลังทำให้มันเป็นที่อยู่หน่วยความจำ (address) ที่ใช้การไม่ได้อย่างสิ้นเชิง เมื่อตัวทำลาย (destructor) ของ `var` ทำงาน มันจะพยายามคืนพื้นที่หน่วยความจำที่อยู่นั้น และ Rust ก็จะกินตับคุณ - หรือแย่กว่านั้น)

ผมหวังว่าตัวอย่างนี้จะช่วยให้กระจ่างว่าทำไม Rust ถึงต้องกำจัดการเปลี่ยนแปลงค่าเมื่อมีการใช้ชื่อเรียกซ้ำซ้อนโดยทั่วไป ไม่ใช่แค่กรณีเฉพาะของบัฟเฟอร์ที่ถูกจองพื้นที่ใหม่ และทำให้ pointer เก่าใช้การไม่ได้เท่านั้น
