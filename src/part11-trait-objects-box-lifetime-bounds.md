# Rust-101, ส่วนที่ 11: Trait Objects, Box, ขอบเขตอายุขัย (Lifetime bounds)

เราจะมาเล่นกับ closures กันอีกสักหน่อย ให้เราลองอิมพลีเมนท์กลไก "callback" (การเรียกกลับ) แบบเจเนอริกบางอย่าง โดยจัดเตรียมฟังก์ชันสองอย่าง: การลงทะเบียน callback ใหม่, และการเรียก callbacks ที่ลงทะเบียนไว้ทั้งหมด

ก่อนอื่น เราต้องหาวิธีเก็บ callbacks ชัดเจนว่า จะต้องขี่ยวข้องกับ `Vec` เพื่อที่เราจะสามารถขยายจำนวนของ callbacks ที่ลงทะเบียนไว้ได้เสมอ Callback จะเป็น closure นั่นคือ สิ่งที่อิมพลีเมนท์ `FnMut(i32)` (เราต้องการเรียกสิ่งนี้หลายครั้ง ดังนั้นชัดเจนว่า `FnOnce` คงใช้ไม่ได้) ดังนั้นความพยายามแรกของเราอาจจะเป็นดังนี้ สำหรับตอนนี้ เราตัดสินใจไปก่อนว่า callbacks มีอาร์กิวเมนต์หนึ่งตัวชนิดข้อมูล `i32`

```rust
struct CallbacksV1<F: FnMut(i32)> {
    callbacks: Vec<F>,
}
```

อย่างไรก็ตาม สิ่งนี้จะไม่ทำงาน จำไว้ว่า "ชนิดข้อมูล" ของ closure นั้นจำเพาะเจาะจงกับสภาพแวดล้อมของตัวแปรที่ถูกจับมา Closures ที่แตกต่างกันซึ่งต่างก็อิมพลีเมนท์ `FnMut(i32)` อาจจะมีชนิดข้อมูลที่แตกต่างกัน อย่างไรก็ตาม `Vec<F>` เป็นคอนเทนเนอร์ที่มีชนิดข้อมูลเป็นแบบเดียวกันหมด (uniformly typed)

เราจึงต้องการวิธีเก็บสิ่งที่มีชนิดข้อมูลแตกต่างกันในเวกเตอร์เดียวกัน เรารู้ว่าชนิดข้อมูลเหล่านี้ทั้งหมดอิมพลีเมนท์ `FnMut(i32)` สำหรับสถานการณ์นี้ Rust จัดเตรียม **trait objects**: ความจริงก็คือ `FnMut(i32)` ไม่ได้เป็นแค่เทรต มันยังเป็นชนิดข้อมูลด้วย ซึ่งสามารถมอบให้กับอะไรก็ตามที่อิมพลีเมนท์เทรตนี้ ดังนั้น เราอาจจะเขียนดังนี้:

```rust
/* struct CallbacksV2 {
    callbacks: Vec<FnMut(i32)>,
} */
```

แต่ Rust บ่นเกี่ยวกับการนิยามนี้ มันพูดอะไรบางอย่างเกี่ยวกับ "Sized" ปัญหาคืออะไร? ดูสิ สำหรับหลายสิ่งที่เราต้องการทำ มันสำคัญมากที่ Rust ต้องรู้ขนาดที่แม่นยำและคงที่ (precise, fixed size) ของชนิดข้อมูล - นั่นคือ ชนิดข้อมูลนี้จะใหญ่แค่ไหนเมื่อถูกแทนค่าในหน่วยความจำ ตัวอย่างเช่น สำหรับ `Vec` สมาชิกจะถูกเก็บต่อๆ กันไปเลย จะเป็นไปได้อย่างไรถ้าไม่มีขนาดที่คงที่? ประเด็นคือ `FnMut(i32)` สามารถมีขนาดเท่าไหร่ก็ได้ เราไม่รู้ว่า "ชนิดข้อมูลที่อิมพลีเมนท์ `FnMut(i32)`" นั้นใหญ่แค่ไหน Rust เรียกสิ่งนี้ว่า **ชนิดข้อมูลที่ไม่มีขนาด (unsized type)** เมื่อไหร่ก็ตามที่เราแนะนำตัวแปรชนิดข้อมูล (type variable) ทาง Rust จะเพิ่มขอบเขต (bound) ให้กับตัวแปรนั้นโดยปริยาย โดยเรียกร้องว่ามันต้องมีขนาด (sized) นั่นคือเหตุผลที่เราไม่ต้องกังวลกับเรื่องนี้มาจนถึงตอนนี้

คุณสามารถเลือกที่จะไม่รับ (opt-out) ขอบเขตโดยปริยายนี้ได้โดยการพูดว่า `T: ?Sized` แล้ว `T` จะมีขนาดหรือไม่ก็ได้

ดังนั้น เราจะทำอย่างไรถ้าเราไม่สามารถเก็บ callbacks ในเวกเตอร์? เราสามารถใส่มันไว้ในกล่อง (box) ในทางความหมาย `Box<T>` เหมือนกับ `T` มาก: คุณเป็นเจ้าของข้อมูลที่เก็บอยู่ที่นั่นอย่างสมบูรณ์ อย่างไรก็ตาม บนเครื่องคอมพิวเตอร์ `Box<T>` คือ pointer ไปยัง `T` ที่ถูกจัดสรรบนฮีป (heap-allocated) มันคล้ายกับ `std::unique_ptr` ใน C++ มาก ในตัวอย่างปัจจุบันของเรา ส่วนที่สำคัญคือเนื่องจากมันเป็น pointer ตัว `T` สามารถไม่มีขนาด (unsized) ได้ แต่ตัว `Box<T>` เองจะมีขนาดคงที่เสมอ ดังนั้นเราสามารถใส่มันใน `Vec` ได้

```rust
pub struct Callbacks {
    callbacks: Vec<Box<FnMut(i32)>>,
}

impl Callbacks {
    // ตอนนี้เราสามารถจัดเตรียมฟังก์ชันบางอย่างได้ ตัวสร้าง (constructor) ควรจะตรงไปตรงมา
    pub fn new() -> Self {
        Callbacks { callbacks: Vec::new() }
    }

    // การลงทะเบียนเพียงแค่เก็บ callback
    pub fn register(&mut self, callback: Box<FnMut(i32)>) {
        self.callbacks.push(callback);
    }

    // เรายังสามารถเขียนเวอร์ชันเจเนอริกของ `register` เพื่อที่มันจะถูกสร้าง instance ด้วย
    // ชนิดข้อมูล closure ที่เป็นรูปธรรม `F` และทำการสร้าง `Box` และการแปลงจาก `F`
    // ไปเป็น `FnMut(i32)` ด้วยตัวเอง
    // 
    // เพื่อให้สิ่งนี้ทำงานได้ เราจำเป็นต้องเรียกร้องว่าชนิดข้อมูล `F` ต้องไม่ประกอบไปด้วย reference
    // ที่มีอายุสั้นใดๆ ท้ายที่สุด เราจะเก็บมันไว้ในรายการ callbacks ของเราอย่างไม่มีกำหนด ถ้า
    // closure ประกอบด้วย pointer ไปยัง stackframe ของผู้เรียกของเรา pointer นั้นอาจจะใช้ไม่ได้
    // ในเวลาที่ closure ถูกเรียก เราสามารถบรรเทาสิ่งนี้ได้โดยการใส่ขอบเขตอายุขัย (bounding by lifetime) ให้ `F`:
    // `F: 'a` บอกว่าข้อมูลทั้งหมดของชนิดข้อมูล `F` จะมีอายุยืนยาวกว่า (นั่นคือ จะใช้งานได้เป็นเวลาอย่างน้อย
    // นานเท่ากับ) lifetime `'a` ที่นี่ เราใช้ lifetime พิเศษ `'static` ซึ่งเป็น lifetime
    // ของทั้งโปรแกรม ขอบเขตเดียวกันนี้ถูกเพิ่มเข้าไปโดยปริยายในเวอร์ชันของ `register`
    // ข้างต้น และในการนิยามของ `Callbacks`
    pub fn register_generic<F: FnMut(i32) + 'static>(&mut self, callback: F) {
        self.callbacks.push(Box::new(callback));
    }

    // และที่นี่เราเรียก callbacks ทั้งหมดที่เก็บไว้
    pub fn call(&mut self, val: i32) {
        // เนื่องจากพวกมันเป็นชนิดข้อมูล `FnMut` เราจำเป็นต้องวนซ้ำแบบแก้ไขได้ (mutably iterate)
        for callback in self.callbacks.iter_mut() {
            // ที่นี่ `callback` มีชนิดข้อมูล `&mut Box<FnMut(i32)>` เราสามารถใช้ประโยชน์จากความจริงที่ว่า
            // `Box` เป็น smart pointer: โดยเฉพาะอย่างยิ่ง เราสามารถใช้มันราวกับว่ามันเป็น
            // reference ปกติ และใช้ `*` เพื่อเข้าไปเอาเนื้อหาของมัน จากนั้นเราได้รับ mutable reference
            // ไปยังเนื้อหาเหล่านี้ เพราะเราเรียก `FnMut`
            (&mut **callback)(val);
            // เหมือนกับกรณีของ reference ปกติ โดยทั่วไปสิ่งนี้เกิดขึ้นโดยปริยาย
            // กับ smart pointers ดังนั้นเราจึงสามารถเรียกฟังก์ชันได้โดยตรง ลองลบ
            // `&mut *` ออกดูสิ
        }
    }
}
```

ความแตกต่างจาก reference คือ `Box` บ่งบอกถึงความเป็นเจ้าของอย่างสมบูรณ์: เมื่อคุณทำ box หล่นหาย (drop) (นั่นคือ เมื่อ instance ของ `Callbacks` ทั้งตัวถูก drop) เนื้อหาที่มันชี้ไปบนฮีปจะถูกคืนหน่วยความจำ (deallocated)

## การสาธิต (Demo)

ตอนนี้เราพร้อมสำหรับการสาธิตแล้ว อย่าลืมแก้ไข `main.rs` เพื่อรันมัน

```rust
pub fn main() {
    let mut c = Callbacks::new();
    c.register(Box::new(|val| println!("Callback 1: {}", val)));
    c.call(0);

    {
        // เราสามารถแม้กระทั่งลงทะเบียน callbacks ที่แก้ไขสภาพแวดล้อมของพวกมัน โดยค่าตั้งต้น Rust จะ
        // พยายามจับ reference ไปยัง `count` เพื่อยืมมัน อย่างไรก็ตาม นั่นใช้ไม่ได้
        // ในครั้งนี้ จำขอบเขต `'static` ข้างบนได้ไหม? การยืม `count` ในสภาพแวดล้อม
        // จะละเมิดขอบเขตนั้น เนื่องจาก reference ใช้งานได้แค่สำหรับบล็อกนี้ ถ้า
        // callbacks ถูกกระตุ้นทีหลัง เราจะเจอปัญหา เราต้องบอก Rust อย่างชัดแจ้งให้
        // ย้าย (move) ความเป็นเจ้าของของตัวแปรเข้าไปใน closure สภาพแวดล้อมของมันก็จะประกอบด้วย
        // `usize` แทนที่จะเป็น `&mut usize` และ closure ก็จะไม่มีผลกระทบต่อตัวแปร local นี้
        let mut count: usize = 0;
        c.register_generic(move |val| {
            count = count + 1;
            println!("Callback 2: {} ({}. time)", val, count);
        });
    }
    c.call(1);
    c.call(2);
}
```

## พฤติกรรมตอนรันไทม์ (Run-time behavior)

เมื่อคุณรันโปรแกรมข้างต้น Rust รู้ได้อย่างไรว่าจะทำอะไรกับ callbacks? เนื่องจากชนิดข้อมูลที่ไม่มีขนาด (unsized type) ขาดข้อมูลบางอย่าง pointer ไปยังชนิดข้อมูลดังกล่าว (ไม่ว่าจะเป็น `Box` หรือ reference) จะจำเป็นต้องเติมข้อมูลนี้ให้เต็ม เราเรียก pointers ไปยัง trait objects ว่า **fat (อ้วน)** พวกมันไม่เพียงเก็บที่อยู่ของวัตถุ แต่ (ในกรณีของ trait objects) ยังเก็บ **vtable**: ตารางของ function pointers ที่กำหนดโค้ดที่จะถูกรันเมื่อเมธอดของเทรตถูกเรียก มีข้อจำกัดบางอย่างสำหรับเทรตที่จะสามารถใช้เป็น trait objects ได้ สิ่งนี้เรียกว่า **object safety (ความปลอดภัยต่อวัตถุ)** และถูกอธิบายในเอกสารและ reference ในกรณีของเทรต `FnMut` มีการกระทำเดียวเท่านั้นที่จะถูกดำเนินการ: การเรียก closure ดังนั้นคุณสามารถคิดว่า pointer ไปยัง `FnMut` เป็น pointer ไปยังโค้ด และ pointer ไปยังสภาพแวดล้อม นี่คือวิธีที่ Rust กู้คืนการเข้ารหัสทั่วไปของ closures ในฐานะกรณีพิเศษของกลไกที่ทั่วไปกว่า

เมื่อไหร่ก็ตามที่คุณเขียนฟังก์ชันเจเนอริก คุณมีทางเลือก: คุณสามารถทำให้มันเป็นเจเนอริก แบบ `register_generic` หรือคุณสามารถใช้ trait objects แบบ `register` แบบหลังจะส่งผลให้มีเวอร์ชันที่ถูกคอมไพล์เพียงเวอร์ชันเดียว (แทนที่จะเป็นหนึ่งเวอร์ชันต่อหนึ่งชนิดข้อมูลที่มันถูกสร้าง instance) สิ่งนี้ทำให้โค้ดเล็กลง แต่คุณจ่ายด้วยค่าใช้จ่ายส่วนเกิน (overhead) ของการเรียกฟังก์ชันเสมือน (virtual function calls) (แน่นอนว่า ในกรณีของ `register` ข้างต้น ไม่มีการเรียกฟังก์ชันบน trait object) มันสวยงามไหมที่เทรตสามารถจัดการกับการแลกเปลี่ยน (tradeoff) นี้ได้อย่างดีเยี่ยม (และทำได้มากกว่านี้มาก อย่างที่เราเห็น เช่น closures และการโอเวอร์โหลดตัวดำเนินการ)?
