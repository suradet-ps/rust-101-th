# Rust-101, ส่วนที่ 03: ข้อมูลนำเข้า (Input)

ในส่วนที่ 00 ผมได้สัญญาไว้ว่าในที่สุดเราจะแทนที่ `read_vec` ด้วยฟังก์ชันที่ถามผู้ใช้ให้กรอกตัวเลขจริงๆ น่าเสียดายที่ I/O (Input/Output) เป็นหัวข้อที่ซับซ้อน ดังนั้นโค้ดส่วนนี้จึงไม่ได้สวยงามนัก - แต่เอาเถอะ เรามาพยายามผ่านมันไปให้ได้ครับ

---

## การนำเข้า I/O (I/O Imports)

I/O ถูกจัดเตรียมไว้โดยโมดูล `std::io` ดังนั้นเราจึงต้องนำเข้า (import) มันด้วย `use` ก่อน เรายังนำเข้า *prelude* ของ I/O ซึ่งจะทำให้เราใช้งานสิ่งต่างๆ ของ I/O ที่ใช้บ่อยๆ ได้โดยตรง

```rust
use std::io::prelude::*;
use std::io;
```

## การอ่านข้อมูลนำเข้า (Reading Input)

เรามาไล่ดูฟังก์ชันนี้ทีละบรรทัดกันครับ เริ่มแรก เราเรียกตัวสร้าง (constructor) ของ `Vec` เพื่อสร้างเวกเตอร์ว่าง ดังที่กล่าวไปในส่วนก่อนหน้านี้ `new` ตรงนี้ก็แค่ฟังก์ชันสแตติกที่ไม่ได้มีการจัดการอะไรเป็นพิเศษ ถึงแม้ว่าจะเป็นไปได้ที่จะเรียก `new` ของชนิดข้อมูลเฉพาะเจาะจง (`Vec::<i32>::new()`) แต่วิธีปกติทั่วไปที่จะทำให้แน่ใจว่าเราได้ชนิดข้อมูลที่ถูกต้องคือการกำกับชนิดข้อมูล (type annotation) ที่ตัวแปร ตัวแปรนี้แหละคือสิ่งที่เราจะต้องโต้ตอบด้วยไปจนจบฟังก์ชัน ดังนั้นการมีชนิดข้อมูลของมันปรากฏให้เห็น (และเข้าถึงได้!) จึงมีประโยชน์กว่ามาก การระบุพารามิเตอร์ชนิดข้อมูลของ `Vec::new` โดยที่ไม่รู้ชนิดข้อมูลที่ส่งกลับ ไม่ได้บอกอะไรเรามากนัก

```rust
fn read_vec() -> Vec<i32> {
    let mut vec: Vec<i32> = Vec::<i32>::new();
```

ตัวจัดการหลักเพื่อเข้าถึงอินพุตมาตรฐาน (standard input) ถูกจัดเตรียมไว้โดยฟังก์ชัน `io::stdin`

```rust
    let stdin = io::stdin();
    println!("Enter a list of numbers, one per line. End with Ctrl-D (Linux) or Ctrl-Z (Windows).");
```

ตอนนี้เราอยากจะวนลูปอ่านอินพุตมาตรฐานทีละบรรทัด เราสามารถใช้ลูป `for` สำหรับงานนี้ได้ แต่มีข้อควรระวังอยู่: จะเกิดอะไรขึ้นถ้ามีโค้ดส่วนอื่นทำงานพร้อมกัน แล้วก็อ่านจากอินพุตมาตรฐานเหมือนกัน? ผลลัพธ์คงจะยุ่งเหยิงน่าดู ดังนั้น Rust จึงบังคับให้เราล็อคอินพุตมาตรฐานถ้าเราต้องการดำเนินการขนาดใหญ่กับมัน (ดูเอกสารสำหรับรายละเอียดเพิ่มเติม)

```rust
    for line in stdin.lock().lines() {
```

ชนิดข้อมูลสำหรับสตริง (แบบไดนามิก, ยืดขยายขนาดได้) ของ Rust คือ `String` อย่างไรก็ตาม ตัวแปร `line` ของเรายังไม่ใช่ชนิดข้อมูลนั้น: มันมีชนิดข้อมูลเป็น `io::Result<String>` ปัญหาของ I/O คือมันสามารถผิดพลาดได้เสมอ ชนิดข้อมูลของ `line` คล้ายกับ `Option<String>` มาก ("String หรือไม่ก็ไม่มีเลย") แต่ในกรณีของ "ไม่มีเลย" นั้น มันมีข้อมูลเพิ่มเติมเกี่ยวกับข้อผิดพลาดด้วย ผมขอแนะนำอีกครั้งให้ตรวจสอบเอกสาร คุณจะเห็นว่า `io::Result` จริงๆ แล้วก็แค่ชื่อแฝงของ `Result` ดังนั้นคลิกที่นั่นเพื่อดูรายการของตัวสร้างและเมธอดทั้งหมดของชนิดข้อมูลนี้

เราจะขี้เกียจกันสักหน่อยตรงนี้แล้วสมมติว่าไม่มีอะไรผิดพลาด: `unwrap` จะคืนค่า `String` ถ้ามี หรือไม่ก็ทำให้โปรแกรมแพนิค (panic) ไปเลย เนื่องจาก `Result` นำรายละเอียดบางอย่างเกี่ยวกับข้อผิดพลาดที่เกิดขึ้นมาด้วย ข้อความแสดงข้อผิดพลาดจึงน่าจะพอฟังรู้เรื่องอยู่บ้าง ถึงกระนั้น คุณคงไม่อยากให้ผู้ใช้เห็นข้อผิดพลาดแบบนี้ ดังนั้นในโปรแกรม "จริง" เราจะต้องจัดการข้อผิดพลาดอย่างเหมาะสม คุณหาเอกสารของ `Result::unwrap` เจอไหม?

ผมเลือกใช้ชื่อเดิม (`line`) สำหรับตัวแปรใหม่เพื่อให้แน่ใจว่าผมจะไม่มีทางเผลอไปใช้ `line` "ตัวเก่า" อีก

```rust
        let line = line.unwrap();
```

ตอนนี้เรามี `String` ของเราแล้ว เราต้องการทำให้มันเป็น `i32` ก่อนอื่นเราตัดส่วนเกิน (trim) บรรทัดเพื่อลบช่องว่างนำหน้าและต่อท้าย `parse` เป็นเมธอดบน `String` ที่สามารถแปลงสตริงเป็นอะไรก็ได้ ลองหาเอกสารของมันดูสิ!

ในกรณีนี้ Rust สามารถรู้ได้เองโดยอัตโนมัติว่าเราต้องการ `i32` (เนื่องจากชนิดข้อมูลที่ส่งกลับของฟังก์ชัน) แต่นั่นดูจะเป็นเวทมนตร์มากเกินไปหน่อยสำหรับรสนิยมของผม เราจะเขียนให้ชัดเจนขึ้นที่นี่: `parse::<i32>` คือ `parse` ที่ถูกกำหนดชนิดข้อมูลเจเนอริกเป็น `i32`

```rust
        match line.trim().parse::<i32>() {
```

`parse` คืนค่า `Result` อีกครั้ง และคราวนี้เราใช้ `match` เพื่อจัดการข้อผิดพลาด (เช่น ผู้ใช้กรอกอะไรบางอย่างที่ไม่ใช่ตัวเลข) นี่เป็นรูปแบบทั่วไปใน Rust: การดำเนินการที่สามารถผิดพลาดได้จะคืนค่า `Option` หรือ `Result` วิธีเดียวที่จะได้ค่าที่เราสนใจออกมาคือผ่านการจับคู่รูปแบบ (pattern matching) (และผ่านฟังก์ชันผู้ช่วยอย่าง `unwrap`) ถ้าเราเรียกฟังก์ชันที่คืนค่า `Result` แล้วทิ้งค่าที่ส่งกลับไปเฉยๆ คอมไพเลอร์จะแจ้งเตือน (warning) ดังนั้นจึงเป็นไปไม่ได้เลยที่เราจะลืมจัดการข้อผิดพลาด หรือเผลอไปใช้ค่าที่ไม่สมเหตุสมผลเพราะเกิดข้อผิดพลาดในการสร้างมันขึ้นมา

```rust
            Ok(num) => {
                vec.push(num);
            },
            // เราไม่ได้สนใจข้อผิดพลาดเฉพาะเจาะจง ดังนั้นเราจึงละเลยมันด้วย `_`
            Err(_) => {
                println!("What did I say about numbers?");
            },
        }
    }

    vec
}
```

สำหรับ `read_vec` ก็มีแค่นี้ครับ ถ้ายังมีคำถามอะไรเหลืออยู่ เอกสารของฟังก์ชันที่เกี่ยวข้องน่าจะช่วยได้มาก ลองหาเอกสารสำหรับ `Vec::push` ดูสิครับ ผมจะไม่ให้ลิงก์ตลอดไปนะครับ เพราะเอกสารนั้นนำทางได้ค่อนข้างง่ายและคุณควรจะคุ้นเคยกับมัน

### แบบฝึกหัด 03.1

เป้าหมายคือการเขียน `SomethingOrNothing::print` เวอร์ชันเจเนอริก เพื่อการนี้ ให้คุณนิยามเทรต `Print` ที่ให้ความสามารถในการพิมพ์แบบเจเนอริก (อย่างง่าย) และอิมพลีเมนท์เทรตนั้นให้กับ `i32` จากนั้นนิยาม `SomethingOrNothing::print2` เพื่อใช้เทรตนั้น และเปลี่ยน `main` ข้างบนให้ใช้ฟังก์ชัน `print2` แบบเจเนอริกตัวใหม่

ผมจะเตรียมโครงร่างสำหรับคำตอบนี้ให้เหมือนเดิม มันยังแสดงให้เห็นวิธีใส่ขอบเขต (bounds) ให้กับการอิมพลีเมนท์แบบเจเนอริกด้วย (ลองเทียบกับบล็อก `impl` จากแบบฝึกหัดที่แล้วดูนะครับ) คุณสามารถอ่านสิ่งนี้ได้ว่า "สำหรับทุกๆ ชนิดข้อมูล `T` ที่มีคุณสมบัติตามเทรต `Print` ฉันจัดเตรียมการอิมพลีเมนท์สำหรับ `SomethingOrNothing<T>` ให้"

สังเกตว่าผมเรียกฟังก์ชันบน `SomethingOrNothing` ว่า `print2` เพื่อให้ไม่ซ้ำกับ `print` ที่นิยามไปก่อนหน้านี้

*คำใบ้*: มีแมโคร `print!` สำหรับการพิมพ์โดยไม่ขึ้นบรรทัดใหม่ด้วยนะ

```rust
pub trait Print {
    /* เพิ่มสิ่งต่างๆ ตรงนี้ */
}
impl Print for i32 {
    /* เพิ่มสิ่งต่างๆ ตรงนี้ */
}
impl<T: Print> SomethingOrNothing<T> {
    fn print2(self) {
        unimplemented!()
    }
}
```
