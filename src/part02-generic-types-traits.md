# Rust-101, ส่วนที่ 02: ชนิดข้อมูลเจเนอริก (Generic types), เทรต (Traits)

ลองกลับมาพิจารณาชนิดข้อมูล `NumberOrNothing` กันสักครู่ มันน่ารำคาญนิดหน่อยไหมที่เราต้องฝัง (hardcode) ชนิดข้อมูล `i32` ลงไปในนั้น? จะเกิดอะไรขึ้นถ้าวันพรุ่งนี้ เราอยากได้ `CharOrNothing` และต่อจากนั้นก็ `FloatOrNothing`? แน่นอนว่าเราไม่อยากเขียน Type และเมธอดภายในตัว (inherent methods) ของมันใหม่ทั้งหมดแน่ๆ

---

## ชนิดข้อมูลเจเนอริก (Generic datatypes)

ทางแก้ของปัญหานี้เรียกว่า *เจเนอริก (generics)* หรือ *โพลีมอร์ฟิซึม (polymorphism)* (คำหลังเป็นภาษากรีก หมายถึง "หลายรูปทรง") คุณอาจจะรู้จักสิ่งที่คล้ายกันนี้จาก C++ (ที่เรียกว่า templates) หรือ Java หรือหนึ่งในภาษาเชิงฟังก์ชัน (functional languages) ทั้งหลาย ดังนั้น ที่นี่ เราจะนิยามชนิดข้อมูลเจเนอริก `SomethingOrNothing`

```rust
pub enum SomethingOrNothing<T> {
    Something(T),
    Nothing,
}
```

แทนที่จะเขียนรูปแบบ (variants) ทั้งหมดออกมา เราสามารถนำเข้า (import) พวกมันทั้งหมดทีเดียวได้เลย

```rust
pub use self::SomethingOrNothing::*;
```

สิ่งที่ทำนี้คือนิยามตระกูลของชนิดข้อมูลทั้งตระกูลจึ้นมา: ตอนนี้เราสามารถเขียน `SomethingOrNothing<i32>` เพื่อกลับไปเป็น `NumberOrNothing` ของเราได้แล้ว

```rust
type NumberOrNothing = SomethingOrNothing<i32>;
```

อย่างไรก็ตาม เรายังสามารถเขียน `SomethingOrNothing<bool>` หรือแม้กระทั่ง `SomethingOrNothing<SomethingOrNothing<i32>>` ได้ด้วย อันที่จริง ชนิดข้อมูลอย่าง `SomethingOrNothing` นั้นมีประโยชน์มากจนมันมีอยู่แล้วในไลบรารีมาตรฐาน: มันเรียกว่าชนิดข้อมูลทางเลือก (option type) เขียนว่า `Option<T>` ลองไปดูเอกสารของมันสิ! (และไม่ต้องกังวล ในนั้นมีเนื้อหาเยอะมากที่เรายังไม่ได้เรียน)

---

## การอิมพลีเมนท์แบบเจเนอริก (Generic `impl`), ฟังก์ชันสแตติก (Static functions)

ชนิดข้อมูลเหล่านี้คล้ายคลึงกันมาก จนเราสามารถสร้างฟังก์ชันเจเนอริกเพื่อสร้าง `SomethingOrNothing<T>` จาก `Option<T>` และในทางกลับกันได้

สังเกตไวยากรณ์สำหรับการให้การอิมพลีเมนท์แบบเจเนอริกกับชนิดข้อมูลเจเนอริก: ให้คิดว่า `<T>` ตัวแรกเป็นการประกาศตัวแปรชนิดข้อมูล (type variable) ("ฉันกำลังทำบางอย่างสำหรับทุกๆ ชนิดข้อมูล T") และ `<T>` ตัวที่สองเป็นการใช้ตัวแปรนั้น ("สิ่งที่ฉันทำ คืออิมพลีเมนท์ `SomethingOrNothing<T>`")

ภายใน `impl` คำว่า `Self` จะอ้างถึงชนิดข้อมูลที่เรากำลังอิมพลีเมนท์ให้ ในที่นี้ มันคือชื่อแฝงของ `SomethingOrNothing<T>` จำไว้ว่า `self` คือ `this` ของ Rust และมีชนิดข้อมูลเป็น `Self` โดยนัย

```rust
impl<T> SomethingOrNothing<T> {
    fn new(o: Option<T>) -> Self {
        match o {
            None => Nothing,
            Some(t) => Something(t)
        }
    }

    fn to_option(self) -> Option<T> {
        match self {
            Nothing => None,
            Something(t) => Some(t)
        }
    }
}
```

สังเกตดูว่า `new` ไม่มีพารามิเตอร์ `self` นี่ตรงกับเมธอดสแตติก (static method) ใน Java หรือ C++ อันที่จริง `new` คือธรรมเนียมปฏิบัติของ Rust สำหรับการนิยามตัวสร้าง (constructors): พวมมันไม่ได้เป็นอะไรพิเศษ ก็แค่ฟังก์ชันสแตติกที่คืนค่า `Self`

คุณสามารถเรียกฟังก์ชันสแตติก และโดยเฉพาะอย่างยิ่งตัวสร้าง ได้ตามที่สาธิตใน `call_constructor`

```rust
fn call_constructor(x: i32) -> SomethingOrNothing<i32> {
    SomethingOrNothing::new(Some(x))
}
```

## เทรต (Traits)

ตอนนี้เรามี `SomethingOrNothing` แบบเจเนอริกแล้ว คงจะดีไม่น้อยถ้าเรามี `vec_min` แบบเจเนอริกด้วย? แน่นอนว่า เราไม่สามารถหาค่าต่ำสุดของเวกเตอร์ของชนิดข้อมูล *อะไรก็ได้* มันต้องเป็นชนิดข้อมูลที่รองรับการทำงาน `min` Rust เรียกคุณสมบัติเหล่านี้ที่เราอาจเรียกร้องจากชนิดข้อมูลว่า *เทรต (traits)*

ดังนั้น เพื่อเป็นก้าวแรกสู่ `vec_min` แบบเจเนอริก เราจะนิยามเทรต `Minimum` ขึ้นมา สำหรับตอนนี้ ให้มองข้าม `Copy` ไปก่อน เราจะกลับมาดูจุดนี้ในภายหลัง เทรตมีความคล้ายคลึงกับอินเทอร์เฟซ (interfaces) ใน Java มาก: คุณนิยามกลุ่มของฟังก์ชันที่คุณต้องการให้มีการอิมพลีเมนท์ รวมถึงชนิดข้อมูลของอาร์กิวเมนต์และการคืนค่าของพวกมัน

ฟังก์ชัน `min` รับอาร์กิวเมนต์สองตัวที่เป็นชนิดข้อมูลเดียวกัน แต่ผมทำให้อาร์กิวเมนต์ตัวแรกเป็นอาร์กิวเมนต์พิเศษ `self` ผมอาจจะเลือกทางเลือกอื่นโดยทำให้ `min` เป็นฟังก์ชันสแตติกดังนี้: `fn min(a: Self, b: Self) -> Self` ก็ได้ อย่างไรก็ตาม ใน Rust ปกติแล้วเราจะชอบใช้เมธอดมากกว่าฟังก์ชันสแตติกถ้าเป็นไปได้

```rust
pub trait Minimum : Copy {
    fn min(self, b: Self) -> Self;
}
```

ถัดมา เราเขียน `vec_min` เป็นฟังก์ชันเจเนอริกบนชนิดข้อมูล `T` ที่เราเรียกร้องว่าจะต้องมีคุณสมบัติตามเทรต `Minimum` ข้อกำหนดนี้เรียกว่า *ขอบเขตของเทรต (trait bound)* ข้อแตกต่างเดียวจากเวอร์ชันก่อนหน้านี้คือเราเรียก `e.min(n)` แทนที่จะเป็น `min_i32(n, e)` Rust จะรู้ได้เองโดยอัตโนมัติว่า `e` เป็นชนิดข้อมูล `T` ซึ่งอิมพลีเมนท์เทรต `Minimum` และดังนั้นเราจึงสามารถเรียกฟังก์ชันนั้นได้

มีความแตกต่างที่สำคัญมากเมื่อเทียบกับ templates ใน C++: เราต้องประกาศจริงๆ ว่าเราต้องการให้ชนิดข้อมูลมีคุณสมบัติตามเทรตไหนบ้าง ถ้าเราละ `Minimum` ทิ้งไป Rust จะบ่นว่าเราไม่สามารถเรียก `min` ได้ ลองดูสิ! นี่ตรงกันข้ามอย่างสิ้นเชิงกับ C++ ที่คอมไพเลอร์จะตรวจสอบรายละเอียดพวกนี้ก็ต่อเมื่อฟังก์ชันถูกใช้งานจริงเท่านั้น

```rust
pub fn vec_min<T: Minimum>(v: Vec<T>) -> SomethingOrNothing<T> {
    let mut min = Nothing;
    for e in v {
        min = Something(match min {
            Nothing => e,
            // ตรงนี้ ตอนนี้เราสามารถเรียกฟังก์ชัน `min` ของเทรตได้แล้ว
            Something(n) => {
                e.min(n)
            }
        });
    }
    min
}
```

ก่อนจะไปต่อ ใช้เวลาสักครู่เพื่อพิจารณาความยืดหยุ่นของแนวทางการทำ abstraction ของ Rust: เราเพิ่งจะนิยามเทรต (อินเทอร์เฟซ) ที่กำหนดเองของเรา แล้วนำไปอิมพลีเมนท์ให้กับชนิดข้อมูลที่มีอยู่แล้ว ด้วยแนวทางลำดับชั้นของ เช่น C++ หรือ Java สิ่งนี้เป็นไปไม่ได้: เราไม่สามารถทำให้ชนิดข้อมูลที่มีอยู่แล้วไปสืบทอด (inherit) มาจากคลาสฐาน (base class) ที่เป็น abstract ของเราในภายหลังได้

ในกรณีที่คุณกังวลเรื่องประสิทธิภาพ ขอให้ทราบว่า Rust ทำการ *monomorphisation* กับฟังก์ชันเจเนอริก: เมื่อคุณเรียก `vec_min` โดยที่ `T` คือ `i32` Rust จะดำเนินการสร้างสำเนาของฟังก์ชันสำหรับชนิดข้อมูลนี้โดยเฉพาะ โดยเติมคำในช่องว่างทั้งหมดให้ ในกรณีนี้ การเรียก `T::min` จะกลายเป็นการเรียกการอิมพลีเมนท์ของเราแบบสแตติก ไม่มีการส่งต่อแบบไดนามิก (dynamic dispatch) เหมือนที่จะเกิดขึ้นกับเมธอดอินเทอร์เฟซของ Java หรือเมธอดเสมือน (virtual methods) ของ C++ พฤติกรรมนี้คล้ายกับ C++ templates ตัวปรับประสิทธิภาพ (Rust ใช้ LLVM) จึงมีข้อมูลทั้งหมดที่มันต้องการเพื่อที่จะทำการ, เช่น, การทำ inline ฟังก์ชัน

## การอิมพลีเมนท์เทรต (Trait implementations)

เพื่อทำให้ `vec_min` สามารถใช้งานกับ `Vec<i32>` ได้ เราต้องอิมพลีเมนท์เทรต `Minimum` ให้กับ `i32`

```rust
impl Minimum for i32 {
    fn min(self, b: Self) -> Self {
        if self < b { self } else { b }
    }
}
```

เราจัดเตรียมฟังก์ชัน `print` ไว้อีกครั้ง สิ่งนี้ยังแสดงให้เห็นว่าเราสามารถมีบล็อก `impl` หลายบล็อกสำหรับชนิดข้อมูลเดียวกันได้ (จำไว้ว่า `NumberOrNothing` เป็นแค่ชื่อแฝงของชนิดข้อมูล `SomethingOrNothing<i32>`) และเราสามารถจัดเตรียมเมธอดบางอย่างให้เฉพาะกับบางอินสแตนซ์ของชนิดข้อมูลเจเนอริกเท่านั้น

```rust
impl NumberOrNothing {
    pub fn print(self) {
        match self {
            Nothing => println!("The number is: <nothing>"),
            Something(n) => println!("The number is: {}", n),
        };
    }
}
```

ตอนนี้เราพร้อมที่จะรันโค้ดใหม่ของเราแล้ว อย่าลืมแก้ `main.rs` ให้เหมาะสมด้วย Rust รู้ได้เองโดยอัตโนมัติว่าเราต้องการให้ `T` ของ `vec_min` เป็น `i32` และ `i32` อิมพลีเมนท์ `Minimum` ดังนั้นทุกอย่างจึงเรียบร้อย

```rust
fn read_vec() -> Vec<i32> {
    vec![18, 5, 7, 3, 9, 27]
}

pub fn main() {
    let vec = read_vec();
    let min = vec_min(vec);
    min.print();
}
```

ถ้ามันพิมพ์ `3` ออกมา แสดงว่า `vec_min` แบบเจเนอริกของคุณทำงานได้! ดังนั้นเตรียมตัวให้พร้อมสำหรับแบบฝึกหัดถัดไป
