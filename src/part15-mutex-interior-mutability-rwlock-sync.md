# Rust-101, ส่วนที่ 15: Mutex, ความสามารถในการแก้ไขภายใน (ต่อ), RwLock, Sync

```rust
use std::sync::{Arc, Mutex};
use std::thread;
use std::time::Duration;
```

เราเห็นแล้วว่าเราสามารถใช้ `Arc` เพื่อแบ่งปันหน่วยความจำระหว่างเธรด อย่างไรก็ตาม `Arc` สามารถให้ได้เพียงการเข้าถึงหน่วยความจำแบบอ่านได้อย่างเดียว (read-only): เนื่องจากมี aliasing โดยทั่วไป Rust จึงไม่สามารถอนุญาตให้แก้ไข (mutation) ได้ เพื่ออิมพลีเมนท์การทำงานพร้อมกันบนหน่วยความจำที่แบ่งปัน (shared-memory concurrency) เราจำเป็นต้องมี aliasing และ mutation - โดยปฏิบัติตามกฎที่เข้มงวดบางอย่างแน่นอน เพื่อให้แน่ใจว่าไม่มี data races ใน Rust shared-memory concurrency ได้มาโดยผ่านทาง interior mutability ซึ่งเราได้อภิปรายกันไปแล้วในบริบทของเธรดเดียวในส่วนที่ 12

## Mutex

ชนิดข้อมูลพื้นฐานที่สุดสำหรับ interior mutability ที่รองรับ concurrency คือ `Mutex<T>` ชนิดข้อมูลนี้อิมพลีเมนท์ critical sections (หรือ locks) แต่ในวิถีทางที่ขับเคลื่อนด้วยข้อมูล (data-driven way): เราต้องระบุชนิดข้อมูลของข้อมูลที่ถูกปกป้องโดย mutex และ Rust ทำให้แน่ใจว่าข้อมูลนั้นถูกเข้าถึงผ่านทาง mutex เท่านั้น พูดอีกอย่างคือ "ล็อกข้อมูล, ไม่ใช่ล็อกโค้ด (lock data, not code)" ถูกบังคับใช้จริงๆ โดยระบบชนิดข้อมูล ซึ่งเป็นไปได้เพราะระเบียบวินัยของความเป็นเจ้าของและการยืม

เป็นตัวอย่าง ลองมาเขียนตัวนับแบบทำงานพร้อมกัน (concurrent counter) เป็นปกติใน Rust ที่เราต้องคิดถึงโครงสร้างข้อมูล (data layout) ของเราก่อน: นั่นจะเป็น `Mutex<usize>` แน่นอนว่า เราต้องการให้หลายเธรดเข้าถึง `Mutex` นี้ได้ ดังนั้นเราจึงห่อมันไว้ใน `Arc`

แทนที่จะให้ชื่อกับทุกฟิลด์ struct สามารถถูกนิยามโดยการให้แค่ลำดับของชนิดข้อมูล (คล้ายกับวิธีที่ variant ของ enum ถูกนิยาม) สิ่งนี้เรียกว่า **tuple struct** มันมักจะถูกใช้เมื่อสร้าง newtype อย่างที่เราทำที่นี่: `ConcurrentCounter` โดยเนื้อแท้เป็นแค่ชื่อใหม่ของ `Arc<Mutex<usize>>` อย่างไรก็ตาม มันเป็นชนิดข้อมูลที่ประกาศแบบ local เราจึงสามารถให้ inherent implementation กับมันและอิมพลีเมนท์เทรตให้มันได้ เนื่องจากฟิลด์เป็น private ไม่มีใครภายนอกโมดูลนี้จะรู้ชนิดข้อมูลที่เรากำลังใช้อยู่ได้เลย

การอิมพลีเมนท์ `Clone` ที่ได้มา (derived) จะโคลน `Arc` ดังนั้นร่างโคลนทั้งหมดจะพูดถึงตัวนับตัวเดียวกันจริงๆ

```rust
#[derive(Clone)]
struct ConcurrentCounter(Arc<Mutex<usize>>);

impl ConcurrentCounter {
    // constructor แค่ห่อ constructor ของ `Arc` และ `Mutex`
    pub fn new(val: usize) -> Self {
        ConcurrentCounter(Arc::new(Mutex::new(val)))
    }

    // การปฏิบัติการหลัก แน่นอนคือ `increment`
    pub fn increment(&self, by: usize) {
        // `lock` บน mutex คืนค่า guard คล้ายกับ `RefCell` มาก guard ให้การเข้าถึง
        // ข้อมูลที่บรรจุอยู่ใน mutex (เราจะอภิปราย `unwrap` เร็วๆ นี้) `.0` คือวิธีที่เรา
        // เข้าถึงองค์ประกอบแรกของ tuple หรือ struct
        let mut counter = self.0.lock().unwrap();
        // guard เป็น smart pointer ไปยังเนื้อหา
        *counter = *counter + by;
        // ที่ตอนจบของฟังก์ชัน `counter` ถูก drop และ mutex ก็ว่างให้ใช้ได้อีกครั้ง
        // สิ่งนี้เกิดขึ้นได้ก็ต่อเมื่อความเป็นเจ้าของ guard อย่างสมบูรณ์ถูกสละไป โดยเฉพาะอย่างยิ่ง เป็นไป
        // ไม่ได้สำหรับเราที่จะรับ reference ไปยังเนื้อหาบางส่วนของมัน, ปลดล็อก
        // mutex, และต่อมาเข้าถึงข้อมูลที่ถูกปกป้องโดยไม่ถือล็อก การบังคับใช้
        // ระเบียบวินัยการล็อกนั้นสามารถแสดงออกได้ในระบบชนิดข้อมูลของ Rust ดังนั้นเราไม่ต้องกังวล
        // เกี่ยวกับ data races ถึงแม้ว่าเรากำลังแก้ไขหน่วยความจำที่แบ่งปันอยู่!
    }

    // แง่มุมที่แนบเนียนอย่างหนึ่งของการล็อกคือการวางยาพิษ (poisoning) ถ้าเธรด panic ในขณะที่มันถือล็อกอยู่
    // มันอาจจะทิ้งโครงสร้างข้อมูลไว้ในสถานะที่ไม่ดี ล็อกจึงถูกพิจารณาว่าถูกวางอดีต (poisoned)
    // ความพยายามในอนาคตที่จะล็อกมันจะล้มเหลว ข้างบน เราเพียงแค่ยืนยันผ่าน `unwrap` ว่าสิ่งนี้จะ
    // ไม่เกิดขึ้น หรืออีกทางเลือก เราสามารถดูสถานะที่ถูกวางยาพิษและพยายามจะ
    // กู้คืน (recover) จากมัน

    // ฟังก์ชัน `get` คืนค่าปัจจุบันของตัวนับ
    pub fn get(&self) -> usize {
        let counter = self.0.lock().unwrap();
        *counter
    }
}
```

ตอนนี้ตัวนับของเราพร้อมปฏิบัติการแล้ว

```rust
pub fn main() {
    let counter = ConcurrentCounter::new(0);

    // เราโคลนตัวนับสำหรับเธรดแรก ซึ่งเพิ่มค่ามันทีละ 2 ทุกๆ 15ms
    let counter1 = counter.clone();
    let handle1 = thread::spawn(move || {
        for _ in 0..10 {
            thread::sleep(Duration::from_millis(15));
            counter1.increment(2);
        }
    });

    // เธรดที่สองเพิ่มค่าตัวนับทีละ 3 ทุกๆ 20ms
    let counter2 = counter.clone();
    let handle2 = thread::spawn(move || {
        for _ in 0..10 {
            thread::sleep(Duration::from_millis(20));
            counter2.increment(3);
        }
    });

    // ตอนนี้เราเฝ้าดูเธรดทำงานกับตัวนับ
    for _ in 0..50 {
        thread::sleep(Duration::from_millis(5));
        println!("Current value: {}", counter.get());
    }

    // สุดท้าย เรารอให้ทุกเธรดเสร็จสิ้นเพื่อมั่นใจว่าเราจับค่าสุดท้าย
    // ของตัวนับได้
    handle1.join().unwrap();
    handle2.join().unwrap();
    println!("Final value: {}", counter.get());
}
```

> **แบบฝึกหัด 15.1**: เพิ่มการปฏิบัติการ `compare_and_inc(&self, test: usize, by: usize)` ที่เพิ่มค่าตัวนับด้วย `by` *ก็ต่อเมื่อ* ค่าปัจจุบันคือ `test`
>
> **แบบฝึกหัด 15.2**: แทนที่จะ panic ในกรณีที่ล็อกถูกวางยาพิษ เราสามารถใช้ `into_inner` บน error เพื่อกู้คืนข้อมูลภายในล็อกได้ เปลี่ยนโค้ดข้างบนให้ทำสิ่งนั้น ลองใช้ `unwrap_or_else` สำหรับงานนี้

## RwLock

นอกเหนือจาก `Mutex` ยังมี `RwLock` ซึ่งจัดเตรียมการล็อกสองแบบ: แบบที่ให้สิทธิเข้าถึงแบบอ่านได้อย่างเดียว (read-only access) กับผู้อ่านจำนวนเท่าไหร่ก็ได้พร้อมกัน, และอีกแบบสำหรับการเข้าถึงแบบเขียนอย่างผูกขาด (exclusive write access) สังเกตว่านี่เป็นรูปแบบเดียวกับที่เราได้เห็นแล้วกับ shared vs. mutable references ดังนั้นอีกวิธีในการอธิบาย `RwLock` คือบอกว่ามันเหมือนกับ `RefCell` แต่ทำงานได้แม้กระทั่งสำหรับการเข้าถึงพร้อมกัน (concurrent access) แทนที่จะ panic เมื่อข้อมูลถูกยืมไปแล้ว `RwLock` จะบล็อกเธรดปัจจุบันจนกว่าล็อกจะว่าง ในมุมมองนี้ `Mutex` คือเวอร์ชันตัดทอนของ `RwLock` ที่ไม่แยกแยะผู้อ่านและผู้เขียน

> **แบบฝึกหัด 15.3**: เปลี่ยนโค้ดข้างบนให้ใช้ `RwLock` เพื่อให้การเรียก `get` หลายครั้งสามารถถูกดำเนินการได้ในเวลาเดียวกัน

## Sync

ชัดเจนว่า ถ้าเราใช้ `RefCell` แทน `Mutex` โค้ดข้างต้นคงทำงานไม่ได้: `RefCell` ไม่ได้ถูกเตรียมมาสำหรับหลายเธรดพยายามเข้าถึงข้อมูลในเวลาเดียวกัน Rust ทำให้แน่ใจได้อย่างไรว่าเราไม่เผลอใช้ `RefCell` ข้ามหลายเธรด?

ในส่วนที่ 13 เราได้พูดถึงชนิดข้อมูลที่ถูกทำเครื่องหมาย `Send` และดังนั้นจึงสามารถย้ายไปอีกเธรดหนึ่งได้ อย่างไรก็ตาม เราไม่ได้พูดถึงคำถามที่ว่า reference เป็น `Send` หรือไม่ สำหรับ `&mut T` คำตอบคือ: มันเป็น `Send` เมื่อไหร่ก็ตามที่ `T` เป็น `Send` `&mut` อนุญาตให้ย้ายค่าย้อนกลับไปกลับมา มันเป็นไปได้แม้กระทั่งสลับเนื้อหาของ mutable references สองตัว ดังนั้นในแง่ของ concurrency การส่ง mutable, unique reference เหมือนกับการส่ง full ownership มาก ในแง่ที่ว่ามันสามารถใช้เพื่อย้ายวัตถุไปยังอีกเธรดหนึ่งได้

แต่แล้ว `&T` ซึ่งเป็น shared reference ล่ะ? ถ้าไม่มี interior mutability มันคงจะโอเคเสมอที่จะส่งค่าเหล่านั้น ท้ายที่สุด ไม่มีการแก้ไขสามารถเกิดขึ้นได้ ดังนั้นจะมีกี่เธรดเข้าถึงข้อมูลก็ได้อย่างที่เราชอบ แต่เมื่อมี interior mutability เรื่องราวซับซ้อนขึ้น Rust แนะนำ marker trait อีกตัวสำหรับจุดประสงค์นี้: `Sync` ชนิดข้อมูล `T` เป็น `Sync` ก็ต่อเมื่อ `&T` เป็น `Send` เช่นเดียวกับ `Send` `Sync` มี default implementation และดังนั้นจึงถูกอิมพลีเมนท์อัตโนมัติสำหรับโครงสร้างข้อมูลถ้าสมาชิกทั้งหมดของมันอิมพลีเมนท์มัน

เนื่องจาก `Arc` จัดเตรียม shared reference ไปยังเนื้อหาของมันให้กับหลายเธรด `Arc<T>` จะเป็น `Send` ก็ต่อเมื่อ `T` เป็น `Sync` ดังนั้นถ้าเราใช้ `RefCell` ข้างบน ซึ่งไม่เป็น `Sync` Rust คงจะจับข้อผิดพลาดนั้นได้ สังเกตอย่างไรก็ตามว่า `RefCell` เป็น `Send`: ถ้าความเป็นเจ้าของของทั้งเซลล์ถูกย้ายไปอีกเธรดหนึ่ง มันก็ยังเป็นไปไม่ได้ที่หลายเธรดจะพยายามเข้าถึงข้อมูลในเวลาเดียวกัน

เกือบทุกชนิดข้อมูลที่เราเห็นมาจนถึงตอนนี้เป็น `Sync` ยกเว้น `Rc` จำไว้ว่า shared reference นั้นดีพอสำหรับการโคลน และเราไม่ต้องการให้เธรดอื่นมาโคลน `Rc` แบบ local ของเรา (พวกเขาจะ race กันเพื่ออัปเดตตัวนับ reference) ดังนั้นมันต้องไม่เป็น `Sync` กฎของ `Mutex` คือการบังคับใช้การประสานเวลา (synchronization) ดังนั้นจึงไม่น่าแปลกใจทั้งหมดที่ `Mutex<T>` เป็น `Send` และ `Sync` โดยมีเงื่อนไขว่า `T` เป็น `Send`

คุณอาจจะสงสัยว่ามีชนิดข้อมูลที่เป็น `Sync` แต่ไม่เป็น `Send` หรือไม่ มีตัวอย่างที่ค่อนข้างลึกซึ้ง (esoteric) ของชนิดข้อมูลเช่นนั้นจริงๆ แต่นั่นไม่ใช่หัวข้อที่ผมอยากจะลงไป ในกรณีที่คุณสงสัย มี Rust RFC หนึ่ง ซึ่งประกอบด้วยชนิดข้อมูล `RcMut` ที่จะเป็น `Sync` และไม่เป็น `Send` คุณอาจจะสนใจ blog post นี้ในหัวข้อดังกล่าวด้วย
